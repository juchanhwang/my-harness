# Cross-Functional

## Engineering Team과 협업

### 핵심 원칙

* **기술을 이해하되, 기술적 결정은 엔지니어에게** — PO는 "무엇을"과 "왜"를 결정, "어떻게"는 엔지니어가
* **기술적 제약을 존중한다** — "간단해 보이는데?"라고 말하지 않는다
* **함께 발견한다** — 스펙을 던지는 것이 아니라 문제를 함께 탐색

### PO가 알아야 할 기술 지식 수준

* API, 데이터베이스, 프론트/백엔드 구조의 **개념적 이해**
* 기술 부채와 아키텍처 결정이 제품에 미치는 **영향** 이해
* "이게 얼마나 어려운가?"에 대한 **감 잡기** (정확한 추정은 엔지니어 몫)
* 기술 트렌드 (AI, 클라우드, 모바일 등)의 **제품 기회** 파악

### Engineering과의 협업 패턴

**1. Problem Brief (문제 공유)**

```
솔루션을 지시하지 말고 문제를 공유한다:

❌ "이 화면에 드래그 앤 드롭 기능 추가해 주세요"
✅ "사용자가 작업 순서를 재배치하는 데 평균 5분이 걸립니다.
    이 시간을 1분 이내로 줄이고 싶습니다."
```

**2. Technical Discovery 참여**

* Engineering이 솔루션 옵션을 탐색할 때 PO도 참여
* 각 옵션의 트레이드오프를 **비즈니스 관점**에서 평가
* 기술적 제약이 scope에 영향을 주면 함께 조정

**3. Definition of Done 합의**

* 무엇이 "완료"인지 사전에 명확히
* 코드 리뷰, 테스트, 성능 기준 포함

## Design Team과 협업

### PO-Designer 협업 모델

**1. 문제 정의 (함께)**

* PO: 비즈니스 맥락, 데이터, 고객 인사이트 제공
* Designer: 사용자 관점 해석, 현재 경험의 문제점 분석

**2. 솔루션 탐색 (함께)**

* Design Sprint (5일): 문제 정의 → 아이디어 → 프로토타입 → 테스트
* 초기 단계에서 다양한 방향 탐색 (수렴하기 전에 발산)

**3. 디자인 리뷰**

```
PO가 디자인 리뷰 시 확인할 것:
- [ ] 사용자 문제를 해결하는가?
- [ ] 비즈니스 목표와 정렬되는가?
- [ ] 기술적으로 구현 가능한 범위인가?
- [ ] 성공 지표를 측정할 수 있는가?

PO가 디자인 리뷰에서 하지 말아야 할 것:
- ❌ 색상, 폰트 등 디자인 디테일 지시
- ❌ "이거 좀 더 예쁘게" 같은 모호한 피드백
- ❌ 디자이너의 전문 영역 침범
```

### Design Sprint (Jake Knapp)

```
월: Map — 문제 정의, 목표 설정
화: Sketch — 솔루션 아이디어 각자 스케치
수: Decide — 최고의 아이디어 선택
목: Prototype — 빠르게 프로토타입 제작
금: Test — 실제 사용자 5명 테스트
```

## Data Team과 협업

### 데이터 요청 프로세스

```markdown
## Data Analysis Request

### 질문
[구체적인 비즈니스 질문 — "얼마나?" "왜?" "누가?"]

### 맥락
[왜 이 분석이 필요한가, 어떤 의사결정에 사용할 것인가]

### 기대 산출물
[대시보드 / 일회성 분석 / 지표 정의]

### 우선순위
[긴급 / 이번 Sprint / 다음 Sprint]

### 필요 시점
[날짜]
```

### PO-Data 협업 원칙

1. **지표 정의를 함께** — "활성 사용자"의 정의를 PO와 Data가 합의
2. **질문을 명확히** — "데이터 좀 봐주세요"(X) → "D7 retention이 떨어진 원인을 세그먼트별로 분석해 주세요"(O)
3. **결과 해석도 함께** — 숫자의 의미를 함께 논의

## Sales/Marketing과 협업

### Sales ↔ Product 피드백 루프

```
Sales (현장) → PO (필터링) → Backlog (우선순위)
     ↑                              │
     └──── 로드맵/기능 업데이트 ─────┘
```

### 협업 원칙

1. **고객 피드백은 귀하지만, 모든 요청을 수용하진 않는다**
2. **패턴을 본다**: 1명의 요청 vs 10명의 요청은 다르다
3. **Win/Loss analysis**: 왜 이겼는지, 왜 졌는지 정기 분석
4. **로드맵 공유**: Sales가 고객에게 정확한 정보를 전달할 수 있도록

### GTM (Go-to-Market) 협업

```
Product → Marketing: 기능 가치, 타겟 사용자, 핵심 메시지
Marketing → Product: 시장 반응, 채널 성과, 경쟁 동향
Product → Sales: 제품 교육, 데모 시나리오, FAQ
Sales → Product: 고객 피드백, 경쟁 정보, 가격 반응
```

## 사일로 조직에서의 PO 리더십

### 사일로의 징후

* 팀 간 정보 공유가 안 된다
* 같은 문제를 여러 팀이 각자 해결한다
* "우리 팀 목표"가 "제품 목표"와 따로 놀다
* 회의에서 방어적 태도가 잦다

### PO가 사일로를 깨는 방법

**1. 공통 목표 설정**

* 팀별 목표가 아닌 **제품 Outcome**으로 정렬
* OKR을 cross-functional하게 설정

**2. 정보 투명성**

* 의사결정 근거를 모든 팀에 공유
* Decision log, 회의록을 공개 채널에

**3. 정기적 Cross-functional Sync**

* 격주 30분: 각 팀 업데이트 + 의존성 확인
* 이 미팅의 목적은 "알려주기"가 아닌 "정렬하기"

**4. 함께 일하기**

* Design Sprint, Discovery Session에 엔지니어 참여
* 사용자 인터뷰에 다양한 직군 참관
* Co-location (물리적/가상 공간 공유)

**5. 성과를 팀이 아닌 제품 단위로**

* "우리 팀이 N개 기능 출시"가 아닌 "제품의 X 지표가 Y% 개선"
