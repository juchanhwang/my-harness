# 의사결정 프레임워크 전체 내용

## 의사결정 프레임워크

### RAPID (Bain & Company)

역할 기반 의사결정 프레임워크. 누가 무엇을 하는지 명확히 한다.

| 역할            | 설명               | 예시 (기능 출시 결정)           |
| ------------- | ---------------- | ----------------------- |
| **R**ecommend | 제안을 만드는 사람       | PO: PRD 작성, 데이터 분석      |
| **A**gree     | 동의가 필요한 사람 (거부권) | Legal, Security (규정 준수) |
| **P**erform   | 실행하는 사람          | Engineering, Design     |
| **I**nput     | 의견을 제공하는 사람      | Sales, CS, Data         |
| **D**ecide    | 최종 결정권자          | VP Product 또는 PO        |

### DACI (Intuit)

| 역할              | 설명                |
| --------------- | ----------------- |
| **D**river      | 의사결정 프로세스를 이끄는 사람 |
| **A**pprover    | 최종 승인자 (1명)       |
| **C**ontributor | 의견/전문성 제공         |
| **I**nformed    | 결정 후 통보받는 사람      |

### RAPID vs DACI

```
소규모 팀/빠른 결정 → DACI (단순함)
대규모 조직/복잡한 결정 → RAPID (역할 세분화)
```

## One-Way Door vs Two-Way Door (Amazon)

Jeff Bezos의 의사결정 분류법:

### One-Way Door (Type 1)

* **되돌릴 수 없거나** 되돌리기 매우 어려운 결정
* 신중하게, 데이터 충분히, 이해관계자 합의 후 결정
* 예: 플랫폼 기술 스택 변경, 가격 모델 근본 변경, 시장 피벗

### Two-Way Door (Type 2)

* **쉽게 되돌릴 수 있는** 결정
* 빠르게 결정하고, 실행하며 학습
* 예: UI 변경, 기능 실험 (feature flag), 마케팅 카피 변경

### 실전 적용

```
결정이 필요할 때:

1. "이 결정은 되돌릴 수 있는가?"
   - Yes → Two-Way Door → 빠르게 결정, 실험으로 검증
   - No → One-Way Door → 아래 프로세스

2. One-Way Door 의사결정 프로세스:
   a. 문제/기회 명확히 정의
   b. 옵션 최소 3개 도출
   c. 각 옵션의 장단점, 리스크 분석
   d. 데이터/고객 인사이트 수집
   e. 이해관계자 의견 수렴
   f. 결정 + 근거 문서화
   g. 커뮤니케이션
```

### 흔한 실수

* Two-Way Door를 One-Way Door처럼 취급 → **속도 저하**
* One-Way Door를 Two-Way Door처럼 취급 → **치명적 실수**
* 대부분의 결정은 Two-Way Door → **더 빠르게 결정해도 된다**

## Decision Log 작성법

### 왜 기록하는가?

* 나중에 "왜 이렇게 결정했지?" 추적 가능
* 새 팀원 온보딩에 활용
* 같은 논의 반복 방지
* 결정의 질 향상 (기록하면 더 신중해짐)

### Decision Log 템플릿

```markdown
## Decision: [결정 제목]

**날짜**: YYYY-MM-DD
**결정자**: [이름/역할]
**참여자**: [이름들]
**상태**: Proposed / Decided / Superseded

### 맥락
[왜 이 결정이 필요한가?]

### 옵션
| 옵션 | 장점 | 단점 | 리스크 |
|------|------|------|--------|
| A (선택) | ... | ... | ... |
| B | ... | ... | ... |
| C | ... | ... | ... |

### 결정
[무엇을 결정했는가]

### 근거
[왜 이 옵션을 선택했는가]

### 결과 (나중에 업데이트)
[실제 결과는 어떠했는가]
```

## Bias 인식과 극복

### 제품 의사결정에서 흔한 편향

| 편향                    | 설명                 | 제품 맥락                  | 대응법                         |
| --------------------- | ------------------ | ---------------------- | --------------------------- |
| **Confirmation Bias** | 자기 가설을 확인하는 데이터만 봄 | "이 기능이 필요하다"는 증거만 수집   | 반증 데이터를 적극적으로 찾기            |
| **Sunk Cost Fallacy** | 이미 투자한 것 때문에 계속    | "3개월 개발했으니 출시해야지"      | "지금 시작한다면 이걸 할까?" 질문        |
| **Anchoring**         | 첫 정보에 과도하게 영향      | CEO가 말한 숫자가 기준점        | 독립적으로 먼저 추정 후 비교            |
| **Survivorship Bias** | 성공 사례만 보고 판단       | "Slack이 그렇게 했으니까"      | 실패 사례도 분석                   |
| **Bandwagon Effect**  | 다수 의견을 따름          | 회의에서 선임의 의견에 동조        | 독립 투표 후 토론 (Planning poker) |
| **Recency Bias**      | 최근 정보에 과도한 비중      | 어제 고객 1명의 불만 → 우선순위 변경 | 장기 데이터와 비교                  |
| **HiPPO**             | 가장 높은 직급의 의견       | "대표님이 좋다고 했으니까"        | 데이터 기반 토론 문화                |

### 디바이어싱(Debiasing) 실전 기법

**1. Pre-mortem (사전 부검)**

```
"이 프로젝트가 6개월 후 완전히 실패했다고 가정합시다.
왜 실패했을까요?"
→ 낙관 편향을 깨고 리스크를 사전 식별
```

**2. Red Team / Devil's Advocate**

```
한 명이 의도적으로 반대 입장을 취한다.
"이 결정의 최악의 시나리오는?"
"이 데이터를 반대로 해석하면?"
```

**3. Outside View**

```
내부 관점이 아닌 외부 기준율(base rate)로 판단.
"비슷한 프로젝트의 성공률은?"
"비슷한 시장 진출의 일반적 결과는?"
```

## Speed vs Quality Tradeoff

### 결정 속도 매트릭스

```
                높은 영향
                    │
   신중하게         │        빠르되 신중하게
   (데이터 수집,    │        (핵심 데이터만,
    합의 도출)      │         시간 제한 설정)
                    │
────────────────────┼────────────────────
                    │         긴급도 높음
   일정에 맞춰      │        즉시
   (큰 리스크 없음  │        (되돌릴 수 있으면
    일반 프로세스)   │         바로 실행)
                    │
                낮은 영향
```

### 70% 규칙 (Jeff Bezos)

> "정보의 70%가 모이면 결정한다. 90%를 기다리면 너무 늦다."

* 완벽한 정보는 없다 → 불완전한 정보로 결정하는 능력이 핵심
* 단, **되돌릴 수 없는 결정**은 더 높은 확신 필요

## 불확실성 속 결정하기

### 불확실성을 다루는 프레임워크

**1. 결정 가능한 것에 집중**

```
통제 가능: 제품 품질, 출시 시점, 마케팅 메시지
통제 불가: 경쟁사 행동, 시장 변화, 규제
→ 통제 가능한 영역에서 최선의 결정
```

**2. 옵션 가치 유지**

```
불확실할 때는 선택지를 열어둔다:
- Feature flag로 점진적 출시
- 작은 실험으로 방향 확인
- 커밋하기 전에 가역적 방법 탐색
```

**3. 결정 시점 설정**

```
"더 많은 정보가 필요하다"는 무한히 미루는 핑계가 될 수 있다.
→ "N일까지 결정한다. 그때까지 가진 정보로."
→ 결정 기한이 없으면 결정이 없다.
```
