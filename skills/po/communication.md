# Communication

## 제품 비전 스토리텔링

### 왜 스토리텔링인가?

데이터와 로직만으로는 사람을 움직이지 못한다. 비전은 **감정적 공명**을 일으켜야 팀이 몰입한다.

### 비전 스토리 구조

```
1. 현재의 문제 (Pain)
   → "지금 사용자들은 이런 고통을 겪고 있습니다"
   → 구체적 사례, 감정적 연결

2. 이상적 미래 (Promise)
   → "우리 제품이 있는 세계에서는 이렇게 됩니다"
   → 생생한 묘사, 구체적 시나리오

3. 어떻게 도달하는가 (Path)
   → "이 비전에 도달하기 위한 우리의 전략은..."
   → 단계별 계획, 첫 번째 마일스톤

4. 왜 우리인가 (Proof)
   → "우리가 이것을 해낼 수 있는 이유는..."
   → 팀의 강점, 초기 성과, 고유 인사이트
```

### 실전 팁

* **숫자 + 이야기**: "30%의 사용자가..." + "김 매니저는 매일 아침..."
* **적의 설정**: 문제/비효율을 적으로 프레이밍 (경쟁사가 아닌)
* **반복**: 같은 비전을 다른 형식으로 7번 이상 전달

## 이해관계자별 커뮤니케이션 맞춤

### 경영진 (C-Level)

**관심사**: 비즈니스 임팩트, ROI, 전략 정합성, 리스크
**형식**: 1-page 요약, 핵심 지표 차트, BLUF (결론 먼저)
**빈도**: 격주-월간
**언어**: "revenue", "market share", "competitive advantage"

```
✅ "이 이니셔티브로 Q2 MRR을 15% 성장시킬 수 있습니다"
❌ "사용자 인터뷰에서 10명 중 7명이 이 기능을 원했습니다"
```

### 개발팀 (Engineering)

**관심사**: 기술적 맥락, 왜 만드는지, 제약 조건, 성공 기준
**형식**: PRD, 기술 문서, 디자인 스펙, 데모
**빈도**: 주간 (Sprint 단위)
**언어**: 구체적 요구사항, 기술 용어 OK

```
✅ "사용자가 작업 순서를 재배치하는 데 5분 걸립니다. 1분 이내로 줄이고 싶습니다"
❌ "드래그 앤 드롭 넣어주세요"
```

### 고객/사용자

**관심사**: 자신의 문제가 해결되는가, 언제 되는가
**형식**: 릴리즈 노트, 블로그, 인앱 공지
**빈도**: 주요 출시 시
**언어**: 기능이 아닌 **혜택** 중심

```
✅ "이제 프로젝트 상태를 한 눈에 파악할 수 있습니다"
❌ "대시보드 v2.1을 출시했습니다"
```

## 문서화 문화

### RFC (Request for Comments)

중요한 의사결정 전에 팀의 의견을 구하는 문서:

```markdown
# RFC: [제목]

**Author**: [이름] | **Status**: Open for comments until [날짜]
**Reviewers**: [이름들]

## Context
[왜 이 결정이 필요한가]

## Proposal
[제안 내용]

## Alternatives Considered
[검토한 대안들과 기각 이유]

## Trade-offs
[이 결정의 장단점]

## Open Questions
[결정하지 못한 부분]

## Comments
[팀원들의 의견 — 인라인 코멘트]
```

### Decision Log (의사결정 로그)

```markdown
## Decision Log

| 날짜 | 결정 | 맥락 | 대안 | 결정자 | 결과 |
|------|------|------|------|--------|------|
| 2025-03-01 | PLG 모델 채택 | 영업팀 부재, 제품 복잡도 낮음 | Inside sales | PO + CEO | 진행 중 |
| 2025-02-15 | React 선택 | 팀 경험, 에코시스템 | Vue, Svelte | Tech Lead | 확정 |
```

### 문서화 원칙

1. **Write things down** — 구두 합의는 증발한다
2. **Single source of truth** — 한 곳에서 관리
3. **Living documents** — 작성 후 방치하지 않고 업데이트
4. **Searchable** — 나중에 찾을 수 있어야 의미 있다
5. **Lightweight** — 문서 작성이 병목이 되면 안 된다

## 회의 퍼실리테이션

### 회의 전

```markdown
## Meeting: [제목]
- 목적: [이 회의에서 달성할 것]
- 참석자: [필수/선택]
- 시간: [XX분]
- 사전 읽기: [있다면 링크]

### 어젠다
1. [토픽 1] (X분) — [목표: 결정/정보공유/브레인스토밍]
2. [토픽 2] (X분)
3. 다음 스텝 확인 (5분)
```

### 회의 중

* **Timeboxing**: 각 어젠다 항목에 시간 제한
* **Parking lot**: 주제에서 벗어나면 기록하고 나중에
* **모든 사람의 의견**: 조용한 참석자에게도 기회 제공
* **결론 명확화**: "정리하면, 우리가 합의한 것은 X이고, 다음 스텝은 Y입니다"

### 회의 후

```markdown
## Meeting Notes — [날짜]

### 결정 사항
- [결정 1]
- [결정 2]

### Action Items
| 액션 | 담당 | 기한 |
|------|------|------|
| | | |

### 다음 회의
- 날짜: [일정]
- 어젠다: [예상 토픽]
```

## Async Communication 원칙

### 왜 비동기인가?

* Deep work 시간 확보
* 시간대가 다른 팀원과 협업
* 더 사려 깊은 의견 (즉각 반응 vs 숙고한 답변)

### Async-First 규칙

| 동기 (회의)       | 비동기 (문서/메시지) |
| ------------- | ------------ |
| 복잡한 토론, 갈등 해결 | 정보 공유, 업데이트  |
| 브레인스토밍        | 의견 수집 (RFC)  |
| 감정적 대화 필요     | 루틴한 보고       |
| 빠른 결정 필요 (긴급) | 비긴급 결정       |

### 좋은 Async 메시지의 조건

1. **맥락 포함**: 배경 없이 질문하지 않는다
2. **구체적 요청**: "어떻게 생각하세요?"(X) → "A와 B 중 어떤 것이 좋은지 금요일까지 의견 주세요"(O)
3. **필요한 정보 첨부**: 상대방이 추가 질문 없이 답할 수 있도록
4. **기한 명시**: 언제까지 답변이 필요한지
5. **채널 구분**: 긴급한 것은 DM, 팀 정보는 공개 채널
