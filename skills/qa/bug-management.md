# Bug Management

## 좋은 버그 리포트

잘 작성된 버그 리포트는 수정 시간을 절반으로 줄인다. 개발자가 리포트만 보고 재현할 수 있어야 한다.

### 버그 리포트 템플릿

```markdown
## 제목
[영역] 간결한 증상 설명

## 환경
- OS: macOS 14.3
- 브라우저: Chrome 122.0
- 디바이스: Desktop (1920x1080)
- 앱 버전: v2.3.1
- 계정 유형: 일반 사용자

## 재현 단계
1. https://app.example.com/login 접속
2. test@test.com / password123 으로 로그인
3. 좌측 메뉴에서 "상품 관리" 클릭
4. "새 상품" 버튼 클릭
5. 가격 필드에 "10000" 입력
6. "저장" 버튼 클릭

## 기대 결과
상품이 저장되고 "저장 완료" 메시지가 표시된다.

## 실제 결과
"서버 오류가 발생했습니다" 에러 메시지가 표시된다.
콘솔에 `POST /api/products 500 Internal Server Error` 로그.

## 재현율
10/10 (항상 재현)

## 스크린샷/영상
[첨부]

## 추가 정보
- 가격을 소수점으로 입력하면 (예: 10000.5) 정상 저장됨
- 정수로만 입력 시 발생
- API 응답: `{"error": "Invalid price format"}`
```

### 좋은 제목 vs 나쁜 제목

```
❌ "버그 발견"
❌ "상품 등록이 안 돼요"
❌ "에러남"

✅ "[상품관리] 정수 가격 입력 시 500 에러 발생"
✅ "[인증] 소셜 로그인 후 프로필 페이지 리다이렉트 실패"
✅ "[결제] Safari에서 카드 결제 시 무한 로딩"
```

## Severity vs Priority

|              | 정의           | 누가 결정? |
| ------------ | ------------ | ------ |
| **Severity** | 기술적 영향도      | QA     |
| **Priority** | 비즈니스 수정 우선순위 | PM/PO  |

### Severity 분류

| Severity          | 설명                  | 예시                   |
| ----------------- | ------------------- | -------------------- |
| **S1 — Critical** | 시스템 불가. 워크어라운드 없음   | 결제 불가, 데이터 유실, 앱 크래시 |
| **S2 — Major**    | 핵심 기능 장애. 워크어라운드 있음 | 검색 결과 잘못, 이메일 발송 실패  |
| **S3 — Minor**    | 비핵심 기능 장애           | 정렬 오류, 잘못된 날짜 포맷     |
| **S4 — Trivial**  | 외관상 문제              | 오타, 정렬 미세 오차, UI 깨짐  |

### Priority 분류

| Priority        | 설명        | SLA    |
| --------------- | --------- | ------ |
| **P1 — Urgent** | 즉시 수정     | 4시간 이내 |
| **P2 — High**   | 현 스프린트 수정 | 2일 이내  |
| **P3 — Medium** | 다음 스프린트   | 2주 이내  |
| **P4 — Low**    | 백로그       | 일정 미정  |

### Severity-Priority 매트릭스

```
        Priority
         P1   P2   P3   P4
S1 │  🔴   🔴   -    -
S2 │  🔴   🟡   🟡   -
S3 │  🟡   🟢   🟢   🟢
S4 │  -    🟢   🟢   ⚪

🔴 즉시/현 스프린트  🟡 빠른 시일  🟢 계획적  ⚪ 백로그
```

**High Severity + Low Priority 예시**: 어드민 전용 기능에서 크래시 → 기술적으로 심각하지만 영향 범위가 작아 우선순위 낮음.

**Low Severity + High Priority 예시**: 랜딩 페이지 오타 → 기술적으로 사소하지만 브랜드 이미지에 즉시 영향.

## 버그 트리아지 (Bug Triage)

### 트리아지 프로세스

```
새 버그 신고
  ↓
[검증] 재현 가능한가?
  ├── 아니오 → 추가 정보 요청 / 닫기
  └── 예 ↓
[분류] Severity 판정
  ↓
[중복 확인] 기존 이슈와 동일?
  ├── 예 → 기존 이슈에 링크, 닫기
  └── 아니오 ↓
[할당] Priority 결정 + 담당자 지정
  ↓
[추적] 상태 관리 (Open → In Progress → Fixed → Verified → Closed)
```

### 트리아지 미팅 (주 1-2회)

**참석**: QA Lead, Tech Lead, PM **시간**: 30분 **안건**:

1. 미분류 버그 리뷰 및 priority 결정
2. 차단(blocker) 버그 상태 확인
3. 오래된 버그 정리 (30일+ Open)
4. 릴리스 차단 버그 확인

## 버그 생명주기 (Lifecycle)

```
[New] → [Confirmed] → [Assigned] → [In Progress] → [Fixed]
                                                       ↓
                                                  [Verified]
                                                    ├── Pass → [Closed]
                                                    └── Fail → [Reopened] → [In Progress]

특수 상태:
[New] → [Duplicate] (중복)
[New] → [Won't Fix] (수정 안 함)
[New] → [Not a Bug] (버그 아님)
[Confirmed] → [Deferred] (연기)
```

## 재현이 안 되는 버그

### 접근 방법

1. **환경 차이 확인**: OS, 브라우저, 해상도, 계정 권한
2. **데이터 상태 확인**: 특정 데이터에서만 발생?
3. **타이밍 확인**: 특정 시간대, 동시 접속, 네트워크 상태
4. **로그 분석**: 서버 로그, 브라우저 콘솔, 네트워크 탭
5. **계속 시도**: 10회, 50회, 100회 반복

### 간헐적 버그 문서화

```markdown
## 재현율: 3/20 (15%)

## 의심 원인
- Race condition: 두 API 호출이 동시에 완료될 때
- 네트워크 지연: 응답이 3초 이상 걸릴 때

## 추가 조사 필요
- [ ] 서버 로그에서 동시 요청 패턴 확인
- [ ] 네트워크 스로틀링으로 재현 시도
- [ ] 관련 코드의 동시성 처리 리뷰
```

## 버그 메트릭

| 메트릭      | 계산            | 목표               |
| -------- | ------------- | ---------------- |
| 버그 발견율   | 스프린트당 발견 버그 수 | 추세 감소            |
| 버그 수정율   | 수정된 / 발견된     | > 90%            |
| 평균 수정 시간 | 신고\~수정 평균     | S1 < 4h, S2 < 2d |
| 재오픈율     | 재오픈 / 수정      | < 10%            |
| 버그 밀도    | 버그 수 / KLOC   | < 5/KLOC         |

## 버그 관리 체크리스트

* [ ] 버그 리포트에 재현 단계가 명확한가
* [ ] Severity와 Priority가 분리되어 있는가
* [ ] 주기적 트리아지 미팅이 있는가
* [ ] 30일+ Open 버그가 리뷰되는가
* [ ] 수정된 버그가 QA 검증(verification)을 거치는가
* [ ] 버그 메트릭을 추적하고 있는가
* [ ] 중복 버그 감지 프로세스가 있는가
