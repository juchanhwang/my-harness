# QA Leadership

## QA 문화 구축

품질은 QA 팀만의 책임이 아니다. **전체 팀이 품질에 대한 주인의식**을 가질 때 진정한 품질이 달성된다.

### 품질 문화의 3단계

```
Level 1 — QA가 품질을 "점검"한다
  ↓ (품질은 QA의 일)
Level 2 — 팀 전체가 품질을 "공유"한다
  ↓ (개발자도 테스트를 쓴다)
Level 3 — 품질이 프로세스에 "내재"되어 있다
  ↓ (품질이 자동으로 보장되는 시스템)
```

### Level 3로 가는 실천 방법

1. **개발자가 테스트를 쓰도록 지원** — 좋은 테스트 예시 공유, 테스트 작성 가이드 제공
2. **PR 리뷰에 QA 관점 포함** — 체크리스트로 체계화
3. **자동화 파이프라인 구축** — 커밋마다 자동 검증
4. **품질 메트릭 투명 공유** — 대시보드로 현황 가시화
5. **Blameless 문화** — 버그는 사람의 잘못이 아니라 시스템의 부족

## Shift Left 실천

### Shift Left란

테스트를 개발 후반이 아닌 **초기 단계**로 옮기는 것. 버그를 늦게 발견할수록 수정 비용이 기하급수적으로 증가한다.

```
비용 곡선:
요구사항 단계: $1
설계 단계:     $5
코딩 단계:     $10
테스트 단계:   $50
프로덕션:      $500
```

### 단계별 QA 참여

| 단계       | QA 활동                            |
| -------- | -------------------------------- |
| **요구사항** | 요구사항 리뷰, 테스트 가능성 확인, 누락된 시나리오 지적 |
| **설계**   | 아키텍처 리뷰에서 테스트 용이성 확인, 리스크 식별     |
| **개발**   | PR 리뷰, 테스트 코드 리뷰, 페어 테스팅         |
| **빌드**   | CI 테스트 결과 모니터링, 플레이키 테스트 관리      |
| **릴리스**  | 회귀 테스트, 탐색적 테스트, 릴리스 체크리스트       |
| **운영**   | 프로덕션 모니터링, 인시던트 분석, DER 추적       |

### 3 Amigos (Three Amigos)

스토리 시작 전에 3명이 모여 10-15분 회의:

* **개발자**: "어떻게 구현할까?"
* **QA**: "어떻게 깨뜨릴 수 있을까? 엣지 케이스는?"
* **PO/PM**: "비즈니스 요구사항이 정확한가?"

**결과**: 시작 전에 엣지 케이스를 발견하고, 인수 기준(AC)을 명확히 함.

## 테스트 전략 문서화

### 테스트 전략 문서 구조

```markdown
# 테스트 전략 문서

## 1. 개요
- 프로젝트 설명
- 품질 목표
- 범위

## 2. 테스트 수준
- 단위 테스트: 도구, 목표 커버리지, 책임
- 통합 테스트: 범위, 도구, 실행 빈도
- E2E 테스트: 핵심 시나리오, 도구, 브라우저
- 탐색적 테스트: 세션 주기, 집중 영역

## 3. 자동화 전략
- 자동화 범위
- 도구 스택
- 유지보수 전략

## 4. 환경
- 로컬, CI, Staging, Production
- 데이터 관리

## 5. 리스크 기반 테스트
- 리스크 매트릭스
- 테스트 우선순위

## 6. CI/CD 통합
- 파이프라인 구조
- 게이트 기준
- 알림 설정

## 7. 메트릭 & 리포팅
- 추적 메트릭
- 대시보드
- 리포팅 주기

## 8. 역할 & 책임
- 누가 무엇을 테스트하는가
- 리뷰 프로세스
```

## 품질 옹호 (Quality Advocacy)

### 품질에 대한 이해관계자 설득

**개발자에게:**

```
"테스트를 작성하면 리팩토링이 자유로워집니다.
 테스트 없이 리팩토링하면 기도 메타입니다."
```

**PM에게:**

```
"테스트 없이 빠르게 가는 건 처음만 빠릅니다.
 기술 부채가 쌓이면 새 기능 하나에 2주가 걸립니다.
 지금 테스트에 2일 투자하면 향후 6개월을 절약합니다."
```

**경영진에게:**

```
"프로덕션 장애 1건 = 개발자 3명 × 1일 + 고객 이탈 + 브랜드 훼손
 QA 투자 = 사전 예방 = 비용 절감"
```

### ROI로 말하기

```
테스트 자동화 ROI 계산:

수동 회귀 테스트: 2명 × 4시간 × 주2회 = 16시간/주
자동화 후: 0시간 (CI 자동 실행) + 유지보수 2시간/주 = 2시간/주
절감: 14시간/주 × 50주 = 700시간/년

자동화 초기 투자: 200시간
ROI: (700 - 200) / 200 = 250% (1년차)
2년차부터: 700 / 50 = 1400% (유지보수만)
```

## QA 엔지니어 성장 경로

```
Junior QA
  ├── 수동 테스트 실행
  ├── 버그 리포팅
  └── 테스트 케이스 작성

Mid QA
  ├── 테스트 자동화 (Unit, E2E)
  ├── CI/CD 파이프라인 구성
  ├── 코드 리뷰 참여
  └── 테스트 계획 수립

Senior QA ← 여기
  ├── 테스트 전략 설계
  ├── 자동화 아키텍처 설계
  ├── 성능/보안 테스트
  ├── 메트릭 기반 품질 개선
  ├── 팀 전체 품질 문화 구축
  └── 기술 멘토링

QA Lead / Manager
  ├── QA 팀 리딩
  ├── 조직 수준 품질 전략
  ├── 이해관계자 관리
  └── 예산/리소스 관리
```

## 인시던트로부터 배우기

### 포스트모템 (Blameless Post-mortem)

```markdown
## 인시던트 포스트모템

**날짜**: 2024-01-15
**영향**: 결제 실패 2시간, 매출 손실 추정 $5,000
**타임라인**:
- 14:00 — 배포 (v2.3.1)
- 14:15 — 결제 에러율 급증 모니터링 알림
- 14:30 — 원인 파악 (환경변수 누락)
- 14:45 — 핫픽스 배포
- 15:00 — 정상 복구 확인

**근본 원인**:
새 결제 모듈에 필요한 환경변수가 프로덕션에 설정되지 않음.
체크리스트에 환경변수 확인 항목 없었음.

**왜 테스트에서 못 잡았나?**:
1. CI에서는 테스트 환경변수가 이미 설정되어 있었음
2. 환경변수 누락 시 에러 대신 기본값으로 동작 (silent failure)

**개선 액션**:
1. [완료] 앱 시작 시 필수 환경변수 검증 (Zod)
2. [진행] 배포 체크리스트에 환경변수 확인 추가
3. [계획] Staging 환경을 프로덕션과 동일하게 구성
4. [계획] 환경변수 차이 감지 자동화 스크립트
```

## 지속적 개선

### 회고에서 품질 이슈 다루기

매 스프린트 회고에서:

1. 이번 스프린트에서 프로덕션까지 빠져나간 버그는?
2. 어떤 테스트가 있었으면 잡았을까?
3. 테스트 프로세스에서 개선할 점은?

### 품질 개선 백로그 유지

```markdown
## QA 개선 백로그
- [ ] [High] 결제 플로우 E2E 테스트 추가
- [ ] [High] CI 테스트 실행 시간 20분 → 10분
- [ ] [Medium] Playwright visual regression 도입
- [ ] [Medium] 성능 테스트 k6 스크립트 작성
- [ ] [Low] Storybook 접근성 테스트 자동화
```

## QA 리더십 체크리스트

* [ ] 팀 전체가 품질에 대한 공동 책임을 인식하는가
* [ ] Shift Left가 실천되고 있는가 (3 Amigos 등)
* [ ] 테스트 전략 문서가 존재하고 최신인가
* [ ] 품질 메트릭이 투명하게 공유되는가
* [ ] 인시던트 포스트모템이 blameless하게 진행되는가
* [ ] QA 개선 백로그가 관리되고 있는가
* [ ] 개발자가 테스트를 스스로 작성하는 문화인가
* [ ] 신규 팀원에게 QA 온보딩이 있는가
