# Regression Strategy

## 회귀 테스트란

이전에 동작하던 기능이 새로운 변경에 의해 **깨지지 않았는지** 확인하는 테스트. "고쳤더니 다른 데가 터졌다"를 방지한다.

## 회귀 테스트 범위 결정

### 전체 회귀 vs 선택적 회귀

| 전략         | 장점     | 단점        | 언제      |
| ---------- | ------ | --------- | ------- |
| **전체 회귀**  | 높은 신뢰도 | 시간 많이 소요  | 메이저 릴리스 |
| **선택적 회귀** | 빠름     | 놓칠 수 있음   | 일반 스프린트 |
| **리스크 기반** | 효율적    | 리스크 평가 필요 | 항상 추천   |

### 선택적 회귀 기준

```
변경된 영역 (Changed)
  └── 직접 영향받는 영역 (Directly Affected)
       └── 간접 영향받는 영역 (Indirectly Affected)
            └── 핵심 비즈니스 플로우 (Always)
```

```typescript
// 변경 영향도 분석 예시
const regressionScope = {
  changed: ['결제 모듈'],
  directlyAffected: ['장바구니', '주문 내역', '환불'],
  indirectlyAffected: ['이메일 알림', '대시보드 통계'],
  alwaysTest: ['로그인', '핵심 결제 플로우', '데이터 무결성'],
};

// 테스트 우선순위
// 1. changed + alwaysTest (필수)
// 2. directlyAffected (높음)
// 3. indirectlyAffected (시간 여유 시)
```

## 회귀 테스트 스위트 관리

### 테스트 분류 (Tagging)

```typescript
// Playwright 태그 사용
import { test } from '@playwright/test';

test('핵심 결제 플로우', { tag: ['@smoke', '@critical', '@regression'] }, async ({ page }) => {
  // ...
});

test('상품 필터 정렬', { tag: ['@regression'] }, async ({ page }) => {
  // ...
});

test('어드민 리포트 다운로드', { tag: ['@regression', '@admin'] }, async ({ page }) => {
  // ...
});
```

```bash
# 스모크 테스트만 실행 (배포 직후)
npx playwright test --grep @smoke

# 전체 회귀 테스트
npx playwright test --grep @regression

# 크리티컬 제외 나머지
npx playwright test --grep @regression --grep-invert @critical
```

### 스위트 구조

```
회귀 테스트 스위트
├── 🔴 Smoke Tests (5-10개, <5분)
│   └── 로그인, 핵심 페이지 로드, 핵심 API 응답
├── 🟡 Critical Path (20-30개, <15분)
│   └── 가입 → 로그인 → 핵심 기능 → 결제
├── 🟢 Full Regression (100+개, <60분)
│   └── 모든 기능의 주요 시나리오
└── 🔵 Extended (200+개, 야간 실행)
    └── 엣지 케이스, 비기능, 크로스 브라우저
```

### 실행 전략

| 이벤트          | 실행 범위                   |
| ------------ | ----------------------- |
| 커밋/PR        | Unit + Integration (자동) |
| PR 머지 (main) | Smoke + Critical Path   |
| 일일 (야간)      | Full Regression         |
| 릴리스 후보       | Extended + 탐색적          |
| 핫픽스          | Smoke + 관련 영역           |

## 테스트 부채 (Test Debt)

### 테스트 부채란

* 플레이키 테스트가 쌓인다
* 중복 테스트가 쌓인다
* 제거된 기능의 테스트가 남아 있다
* 테스트가 있지만 assert가 약하다
* 테스트 실행 시간이 계속 늘어난다

### 테스트 부채 지표

```
🚨 경고 신호:
- CI 전체 실행 시간 > 30분
- 플레이키 테스트 > 5%
- 스킵된 테스트 > 10개
- 마지막 테스트 정리 > 3개월 전
- "이 테스트가 뭘 하는지 모르겠는데..." 반응
```

### 테스트 부채 상환 계획

```markdown
## 테스트 부채 상환 스프린트 계획

### 주 1: 플레이키 테스트 정리
- [ ] 상위 5개 플레이키 테스트 분석 및 수정
- [ ] 재현 불가 테스트 제거 또는 재작성
- [ ] flaky test 모니터링 대시보드 구축

### 주 2: 중복/불필요 테스트 정리
- [ ] 동일한 검증을 하는 테스트 통합
- [ ] 제거된 기능의 테스트 삭제
- [ ] 스킵된 테스트 리뷰 → 수정 또는 삭제

### 주 3: 테스트 품질 개선
- [ ] assert 없는 테스트에 검증 추가
- [ ] 하드코딩 대기(waitForTimeout) 제거
- [ ] 테스트 헬퍼/팩토리 리팩토링

### 주 4: 실행 시간 최적화
- [ ] 느린 테스트 분석 (상위 10개)
- [ ] 병렬화 개선
- [ ] 불필요한 setup/teardown 최적화
```

## 테스트 정리/삭제 기준

### 삭제해야 하는 테스트

1. **제거된 기능의 테스트**: 기능이 없으면 테스트도 없어야 한다
2. **영구 스킵된 테스트**: 3개월 이상 skip 상태 → 삭제 또는 수정
3. **중복 테스트**: 같은 로직을 다른 방식으로 두 번 테스트
4. **가치 없는 테스트**: `expect(true).toBe(true)` 수준
5. **구현 세부사항 테스트**: 리팩토링마다 깨지는 테스트

### 삭제하면 안 되는 테스트

1. **과거 프로덕션 버그에 대한 테스트**: 같은 버그 재발 방지
2. **핵심 비즈니스 로직 테스트**: 결제, 인증, 데이터 무결성
3. **보안 관련 테스트**: XSS, injection, 인가
4. **규제 요구사항 테스트**: 법적 의무 사항

### 삭제 프로세스

```
1. 삭제 후보 목록 작성
2. 팀 리뷰 (왜 삭제하는지 설명)
3. 관련 커버리지 영향 확인
4. 삭제 + PR (이유를 커밋 메시지에)
5. 커버리지 하락 없는지 확인
```

## 회귀 테스트 자동화 우선순위

### 자동화 우선순위 결정 매트릭스

```
        자동화 용이성
         높음    낮음
중요도 │        │
 높음  │ 1순위  │ 3순위 (도구 개선 후)
       │        │
 낮음  │ 2순위  │ 수동 유지
```

### 자동화하지 말아야 할 테스트

* 한 번만 실행하는 테스트
* UI가 매우 자주 변하는 영역
* 주관적 판단이 필요한 테스트 (UX, 디자인 감성)
* 자동화 비용 > 수동 비용인 테스트

```markdown

**실행 일시**: 2024-01-20 02:00 AM (야간)
**환경**: Staging
**총 테스트**: 245개
**결과**: 241 Pass / 2 Fail / 2 Skip

### 실패 테스트
| # | 테스트 | 상태 | 담당자 | 분석 |
|---|--------|------|--------|------|
| 1 | 장바구니 수량 변경 | ❌ Fail | Alice | 새 API 호환 이슈 → S2 생성 |
| 2 | 이메일 인증 링크 | ❌ Fail | Bob | 플레이키 (타이밍) → 수정 중 |

### 스킵된 테스트
| # | 테스트 | 이유 |
|---|--------|------|
| 1 | 결제 PG 연동 | PG 샌드박스 점검 중 |
| 2 | SMS 발송 | SMS API 쿼터 초과 |

### 릴리스 판단
⚠️ **조건부 승인** — #1 수정 후 해당 영역 재테스트 필요
```

## 체크리스트

* [ ] 회귀 테스트 스위트가 분류(smoke, critical, full)되어 있는가
* [ ] 변경 영향도에 따라 선택적 회귀 테스트가 가능한가
* [ ] CI에서 적절한 시점에 적절한 범위가 실행되는가
* [ ] 테스트 부채가 정기적으로 관리되는가
* [ ] 스킵된 테스트가 3개월 이내에 리뷰되는가
* [ ] 플레이키 테스트가 추적/수정되는가
* [ ] 회귀 테스트 결과 리포트가 릴리스 전 작성되는가
* [ ] 테스트 삭제 기준이 팀에 합의되어 있는가
