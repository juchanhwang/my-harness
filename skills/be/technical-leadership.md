# Technical Leadership

## Architecture Decision Records (ADR)

기술 결정의 맥락, 선택지, 근거를 문서화. 미래의 팀원이 "왜 이렇게 했는가?"를 이해할 수 있게.

### ADR 템플릿

```markdown
# ADR-001: 메시지 큐로 BullMQ 선택

## Status
Accepted (2024-03-15)

## Context
주문 처리 후 이메일 발송, 재고 업데이트 등 비동기 작업이 필요.
현재는 API 응답 전에 동기적으로 처리하여 응답 시간이 2초 이상.

## Decision Drivers
- 기존 Redis 인프라 활용 가능
- Node.js 생태계 호환성
- 팀의 학습 비용
- 운영 복잡도

## Considered Options
1. **BullMQ** — Redis 기반, Node.js 네이티브
2. **RabbitMQ** — 범용 메시지 브로커
3. **Kafka** — 대규모 이벤트 스트리밍

## Decision
BullMQ 선택.

## Rationale
- 이미 Redis를 캐시로 사용 중 → 추가 인프라 불필요
- Node.js 전용이므로 API가 직관적
- 현재 트래픽(~1000 QPS)에 충분
- RabbitMQ/Kafka는 현 단계에서 과도한 복잡성

## Consequences
### Positive
- Redis 하나로 캐시 + 큐 운영
- 학습 비용 최소

### Negative
- Redis 장애 시 캐시 + 큐 동시 영향
- 대규모 이벤트 스트리밍에는 부적합

### Risks
- 트래픽이 10x 이상 증가하면 Kafka로 마이그레이션 필요
- 마이그레이션 시점: QPS > 10,000 or 큐 depth > 100,000
```

### ADR 관리 규칙

1. **번호 순서**: `docs/adr/001-*.md`, `002-*.md`
2. **불변**: 한번 Accepted된 ADR은 수정하지 않음. 변경 시 새 ADR로 Superseded 표시
3. **코드 리뷰에 포함**: ADR이 필요한 변경은 PR에 ADR 포함
4. **가볍게 유지**: 1~2페이지, 핵심만

## RFC (Request for Comments) 프로세스

대규모 변경 전 팀의 피드백을 받는 프로세스.

### RFC 흐름

```
1. 작성자가 RFC 문서 초안 작성
2. 팀 공유 (Notion/Google Docs + Slack 링크)
3. 피드백 기간 (3-5 영업일)
4. 의견 반영 + 최종 결정
5. RFC 승인/거부 → 실행
```

### 언제 RFC가 필요한가

* 새로운 기술/프레임워크 도입
* 아키텍처 변경 (DB 교체, 서비스 분리 등)
* 팀 프로세스 변경
* 여러 팀에 영향을 미치는 변경

## 코드 리뷰 문화

### Constructive Feedback

```
❌ "이 코드 이상한데요"
✅ "이 부분에서 race condition이 발생할 수 있을 것 같습니다.
    두 요청이 동시에 balance를 읽으면 하나가 덮어쓸 수 있어요.
    SELECT FOR UPDATE나 atomic update를 사용하면 어떨까요?"

❌ "왜 이렇게 했어요?"
✅ "궁금한 점: X 대신 Y 방식을 선택한 이유가 있나요?
    X가 성능 면에서 유리할 수 있어서 여쭤봅니다."
```

### 리뷰 우선순위

```
P0 (반드시 수정): 버그, 보안 취약점, 데이터 손실 가능성
P1 (강력 권장): 성능 문제, 에러 처리 누락
P2 (제안): 가독성 개선, 네이밍, 패턴 개선
P3 (nitpick): 코드 스타일, 사소한 개선 (nit: 접두사)
```

### 리뷰 원칙

1. **코드를 비판하되, 사람을 비판하지 마라**
2. **"왜"를 설명하라**: 무엇이 문제인지 + 왜 문제인지
3. **대안을 제시하라**: 문제만 지적하지 말고 해결 방향도
4. **칭찬도 하라**: 좋은 코드에는 "좋습니다!" 코멘트
5. **24시간 내 리뷰**: 리뷰 지연 = 전체 개발 속도 저하

## 기술 부채 관리

### 기술 부채 분류

| 유형                   | 예시                  | 처리            |
| -------------------- | ------------------- | ------------- |
| **Deliberate** (의도적) | "일단 하드코딩, 나중에 설정으로" | 기한 설정, 이슈 등록  |
| **Accidental** (실수)  | 알려지지 않은 안티패턴        | 코드 리뷰에서 catch |
| **Bit rot** (노후화)    | 오래된 라이브러리, 방치된 코드   | 주기적 업데이트 스프린트 |

### 관리 전략

1. **가시화**: 기술 부채 백로그 유지 (Linear 라벨: `tech-debt`)
2. **20% 규칙**: 매 스프린트의 20%를 기술 부채 해소에 할당
3. **Boy Scout Rule**: "왔을 때보다 깨끗하게" — 건드린 파일의 작은 개선
4. **부채 이자 계산**: "이 부채를 안 갚으면 매주 N시간 낭비"로 우선순위 정당화

## 온콜 & 인시던트 대응

### Severity Levels

| Level | 기준        | 응답 시간  | 예시                |
| ----- | --------- | ------ | ----------------- |
| SEV1  | 전체 서비스 다운 | 15분    | 결제 불가, 전면 장애      |
| SEV2  | 주요 기능 장애  | 30분    | 로그인 불가, 일부 API 장애 |
| SEV3  | 부분 기능 장애  | 2시간    | 알림 지연, 비핵심 기능 오류  |
| SEV4  | 경미한 이슈    | 다음 영업일 | UI 깨짐, 로그 경고      |

### Runbook 템플릿

```markdown
# Runbook: 결제 서비스 장애

## 증상
- 결제 API 5xx 에러 증가
- Stripe webhook 처리 실패

## 진단
1. Stripe 상태 확인: https://status.stripe.com
2. DB 연결 확인: `SELECT 1` from payment_db
3. Redis 연결 확인: `redis-cli ping`
4. 에러 로그 확인: `kubectl logs -l app=payment-service`

## 대응
### Stripe 장애인 경우
- 큐에 재시도 예약 (자동)
- 사용자에게 "잠시 후 다시 시도" 안내

### DB 연결 실패인 경우
1. Connection pool 상태 확인
2. DB 서버 상태 확인
3. 필요 시 서비스 재시작

## 에스컬레이션
- 15분 내 해결 불가 → 팀 리드 호출
- 30분 내 해결 불가 → CTO 호출
```

## 포스트모템 (Blameless Postmortem)

### 템플릿

```markdown
# 포스트모템: 2024-03-15 결제 서비스 장애

## Summary
15:30-16:15 (45분) 동안 결제 API가 간헐적으로 5xx 에러 반환.
영향: 결제 시도의 약 30%가 실패. 약 500건의 결제 실패.

## Timeline
- 15:25 배포 (PR #234 — 결제 로직 리팩토링)
- 15:30 Sentry 에러 알림 시작
- 15:35 온콜 엔지니어 인지, 조사 시작
- 15:45 DB connection pool exhaustion 확인
- 15:55 원인 파악: 새 코드에서 트랜잭션 미종료
- 16:05 이전 버전으로 롤백
- 16:15 정상 확인

## Root Cause
PR #234에서 결제 처리 중 에러 발생 시 트랜잭션을 rollback하지 않는 코드 경로 존재.
미종료 트랜잭션이 connection을 점유하여 pool 고갈.

## What Went Well
- 모니터링이 5분 내에 이상 감지
- 롤백이 10분 내에 완료

## What Went Wrong
- 코드 리뷰에서 트랜잭션 미종료 패턴을 발견하지 못함
- 스테이징에서 부하 테스트를 하지 않아 재현 불가했음

## Action Items
- [ ] 트랜잭션 자동 종료 wrapper 추가 (@abel, 3/20)
- [ ] 코드 리뷰 체크리스트에 트랜잭션 항목 추가 (@mark, 3/18)
- [ ] 스테이징 부하 테스트 자동화 (@abel, 3/25)
- [ ] Connection pool 모니터링 알림 추가 (@mark, 3/20)
```

### 원칙

* **Blameless**: "누가 잘못했나"가 아니라 "시스템이 어떻게 실패했나"
* **5 Whys**: 표면적 원인이 아닌 근본 원인까지 파고들기
* **Action Items**: 구체적, 담당자 지정, 기한 설정

## 멘토링 패턴

### 코드 리뷰를 통한 멘토링

```
1. 문제를 지적하기 전에 질문하기
   "이 부분은 어떤 의도로 작성하셨나요?"

2. 원리를 설명하기
   "이 패턴은 N+1 문제를 일으킬 수 있어요.
    N+1이란... [설명].
    해결 방법은... [예시]."

3. 참고 자료 공유
   "이 주제에 대해 [링크]가 좋은 설명을 하고 있어요."

4. 점진적 난이도
   처음: 스타일, 네이밍 피드백
   중간: 패턴, 아키텍처 피드백
   나중: 시스템 설계, 트레이드오프 논의
```

### 1:1 멘토링

* **Weekly 1:1**: 30분, 기술적 어려움 + 성장 방향 논의
* **Pair Programming**: 복잡한 문제를 함께 해결하며 사고 과정 공유
* **Stretch Assignment**: 약간 어려운 태스크를 맡기고 가이드
* **Tech Talk**: 팀 내 지식 공유 세션 (30분, 매주/격주)
