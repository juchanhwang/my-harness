## 1. 라이브러리 없이 Intl API로 충분한 경우

**먼저 라이브러리 도입이 오버엔지니어링은 아닌지 판단하라.** 2026년 브라우저는 `Intl.*` API를 광범위하게 지원한다.

다음 조건을 **모두** 만족하면 라이브러리 없이 `Intl.*` API(`DateTimeFormat`, `NumberFormat`, `PluralRules` 등) + 작은 `t(key)` 유틸로 충분하다:

- 번역 메시지가 **수십 개 미만**, 복수형/select 규칙이 단순
- locale별 URL 라우팅이 불필요
- SSR/RSC 복잡성 없음 (순수 SPA)

### TC39 `Intl.MessageFormat` — 아직 기다리지 마라

`Intl.MessageFormat`은 TC39 **Stage 1**(2026-04 기준)이고, 브라우저 구현은 전무하다. "곧 나올 것이니 라이브러리 없이 기다린다"는 전략은 부적합하다. 최소 수년은 대기해야 한다.

> 📎 출처: [TC39 proposal-intl-messageformat](https://github.com/tc39/proposal-intl-messageformat)

## 2. 라이브러리 선택 기준

Next.js와 순수 React의 라이브러리 선택지가 다르다. 먼저 프로젝트 유형을 확인하고, 그 다음 요구사항을 판단한다.

### 2.1 Next.js App Router 프로젝트

**→ `next-intl` (v4.9+)**

- RSC 1급 지원 (`getTranslations` / `useTranslations` 구분)
- `routing` config로 locale prefix 라우팅 자동 처리
- 타입 안전 `AppConfig` (v4)
- Next.js 팀과 긴밀한 협업 (root-params, cacheComponents 이슈 직접 논의)
- 번들: ~13KB gzip

**대안 — `next-i18next` v16:** "죽었다"는 건 2023-2024 정보다. v16.0.5(2026-04)에서 App Router/RSC를 full 지원하며 활발히 유지된다. 팀이 이미 i18next 생태계에 익숙하거나 번역 SaaS 워크플로가 있다면 선택지다. 다만 App Router 전용으로는 next-intl이 더 타이트한 통합을 제공한다.

### 2.2 순수 React 프로젝트 (Vite/CRA/React Router/React Native)

**→ 요구사항에 따라 3가지 중 선택**

| 상황 | 권장 | 근거 |
|---|---|---|
| **신규, 번들 크기 중시, TypeScript heavy** | **Lingui** (v5.9+) | ~3.7KB gzip (최소), 매크로로 키 자동 생성/타입 안전 |
| **신규, 리치 Unicode/CLDR 필요** | **react-intl** (FormatJS v10+) | Unicode/TC39 팀이 관리, ICU native, 미래 MF2 이행 경로 최단 |
| **신규/유지보수, 생태계 중시, 번역 SaaS 연동** | **react-i18next** (v17+/i18next v26) | 주간 다운로드 10M(1위), 플러그인 생태계, locize/Crowdin 호환성 |
| **기존 i18next 구축됨** | **유지** | 마이그레이션 ROI 낮음. v25.4 selector API, i18next-cli 도입으로 현대화 |
| **React Native + Hermes** | **Lingui (Expo ≥52)** 또는 **react-i18next** | Intl 폴리필(`@formatjs/intl-*`) 필수 |

### 2.3 Claude가 자주 틀리는 버전 정보

- **react-intl v10은 React 19 전용.** React 18은 v9.x
- **Lingui 매크로 경로**: `@lingui/macro` (구식) → **`@lingui/react/macro`** (v5+)
- **FormatJS 빌드툴**: Babel plugin 아닌 **`@formatjs/unplugin`** (Vite/esbuild/Rollup 범용)
- **i18next는 ICU 기본 미지원** — `i18next-icu` 플러그인 필수
- **"next-i18next 죽었다"** — 틀림. v16.0.5(2026-04)에서 App Router 지원

## 3. 공통 설계 원칙 (라이브러리 무관)

### 3.1 번역 키 네이밍

**Namespace는 컴포넌트/기능 단위로 구성한다.** 페이지 단위보다 재사용성이 높다.

```json
{
  "auth": {
    "SignUp": {
      "title": "회원가입",
      "form": {
        "emailPlaceholder": "이메일",
        "submit": "가입하기"
      }
    }
  },
  "common": {
    "cancel": "취소",
    "submit": "제출",
    "loading": "로딩 중..."
  }
}
```

**규칙:**
- Namespace는 기능/컴포넌트 단위 (`HomePage`, `Navigation`, `FAQEntry`)
- 키 자체에 `.` 금지 — `.`은 중첩 표기 전용
- placeholder는 alphanumeric + underscore만 (`{user_name}` O, `{user-name}` X)
- 공통 키는 `common.*` 네임스페이스에 분리
- 네스팅 권장 (flat보다) — 번역 SaaS 도구 지원이 더 좋다

### 3.2 번역 파일 분할 시점

**기본은 단일 `messages/{locale}.json`**이다. 분할은 lazy-loading 목적이 아니라 **팀 번역 워크플로** 필요성으로만 한다.

### 3.3 하드코딩 금지

```tsx
// ❌ 하드코딩 — 나중에 리팩토링 지옥
<button>제출</button>
<p>안녕하세요, {user.name}님!</p>

// ✅ 번역 키
<button>{t('common.submit')}</button>
<p>{t('dashboard.greeting', { name: user.name })}</p>
```

처음부터 i18n을 고려하지 않으면 나중에 붙이는 비용이 실제 개발 비용보다 크다.

## 4. ICU MessageFormat 함정

### 4.1 한국어 복수형 — `other`만 사용

`Intl.PluralRules` 기준으로 언어별 카테고리가 다르다.

| 언어 | 카테고리 |
|---|---|
| **한국어/중국어/일본어** | `other`만 |
| **영어** | `one`, `other` |
| **러시아어** | `one`, `few`, `many`, `other` |
| **아랍어** | `zero`, `one`, `two`, `few`, `many`, `other` |

```json
// ✅ 한국어 — other만 사용
{
  "followers": "{count, plural, other {팔로워 #명}}"
}

// ✅ 영어
{
  "followers": "{count, plural, one {# follower} other {# followers}}"
}
```

한국어에 `one {팔로워 1명}`을 넣어도 동작하지만, grammatical number 구분이 없으므로 불필요하다.

### 4.2 자주 틀리는 ICU 문법

```
❌ {count-value, plural, ...}   // dash 금지
✅ {count_value, plural, ...}   // underscore OK

❌ {status, select, active {...}}        // other 누락 → 런타임 에러
✅ {status, select, active {...} other {...}}

❌ "auth.signup.title" 내부에 .           // 중첩 표기로 해석됨
✅ "auth": { "signUp": { "title": "..." } }

❌ "Welcome {name"                        // 닫히지 않은 중괄호
✅ "Welcome {name}"

❌ <br>                                   // rich text는 닫는 태그 필요
✅ <br></br>
```

## 5. 공통 함정 (라이브러리 무관)

### 5.1 모든 locale을 초기 번들에 포함

```ts
// ❌ 정적 import — 모든 locale이 번들에 포함됨
import ko from './ko.json'; import en from './en.json'; import ja from './ja.json'

// ✅ 동적 import — locale별 코드 스플리팅
const messages = (await import(`./messages/${locale}.json`)).default
```

### 5.2 Hydration mismatch (SSR)

서버와 클라이언트의 locale/메시지가 달라지면 hydration 에러가 발생한다. 서버에서 번역 리소스를 미리 주입해야 한다 (next-intl v4는 `NextIntlClientProvider`가 자동 상속, react-i18next는 `useSSR()`, react-intl은 `IntlProvider messages=`).

### 5.3 fallback locale 누락

개발 중 번역 누락이 조용히 지나간다. i18next는 `fallbackLng` 필수, react-intl은 `defaultLocale` + `onError`, Lingui는 매크로 원문이 폴백으로 쓰여 비교적 안전.

### 5.4 런타임 Intl 데이터 부족

- **Alpine 컨테이너/small-icu Node**: `Intl.DateTimeFormat('ko-KR')`이 `en-US`로 폴백. `full-icu` 확인 필요
- **React Native Hermes**: `Intl.Locale`/`Intl.PluralRules` 미지원 → FormatJS 폴리필 필수

***

## 6. Server vs Client Component 번역 경계

**메시지를 클라이언트에 내려보내지 말고, 서버에서 번역한 결과를 props로 전달하라.** 번들 크기와 성능 모두 이득이다. Client Component에 번역 context가 꼭 필요하면 **관련 namespace만** 선별해서 Provider에 전달한다 (전체 messages 금지).

## 7. ⚡ Next.js App Router 특화 (next-intl 사용 시)

이 섹션은 Next.js App Router + next-intl v4를 사용하는 경우에만 해당한다. 순수 React 프로젝트는 이 섹션을 건너뛰어도 된다.

### 7.1 Next.js 16 `proxy.ts` — 파일명만 변경

Next.js 16에서 `middleware.ts`가 `proxy.ts`로 이름 변경되었다. **하지만 next-intl의 import 경로와 함수명은 그대로**다.

```ts
// src/proxy.ts (파일명만 proxy.ts)
import createMiddleware from 'next-intl/middleware'  // import 경로 그대로
import { routing } from './i18n/routing'

export default createMiddleware(routing)              // 함수명도 그대로

export const config = {
  matcher: '/((?!api|trpc|_next|_vercel|.*\\..*).*)',
}
```

**Claude가 자주 틀리는 것:**
- ❌ `import createProxy from 'next-intl/proxy'` — 이 경로는 **존재하지 않는다**
- ❌ `proxy.ts`가 Edge runtime 기본 — 실제로는 **Node.js 런타임이 기본**

### 7.2 v3 → v4 Breaking Changes

next-intl v4는 2025-03 릴리스되었고 여러 breaking change가 있다. Claude의 학습 데이터는 v3 패턴을 기억할 가능성이 높다.

#### 7.2.1 `AppConfig` 인터페이스 (타입 안전 번역)

```ts
// ❌ v3 패턴 — v4에서 동작 안 함
declare global {
  interface IntlMessages extends Messages {}
}

// ✅ v4 패턴
import { routing } from '@/i18n/routing'
import messages from '@/messages/en.json'

declare module 'next-intl' {
  interface AppConfig {
    Locale: (typeof routing.locales)[number]
    Messages: typeof messages
  }
}
```

이것만 설정하면 `useTranslations('...')`의 키가 자동완성되고 존재하지 않는 키 사용 시 컴파일 에러가 발생한다.

#### 7.2.2 `getRequestConfig` — `requestLocale` + 명시적 `locale` 반환

```ts
// src/i18n/request.ts
import { getRequestConfig } from 'next-intl/server'
import { routing } from './routing'

export default getRequestConfig(async ({ requestLocale }) => {
  // 1. requestLocale은 Promise — await 필수
  let locale = await requestLocale

  // 2. 유효성 검증 후 fallback
  if (!locale || !routing.locales.includes(locale as any)) {
    locale = routing.defaultLocale
  }

  return {
    locale,  // 🔴 v4에서 반드시 명시 반환 (v3의 암묵적 locale 동작 제거)
    messages: (await import(`../../messages/${locale}.json`)).default,
  }
})
```

**Claude가 자주 틀리는 것:**
- ❌ `async ({ locale }) =>` — v3 패턴, v4에서 deprecated
- ❌ `requestLocale`을 `await` 없이 사용 → Promise 객체가 locale에 할당됨
- ❌ 반환 객체에서 `locale` 생략 → v4 런타임 에러

### 7.3 `setRequestLocale` — 정적 렌더링 필수 조건

**이것을 빼먹으면 모든 route가 dynamic rendering으로 전환된다.** 에러는 안 나지만 `generateStaticParams`를 선언해도 정적 생성이 안 된다.

```tsx
// app/[locale]/layout.tsx
import { setRequestLocale } from 'next-intl/server'
import { routing } from '@/i18n/routing'
import { notFound } from 'next/navigation'

export function generateStaticParams() {
  return routing.locales.map((locale) => ({ locale }))
}

export default async function LocaleLayout({
  children,
  params,
}: {
  children: React.ReactNode
  params: Promise<{ locale: string }>
}) {
  const { locale } = await params
  if (!routing.locales.includes(locale as any)) notFound()

  setRequestLocale(locale)  // 🔴 반드시 next-intl 함수 호출 전에

  return <html lang={locale}><body>{children}</body></html>
}
```

**왜 필요한가:** next-intl이 원래 `headers()`로 locale을 읽는데, 이것이 route를 dynamic rendering으로 opt-in 시킨다. `setRequestLocale`을 호출하면 React `cache()`에 저장되어 이후 함수가 header 대신 캐시된 값을 읽는다 → 정적 렌더링 가능.

**빌드 output에서 확인**: `ƒ (Dynamic)` 마크로 렌더링 타입 확인 가능.

### 7.4 `getTranslations` vs `useTranslations`

| API | 사용처 | 비고 |
|---|---|---|
| `useTranslations('ns')` | **non-async** Server Components, Client Components | hook, 동기 |
| `getTranslations('ns')` | **async** Server Components (await 필요) | awaitable function |

**판단 기준:** 컴포넌트가 `async`인가? → `getTranslations`. 아니면 `useTranslations`. 성능 차이는 없다(config는 요청당 한 번만 로드 + `cache()`).

### 7.5 Cache Components 충돌 — 최신 함정

Next.js 16 `cacheComponents: true` + `'use cache'` 사용 시 next-intl에 알려진 충돌이 있다.

**문제:** `getTranslations()`가 내부적으로 `headers()`를 읽기 때문에 `"headers() inside use cache"` 에러 발생.

**워크어라운드 1 — `locale` 명시적 전달:**

```tsx
async function CachedComponent({ locale }: { locale: string }) {
  'use cache'
  const t = await getTranslations({ locale, namespace: 'MyComponent' })
  // ...
}
```

**워크어라운드 2 — `<Link>` children 패턴:** `<Link>`는 `locale` prop을 명시해도 `'use cache'` 내부에서 **여전히 실패**한다. children으로 받는 패턴으로 우회해야 한다.

```tsx
async function CachedCard({ children }: { children: React.ReactNode }) {
  'use cache'
  return <div>{children}</div>
}

// 사용처 — Link는 캐시 밖에서 생성해서 children으로 전달
<CachedCard>
  <Link href="/about">{t('about')}</Link>
</CachedCard>
```

**미래 해결책:** `next/root-params`가 정식 릴리스되면 해결될 예정(canary 단계). `unstable_rootParams`는 Next.js 16에서 **이미 제거**되었으므로 사용 금지.

> 📎 출처: [next-intl issue #2229](https://github.com/amannn/next-intl/issues/2229)

### 7.6 `'use client'` 컴포넌트에 비직렬화 props 전달 불가

`NextIntlClientProvider`의 `onError`, `getMessageFallback` 같은 함수 props는 Server Component에서 직접 전달 불가다. Client wrapper 컴포넌트를 만들어 그 안에서 전달한다.

## 8. 안티패턴

### 공통

- **하드코딩**: `<button>제출</button>` → `t('common.submit')`
- **날짜 직접 포맷**: `new Date().toLocaleDateString()` → `Intl.DateTimeFormat`
- **모든 locale 정적 import** → 동적 import로 코드 스플리팅
- **TC39 Intl.MessageFormat 대기** — Stage 1, 수년 대기

### Next.js + next-intl

- **v3 global 타입**: `declare global { interface IntlMessages }` → v4 `AppConfig`
- **v3 getRequestConfig**: `async ({ locale })` → v4 `async ({ requestLocale })` + **await** + locale 명시 반환
- **`next-intl/proxy` import** — 존재하지 않는 경로. `next-intl/middleware` 그대로
- **`unstable_rootParams`** — Next.js 16에서 제거됨
- **`setRequestLocale` 생략** — 모든 route가 dynamic
- **`'use cache'` 내부에서 `<Link>`** — locale prop 전달해도 실패, children 패턴 필요

### 버전 오해

- react-intl **v10은 React 19 전용** (18은 v9.x)
- Lingui v5 매크로: `@lingui/macro` → **`@lingui/react/macro`**
- FormatJS 빌드툴: Babel plugin → **`@formatjs/unplugin`**
- i18next: ICU **기본 미지원** → `i18next-icu` 플러그인 필수
- next-i18next: **죽지 않았다** — v16.0.5(2026-04)에서 App Router 지원
