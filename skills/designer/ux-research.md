# UX Research

## 1. 리서치 방법론 매트릭스

| | 정성적 (Qualitative) | 정량적 (Quantitative) |
|---|---|---|
| **태도 (Attitudinal)** | 인터뷰, 포커스 그룹 | 서베이, 카드 소팅 (대규모) |
| **행동 (Behavioral)** | 사용성 테스트, 필드 스터디 | A/B 테스트, 분석 데이터, 클릭맵 |

### 언제 무엇을 사용하나?

**탐색 (Discover)**: 문제 발견, 사용자 이해 → 인터뷰, 필드 스터디, 다이어리 스터디

**정의 (Define)**: 구조, 분류, 우선순위 → 카드 소팅, 트리 테스트, 서베이

**개발 (Develop)**: 디자인 검증, 개선 → 사용성 테스트, A/B 테스트, 프로토타입 테스트

**전달 (Deliver)**: 출시 후 모니터링 → 분석 데이터, NPS, 사용자 피드백

## 2. 사용성 테스트 (Usability Testing)

### 개요

실제 사용자가 실제 태스크를 수행하는 것을 관찰. UX 리서치의 **가장 강력한 도구**.

### 유형

**Moderated (조절된)**

* 진행자가 실시간으로 참여자와 함께
* 심층 탐색 가능, 후속 질문 가능
* 대면 또는 원격 (Zoom, Lookback)
* 참여자: "5명이면 문제의 85% 발견" (Jakob Nielsen)

**Unmoderated (비조절)**

* 참여자가 혼자서 태스크 수행
* 더 많은 참여자, 더 빠른 실행
* 도구: Maze, UserTesting, Lyssna
* 정량적 데이터 (완료율, 시간) 수집 용이

### 사용성 테스트 프로세스

1. **목표 설정**: "결제 플로우의 이탈 원인을 파악한다"
2. **태스크 시나리오 작성**: 구체적이지만 답을 유도하지 않는
   * "생일 선물을 사려고 합니다. 원하는 상품을 찾아 구매해 보세요"
   * "장바구니에 상품을 추가하고 결제 버튼을 클릭하세요" (X)
3. **참여자 모집**: 실제 타겟 사용자 (5-8명)
4. **파일럿 테스트**: 1-2명으로 테스트 자체를 테스트
5. **실행**: Think-aloud 프로토콜. 녹화 동의
6. **분석**: 발견사항 분류, 심각도 평가
7. **보고**: 핵심 발견 + 개선 권고안

### 심각도 분류 (Severity Rating)

| 레벨 | 설명 | 액션 |
|------|------|------|
| 0 | 문제 아님 | - |
| 1 | 미용적 (Cosmetic) | 여유 있을 때 |
| 2 | 경미한 사용성 문제 | 낮은 우선순위 |
| 3 | 주요 사용성 문제 | 높은 우선순위 |
| 4 | 사용 불가 (Catastrophe) | 출시 전 반드시 수정 |

## 3. 사용자 인터뷰 (User Interview)

### 인터뷰 원칙

**The Mom Test (Rob Fitzpatrick)**

* 엄마에게 물어도 유용한 답이 나오는 질문을 하라
* "이 앱 좋아요?" (항상 "좋아"라고 답함) (X)
* "마지막으로 이 문제를 겪었을 때 어떻게 해결하셨어요?" (O)

**질문 유형**

* **Open-ended**: "어떻게 하시나요?", "그때 어떤 느낌이었나요?"
* **Follow-up**: "그건 왜 그렇게 하셨어요?", "좀 더 자세히 말씀해 주세요"
* **피하기**: 유도 질문, Yes/No 질문, 가정적 질문 ("만약 ~한다면 쓰시겠어요?")

### 인터뷰 구조 (60분)

1. **인트로 (5분)**: 소개, 목적 설명, 동의
2. **워밍업 (5분)**: 가벼운 배경 질문
3. **핵심 질문 (35분)**: 행동, 동기, 맥락 탐색
4. **마무리 (10분)**: 추가 의견, 감사
5. **디브리프 (5분)**: 즉시 핵심 인사이트 메모

## 4. 서베이 (Survey)

### 언제 사용하나?

* 대규모 정량 데이터 필요할 때
* 사용자 세그먼트 파악
* 만족도/NPS 측정
* 인터뷰로 발견한 가설의 정량 검증

### 질문 설계 원칙

* 한 번에 하나만 묻기 (Double-barreled 질문 금지)
* 중립적 표현 (유도 금지)
* 척도: Likert 5점 또는 7점
* 필수 질문 최소화 (이탈 방지)
* 10분 이내 완료 가능하게

### NPS (Net Promoter Score)

"이 제품을 동료/친구에게 추천할 의향이 얼마나 되나요?" (0-10)

* 9-10: Promoters
* 7-8: Passives
* 0-6: Detractors
* NPS = %Promoters - %Detractors

## 5. A/B 테스트

### 원칙

* **하나의 변수만 변경**: 버튼 색상 OR 텍스트 OR 위치 (한 번에 하나)
* **충분한 샘플**: 통계적 유의성 필요 (p < 0.05)
* **충분한 기간**: 최소 1-2주 (요일별 편차 제거)
* **명확한 성공 지표**: 전환율, 클릭률, 완료율 등

### A/B 테스트 프로세스

1. 가설 수립: "CTA 버튼 색상을 초록→파랑으로 바꾸면 전환율이 5% 증가할 것이다"
2. 변형 설계: Control (A) vs Variant (B)
3. 트래픽 분배: 50/50 또는 90/10 (리스크 관리)
4. 데이터 수집: 충분한 기간
5. 결과 분석: 통계적 유의성 확인
6. 의사결정: Winner 적용 또는 추가 실험

## 6. 카드 소팅 (Card Sorting)

정보 구조(IA) 설계를 위한 리서치 방법.

### 유형

**Open Sort**: 참여자가 자유롭게 그룹핑 + 카테고리 이름 부여

**Closed Sort**: 미리 정한 카테고리에 카드 분류

**Hybrid**: 카테고리 제공하되 새 카테고리 생성 허용

### 프로세스

1. 콘텐츠/기능 목록을 카드로 작성 (30-60개)
2. 참여자 15-30명 모집
3. 온라인 (OptimalSort, UserZoom) 또는 오프라인 (포스트잇)
4. 결과 분석: 덴드로그램, 유사성 매트릭스
5. IA 초안에 반영

## 7. 트리 테스트 (Tree Testing)

네비게이션 구조의 효과를 검증하는 방법. "Reverse card sorting".

### 프로세스

1. 사이트맵/네비게이션 구조를 텍스트 트리로 작성
2. 태스크 시나리오 작성: "환불 정책을 찾아보세요"
3. 참여자가 트리를 탐색하여 정답 위치 선택
4. 성공률, 직접 성공률, 시간 분석
5. 문제 지점 발견 → IA 개선

## 8. 리서치 결과 전달

### 핵심 원칙

* **발견 → 인사이트 → 권고안**: 데이터 나열이 아닌 해석과 제안
* **스토리텔링**: 사용자의 실제 인용문, 영상 클립 포함
* **우선순위**: 심각도 x 빈도로 순위화
* **Action-oriented**: "개선해야 한다"가 아닌 구체적 "이렇게 개선하라"

### Atomic Research (다니엘 토레스)

리서치 결과를 재사용 가능한 단위로 구조화:

* **Experiment**: 어떤 리서치를 했나
* **Fact**: 관찰한 사실
* **Insight**: 사실에서 도출한 의미
* **Recommendation**: 행동 제안
