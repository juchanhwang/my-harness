# UX Writing

## 1. UX 라이팅이란?

인터페이스의 모든 텍스트 — 버튼 라벨, 에러 메시지, 온보딩 안내, 빈 상태 텍스트, 툴팁 — 를 설계하는 것. 작은 텍스트가 사용자 경험의 성패를 가른다.

---

## 2. 마이크로카피 (Microcopy)

### 원칙

**명확 (Clear)**: 모호함 없이 한 가지 의미만 전달

* "처리" (X) → "결제하기" (O)
* "확인" (X) → "삭제 확인" 또는 "변경사항 저장" (O)

**간결 (Concise)**: 최소한의 단어로 최대한의 의미

* "이 버튼을 클릭하면 계정이 생성됩니다" (X) → "계정 만들기" (O)
* 버튼: 2-4 단어. 설명: 1-2문장.

**유용 (Useful)**: 사용자의 다음 행동을 안내

* 단순히 상태를 알리는 것이 아니라, **무엇을 해야 하는지** 알려주기

**인간적 (Human)**: 기계가 아닌 사람이 말하는 것처럼

* "Invalid credentials" (X) → "이메일 또는 비밀번호가 맞지 않습니다" (O)
* "Error 404" (X) → "찾으시는 페이지가 없습니다" (O)

### 버튼 라벨

**동사로 시작**: 행동을 나타내는 구체적 동사

```
(X) 일반적       (O) 구체적
확인            저장하기
제출            메시지 보내기
OK              계속 진행
취소            되돌리기 / 삭제 취소
```

**맥락 포함**:

* "삭제" → "프로젝트 삭제" (무엇을 삭제하는지)
* "추가" → "팀원 초대" (누구를 추가하는지)

## 3. 에러 메시지

### 좋은 에러 메시지 공식

```
[무엇이 잘못되었는지] + [어떻게 해결하는지]
```

### 예시 모음

| 상황 | 나쁨 | 좋음 |
|------|------|------|
| 필수 필드 미입력 | "필수 항목입니다" | "이메일을 입력해 주세요" |
| 이메일 형식 오류 | "유효하지 않음" | "이메일 형식이 올바르지 않습니다 (예: name@email.com)" |
| 비밀번호 약함 | "비밀번호가 약합니다" | "영문, 숫자를 포함해 8자 이상으로 설정해 주세요" |
| 서버 에러 | "500 Internal Server Error" | "일시적인 문제가 발생했습니다. 잠시 후 다시 시도해 주세요" |
| 네트워크 끊김 | "Network Error" | "인터넷 연결을 확인해 주세요" |
| 권한 없음 | "403 Forbidden" | "이 페이지를 볼 권한이 없습니다. 관리자에게 문의하세요" |
| 파일 크기 초과 | "File too large" | "10MB 이하의 파일만 업로드할 수 있습니다 (현재: 15MB)" |

### 에러 메시지 톤

* **비난하지 않기**: "틀렸습니다" (X) → "확인해 주세요" (O)
* **기술 용어 피하기**: "Timeout", "Null", "Exception" (X)
* **긍정적 표현**: "~할 수 없습니다" → "~하면 됩니다"
* **유머는 신중하게**: 결제 실패에 "Oops!" 는 부적절

## 4. 빈 상태 (Empty State)

사용자가 처음 만나는 화면. 첫인상과 온보딩의 기회.

### 빈 상태 유형

**첫 사용 (First Use)**

```
아직 프로젝트가 없습니다.
첫 번째 프로젝트를 만들어 보세요!

[+ 새 프로젝트 만들기]
```

* 이 기능이 무엇인지 설명
* 가치 제안 전달
* 명확한 CTA

**검색 결과 없음**

```
'디자인 시스템'에 대한 검색 결과가 없습니다.

다른 키워드로 검색하거나, 필터를 조정해 보세요.
[필터 초기화]
```

* 이유 설명
* 대안 제시
* 오타 자동 수정 제안

**작업 완료**

```
모든 할 일을 완료했습니다!
오늘도 수고하셨습니다.
```

* 축하/격려
* 다음 단계 안내 (필요 시)

**에러로 인한 빈 화면**

```
데이터를 불러오는 데 문제가 발생했습니다.

[다시 시도]
```

* 원인 간략 설명
* 재시도 옵션

## 5. 온보딩 텍스트

### 온보딩 유형

**기능 투어 (Feature Tour)**

* 3-5단계 이내
* 각 단계 한 가지 기능만 소개
* "건너뛰기" 항상 제공
* 진행 인디케이터 표시

**Tooltip 가이드**

* 실제 UI 요소를 가리키며 설명
* 컨텍스트에 맞는 짧은 설명
* 순차적 또는 온디맨드

**Progressive Onboarding**

* 사용자 행동에 따라 점진적 안내
* "첫 번째 프로젝트를 만들었군요! 이제 팀원을 초대해 보세요"
* 가장 자연스럽고 효과적

### 온보딩 카피 원칙

* **혜택 중심**: 기능이 아니라 사용자에게 주는 가치
  * "실시간 동기화 기능" (X) → "모든 기기에서 항상 최신 상태를 유지하세요" (O)
* **짧게**: 한 화면에 1-2문장
* **행동 유도**: 각 단계 끝에 구체적 행동 제안
* **스킵 가능**: 강제하지 않음

## 6. 톤앤매너 (Tone of Voice)

### 톤 스펙트럼

| 축 | 한쪽 | 반대쪽 |
|---|---|---|
| 격식 | 격식체 ("~합니다") | 비격식체 ("~해요") |
| 태도 | 진지함 | 유머러스 |
| 열정 | 차분함 | 열정적 |
| 존중 | 겸손 | 자신감 |

### 상황별 톤 조절

| 상황 | 톤 | 예시 |
|------|---|------|
| 성공 | 따뜻하고 축하하는 | "환영합니다! 모든 준비가 완료되었어요" |
| 경고 | 차분하고 정보 전달 | "저장하지 않은 변경사항이 있습니다" |
| 에러 | 공감적이고 해결 지향 | "문제가 발생했습니다. 다시 시도해 주세요" |
| 파괴적 액션 | 신중하고 명확 | "이 프로젝트를 삭제하면 복구할 수 없습니다" |
| 로딩/대기 | 가볍고 인내심 | "거의 다 됐어요..." |

### Voice Chart 작성법

프로덕트의 성격을 4가지 특성으로 정의:

```
We are:          But not:
─────────        ─────────
친근한           가벼운
전문적           딱딱한
도움이 되는       참견하는
자신감 있는       거만한
```

## 7. 확인 다이얼로그 (Confirmation Dialog)

### 구조

```
[제목]: 구체적 행동 설명
[본문]: 결과 설명 (되돌릴 수 있는지)
[Primary 버튼]: 확인 행동 (구체적 동사)
[Secondary 버튼]: 취소
```

### 예시

```
프로젝트 "디자인 시스템"을 삭제할까요?

이 프로젝트의 모든 파일과 히스토리가 영구적으로 삭제됩니다.
이 작업은 되돌릴 수 없습니다.

[프로젝트 삭제]  [취소]
(destructive)   (secondary)
```

**안티패턴**:

* 제목: "확인" / 버튼: "확인" "취소" (무엇을 확인하는지 불명) (X)
* 이중 부정: "삭제를 취소하시겠습니까?" (X)

## 8. 접근성 관련 텍스트

### 대체 텍스트 (Alt Text)

* 정보 전달 이미지: 내용 설명 (`alt="3월 매출 그래프: 전월 대비 15% 증가"`)
* 장식 이미지: 빈 alt (`alt=""`)
* 링크 내 이미지: 링크 목적지 (`alt="트위터에서 팔로우"`)
* 아이콘 버튼: `aria-label="닫기"`, `aria-label="검색"`

### 스크린 리더 전용 텍스트

```html
<span class="sr-only">새 알림 3개</span>
```

---

## 9. 안티패턴

* **Lorem Ipsum 출시**: 실제 텍스트 없이 디자인 확정
* **"여기를 클릭하세요"**: 링크 텍스트에 맥락 없음 (접근성 위반)
* **과도한 전문용어**: "Sync failed" 대신 사용자 언어로
* **일관성 없는 용어**: 같은 기능을 "저장/세이브/보관" 혼용
* **패시브 공격적 톤**: "정말로 나가시겠습니까?" (guilt trip)
