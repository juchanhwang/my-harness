# Design Leadership

## 1. 디자인 문화 구축

### 디자인 성숙도 모델 (InVision Design Maturity Model)

**Level 1: Producers** — 디자인 = 예쁘게 만들기. 프로세스 후반에 참여.

**Level 2: Connectors** — 디자인이 다른 팀과 연결. 패턴/가이드 존재.

**Level 3: Architects** — 디자인이 비즈니스 전략에 통합. 디자인 시스템 구축.

**Level 4: Scientists** — 데이터 기반 디자인. 지속적 실험과 측정.

**Level 5: Visionaries** — 디자인이 비즈니스 혁신을 주도. 조직 전체의 디자인 사고.

### 디자인 문화의 핵심 요소

**사용자 중심주의**

* 모든 의사결정에서 "사용자에게 어떤 영향이 있는가?" 질문
* 정기적 사용자 리서치 (월 최소 1회)
* 사용성 테스트 결과 팀 전체 공유

**심리적 안전**

* 아이디어를 자유롭게 공유할 수 있는 환경
* "나쁜 아이디어"를 두려워하지 않기
* 실패에서 배우는 문화 (blame-free retrospective)

**투명한 의사결정**

* 디자인 결정의 근거 문서화 (DDR)
* 크리틱 세션에서 열린 피드백
* 결정 과정에 팀원 참여

**지속적 학습**

* 디자인 북클럽/스터디
* 외부 컨퍼런스/밋업 참여
* 내부 지식 공유 세션

## 2. 디자인 시스템 거버넌스

### 거버넌스 모델

**Centralized (중앙 집중형)**

* 전담 디자인 시스템 팀이 관리
* 일관성 높음, 병목 위험
* 대규모 조직에 적합

**Federated (연합형)**

* 각 팀에서 기여, 코어 팀이 조율
* 유연하고 스케일 가능
* 중규모 조직에 가장 권장

**Distributed (분산형)**

* 모든 팀이 공동 소유
* 빠른 의사결정, 일관성 위험
* 소규모/초기 조직

### 컨트리뷰션 프로세스

```
1. Proposal (제안)
   └── Issue 생성: 새 컴포넌트/수정 요청 + 유스케이스 설명

2. Review (검토)
   └── DS 팀 검토: 기존으로 해결 가능? / 범용적인가? / 우선순위

3. Design (설계)
   └── Figma에서 설계 + 디자인 크리틱

4. Build (구현)
   └── 코드 구현 + 코드 리뷰 + 테스트

5. Document (문서화)
   └── Storybook + 사용 가이드 + Do/Don't

6. Release (배포)
   └── Semantic versioning + Changelog + 공지
```

### "시스템에 없는 것이 필요할 때" 의사결정 트리

```
필요한 컴포넌트가 시스템에 없다
    │
    ├── 기존 컴포넌트 조합으로 해결 가능?
    │   └── Yes → 기존 활용 (새로 만들지 않음)
    │
    ├── 프로젝트 특수한 것인가?
    │   └── Yes → 로컬 컴포넌트 (프로젝트 내)
    │
    └── 다른 프로젝트에서도 쓸 수 있는가?
        └── Yes → 디자인 시스템에 추가 제안
```

## 3. 멘토링

### 디자이너 성장 프레임워크

**Junior (0-2년)**

* Focus: 기본 스킬 (시각 디자인, 도구 숙련)
* 지도: 구체적 태스크, 페어 디자인, 코드 리뷰 스타일 크리틱
* 성장 포인트: 디자인 원칙 이해, 피드백 수용, 기본 리서치

**Mid-level (2-5년)**

* Focus: 문제 정의, 프로세스 주도
* 지도: 방향성 제시, 자율성 부여, 전략적 사고 훈련
* 성장 포인트: 독립적 프로젝트 리드, 크리틱 참여, 사용자 리서치 설계

**Senior (5년+)**

* Focus: 전략, 영향력, 멘토링
* 지도: 코칭 (답을 주지 않고 질문), 조직적 영향력
* 성장 포인트: 디자인 전략, 팀 문화, 이해관계자 관리

### 1:1 미팅 구조

```
주간 1:1 (30분)
├── Check-in (5분): 기분, 에너지 레벨
├── 현재 프로젝트 (10분): 진행 상황, 막히는 점
├── 성장 (10분): 배운 점, 도전, 피드백
└── 액션 아이템 (5분): 다음 주 목표
```

### 피드백 문화

**SBI (Situation-Behavior-Impact) 모델**

* "지난 크리틱에서 (Situation)"
* "에러 메시지 UX에 대한 깊은 분석을 제시했는데 (Behavior)"
* "팀이 에러 플로우를 전면 개선하게 되었다 (Impact)"

**성장형 피드백**

* 강점 기반: "이것이 잘 됐으니까, 여기에도 같은 접근을 적용해 볼까?"
* 구체적: "전반적으로 좋아요" (X) → "이 유저 플로우에서 에러 케이스 처리가 특히 좋았다" (O)
* 시의적: 가능한 빨리 (크리틱 직후, 리뷰 직후)

## 4. Design Ops

디자인 팀의 운영 효율을 높이는 실천. 개발의 DevOps처럼.

### Design Ops 영역

**People Ops**

* 채용/온보딩 프로세스
* 스킬 매트릭스/성장 프레임워크
* 팀 구조 (embedded vs centralized)

**Workflow Ops**

* 디자인 프로세스 표준화
* 도구 선정 및 관리
* 템플릿, 체크리스트
* 파일 구조/네이밍 컨벤션

**Craft Ops**

* 디자인 시스템 거버넌스
* 디자인 QA 프로세스
* 리서치 레포지토리 관리
* 디자인 에셋 관리

### 디자인 팀 메트릭

| 메트릭 | 측정 대상 | 방법 |
|--------|----------|------|
| Design System Adoption | 시스템 컴포넌트 사용률 | 코드 분석 |
| Design Iteration Speed | 첫 디자인 → 핸드오프 기간 | 프로젝트 추적 |
| Usability Score | 사용성 테스트 태스크 성공률 | 정기 테스트 |
| Developer Handoff Quality | 핸드오프 관련 질문/이슈 수 | 이슈 트래커 |
| User Satisfaction | NPS, CSAT, SUS 점수 | 서베이 |

## 5. 이해관계자 관리

### 디자인 제안 프레젠테이션

**구조**:

1. **문제**: "사용자가 겪는 문제는..."
2. **증거**: "리서치/데이터에 따르면..."
3. **솔루션**: "이 디자인은..."
4. **트레이드오프**: "이 접근법의 장단점은..."
5. **다음 단계**: "검증을 위해..."

**팁**:

* 이해관계자의 관심사에 맞게 프레임 (CEO=비즈니스 임팩트, 개발리드=기술적 실현성)
* 데이터와 사용자 인용으로 주장 뒷받침
* 대안도 보여주되, 왜 이 안을 추천하는지 근거 제시
* "이것이 예뻐서"는 근거가 아님. "이것이 전환율을 높일 것이기 때문에"가 근거

### RACI 매트릭스 (의사결정 구조)

| | 디자이너 | PM | 개발자 | CEO |
|---|---|---|---|---|
| UX 패턴 결정 | R/A | C | C | I |
| 비주얼 스타일 | R/A | I | I | C |
| 기능 범위 | C | R/A | C | C |
| 기술 구현 | C | I | R/A | I |
| 출시 결정 | I | R | C | A |

R=Responsible, A=Accountable, C=Consulted, I=Informed

## 6. 디자인 팀 의식 (Rituals)

| 의식 | 빈도 | 목적 |
|------|------|------|
| 디자인 크리틱 | 주 1-2회 | 피드백, 품질 향상 |
| 디자인 쇼앤텔 | 격주 | 팀 간 가시성, 영감 |
| 디자인 스프린트 | 프로젝트 시작 시 | 빠른 방향 검증 |
| 사용성 테스트 | 월 1회+ | 사용자 인사이트 |
| 디자인 시스템 싱크 | 격주 | 시스템 업데이트 |
| 회고 | 프로젝트 완료 후 | 프로세스 개선 |
| 디자인 북클럽 | 월 1회 | 학습, 성장 |

## 7. 안티패턴

* **디자인 경찰**: 규칙 강제만 하고 지원하지 않음
* **아이보리 타워**: 디자인팀이 격리되어 현실과 괴리
* **한 사람의 취향**: 리더의 개인 취향이 시스템이 됨
* **프로세스 과잉**: 모든 것에 5단계 승인 프로세스
* **측정 없는 디자인**: "좋은 디자인"의 기준이 주관적
