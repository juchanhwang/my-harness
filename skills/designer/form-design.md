# Form Design

## 1. 폼 디자인의 중요성

폼은 사용자와 시스템 간의 대화다. 회원가입, 결제, 검색, 설정 — 거의 모든 프로덕트의 핵심 인터랙션이 폼을 통해 이루어진다. 나쁜 폼은 전환율을 죽이고, 좋은 폼은 보이지 않게 작동한다.

---

## 2. 레이아웃 원칙

### Single Column Layout

* 폼은 **단일 열(single column)**이 기본. 다중 열은 스캔 패턴을 깨뜨린다
* 예외: 짧은 관련 필드 (이름 + 성, 시/도 + 우편번호)
* Luke Wroblewski 연구: 단일 열이 다중 열보다 완료율 높음

### 라벨 위치

| 위치 | 장점 | 단점 | 사용 |
|------|------|------|------|
| 상단 (Top) | 가장 빠른 완료율, 반응형 친화적 | 수직 공간 차지 | **기본 권장** |
| 인라인 (Placeholder) | 공간 절약 | 입력 시 라벨 사라짐 → 맥락 손실 | **단독 사용 금지** |
| Floating Label | 공간 절약 + 라벨 유지 | 구현 복잡, 작은 라벨 가독성 | 선택적 사용 |
| 좌측 (Left) | 긴 폼에서 스캔 효율 | 라벨-입력 간 시선 이동 | 관리 화면 |

### 라벨 규칙

* **명확한 라벨**: "이름" (O), "필드 1" (X)
* **Placeholder는 라벨 대체 불가**: 힌트/예시로만 사용. "예: kim@email.com"
* **Required/Optional 표시**: 필수 필드가 대부분이면 선택 필드에 "(선택)" 표시

## 3. 입력 필드 디자인

### 필드 유형 선택

| 데이터 | 최적 필드 | 이유 |
|--------|----------|------|
| 짧은 텍스트 | Text Input | 단순, 직관적 |
| 긴 텍스트 | Textarea | 여러 줄 입력 |
| 이메일/전화 | 전용 Input (type) | 모바일 키패드 최적화 |
| 날짜 | Date Picker | 형식 오류 방지 |
| 2-5개 옵션 | Radio Button | 모든 옵션이 보임 |
| 6+ 옵션 | Select/Dropdown | 공간 절약 |
| On/Off | Toggle Switch | 즉시 적용 |
| 복수 선택 | Checkbox Group | 독립적 선택 |
| 범위 값 | Slider | 직관적 범위 조정 |

### 입력 필드 크기

* 예상되는 입력 길이에 맞는 너비 (우편번호: 짧게, 주소: 길게)
* 높이: 최소 40px (터치 타겟 고려)
* 패딩: 수평 12-16px, 수직 8-12px

### 입력 보조

* **Autocomplete**: 브라우저 자동완성 활성화 (`autocomplete="email"`)
* **Input Mask**: 전화번호, 카드번호 등 형식 자동 적용
* **Character Counter**: 최대 길이 있을 때 실시간 카운트
* **Default Value**: 가장 일반적인 값을 기본값으로

## 4. 유효성 검증 UX

### 검증 시점

| 시점 | 설명 | 권장도 |
|------|------|--------|
| 제출 후 (Submit) | 전송 버튼 클릭 후 검증 | 기본 |
| 포커스 아웃 (Blur) | 필드 떠날 때 검증 | 최적 |
| 실시간 (Keyup) | 타이핑마다 검증 | 신중하게 |
| 하이브리드 | Blur + Submit 조합 | 권장 |

### 최적 전략: Reward Early, Punish Late

* **첫 번째 시도**: 제출 시 또는 blur 시 검증 (너무 빨리 에러 보여주지 않음)
* **에러 수정 중**: 실시간 검증 (빨리 성공 피드백)
* 즉, 처음엔 관대하게, 에러 수정 시엔 즉각 피드백

### 인라인 검증 패턴

```
[라벨]
[입력 필드 ─────────────── ✓]  ← 성공 시 체크 아이콘
 힌트 텍스트 (회색)

[라벨]
[입력 필드 ─────────────── ✗]  ← 에러 시 빨간 보더
 에러 메시지 (빨간색)
```

## 5. 에러 메시지

### 좋은 에러 메시지의 3요소

1. **무엇이 잘못되었는지** (문제 설명)
2. **왜 잘못되었는지** (원인)
3. **어떻게 고칠 수 있는지** (해결책)

### 예시

| 나쁨 | 좋음 |
|------|------|
| "유효하지 않은 입력" | "이메일 형식이 올바르지 않습니다. 예: name@email.com" |
| "에러" | "비밀번호는 8자 이상이어야 합니다 (현재 5자)" |
| "필수 필드" | "이름을 입력해 주세요" |
| "서버 에러" | "잠시 후 다시 시도해 주세요. 문제가 계속되면 고객센터에 문의하세요" |

### 에러 메시지 톤

* **비난하지 않기**: "잘못 입력했습니다" (X) → "이메일 형식을 확인해 주세요" (O)
* **기술 용어 피하기**: "Validation failed" (X) → "입력을 확인해 주세요" (O)
* **유머는 신중하게**: 에러 상황에서 유머는 짜증을 유발할 수 있음

### 에러 표시 위치

* **인라인**: 해당 필드 바로 아래 (최적)
* **요약**: 폼 상단에 전체 에러 목록 (긴 폼에서 추가)
* 에러 필드로 자동 스크롤 + 포커스

## 6. Progressive Disclosure (점진적 공개)

### 원칙

모든 정보를 한 번에 보여주지 않는다. 사용자가 필요할 때 추가 옵션을 노출.

### 패턴

**Multi-step Form (스텝 폼)**

```
Step 1: 기본 정보 → Step 2: 상세 정보 → Step 3: 확인/완료
[●──○──○] 프로그레스 인디케이터
```

* 각 단계 3-5개 필드
* 이전/다음 버튼
* 완료율 표시 (프로그레스 바)

**Conditional Fields**

* "배송 주소가 청구 주소와 다른가요?" → [Yes] → 추가 필드 표시
* "기타"를 선택하면 텍스트 입력 필드 등장

**Accordion/Expandable Sections**

* 고급 설정, 선택적 정보를 접을 수 있는 섹션에
* 기본값이 충분한 경우 접힌 상태로 시작

**Show More / Advanced**

* "고급 옵션" 링크로 추가 필드 노출
* 대부분의 사용자는 열지 않아도 됨

## 7. 폼 제출 (Submission)

### 버튼

* **라벨**: 구체적 행동 동사 사용. "제출" (X) → "계정 만들기" (O)
* **위치**: 폼 하단 좌측 (LTR) 또는 마지막 필드 바로 아래
* **Primary 하나만**: 주요 액션 = Primary Button, 취소 = Ghost/Link
* **Loading 상태**: 클릭 후 스피너 + disabled + "처리 중..."
* **중복 제출 방지**: 클릭 즉시 비활성화

### 저장/자동 저장

* **긴 폼**: 자동 저장 (Draft) 제공. "마지막 저장: 2분 전"
* **중요한 폼**: 페이지 이탈 시 확인 다이얼로그
* **제출 후**: 명확한 성공 피드백 + 다음 단계 안내

## 8. 접근성 (Form Accessibility)

### 필수 사항

* 모든 입력 필드에 연결된 `<label>` (또는 `aria-label`)
* 에러 메시지를 `aria-describedby`로 필드와 연결
* 에러 발생 시 `aria-invalid="true"`
* Required 필드에 `aria-required="true"` (시각적 * 외에도)
* Tab 순서가 시각적 순서와 일치
* 에러 요약 영역에 `role="alert"` 또는 `aria-live="polite"`

---

## 9. 안티패턴

* **불필요한 필드**: 수집할 이유 없는 데이터를 요구 (성별, 팩스 번호)
* **Placeholder as Label**: 입력 시작하면 라벨 사라짐
* **Reset Button**: 실수로 모든 입력 삭제. 거의 항상 불필요
* **CAPTCHA 남용**: 사용자 경험 파괴. 보이지 않는 CAPTCHA 사용
* **비밀번호 제한**: "특수문자 반드시 포함" + "16자 이내" 같은 불합리한 제한
* **Select for 2 options**: 옵션이 2개면 Radio Button 사용
