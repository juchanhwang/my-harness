# User Flows

## 1. 유저 플로우란?

사용자가 특정 목표를 달성하기 위해 제품 내에서 거치는 경로의 시각적 표현. 화면(screens), 결정(decisions), 액션(actions)의 연속.

---

## 2. 유저 플로우 구성 요소

### 기본 기호

```
[○] 시작/종료 (터미널)
[□] 화면/페이지 (프로세스)
[◇] 결정/분기 (Decision)
[→] 이동 방향 (Flow)
[▭] 시스템 액션 (이메일 발송 등)
```

### 예시: 회원가입 플로우

```
[○ 시작]
    │
[□ 랜딩 페이지]
    │
[□ 회원가입 폼]
    │
[▭ 이메일 인증 발송]
    │
[□ 이메일 확인 안내]
    │
◇ 이메일 인증 완료?
├─ Yes → [□ 프로필 설정]
│              │
│        [□ 온보딩 튜토리얼]
│              │
│        [○ 대시보드 (완료)]
│
└─ No → [□ 재발송 안내]
              │
         ◇ 재시도?
         ├─ Yes → [▭ 이메일 재발송]
         └─ No → [○ 이탈]
```

## 3. Task Analysis (태스크 분석)

유저 플로우 설계 전, 사용자의 태스크를 체계적으로 분석.

### Hierarchical Task Analysis (HTA)

태스크를 하위 태스크로 분해:

```
0. 온라인으로 피자 주문하기
├── 1. 메뉴 탐색
│   ├── 1.1 카테고리 선택
│   ├── 1.2 피자 상세 확인
│   └── 1.3 커스터마이징 (토핑, 사이즈)
├── 2. 장바구니 관리
│   ├── 2.1 수량 조정
│   └── 2.2 쿠폰 적용
├── 3. 배달 정보 입력
│   ├── 3.1 주소 입력/선택
│   └── 3.2 배달 시간 선택
├── 4. 결제
│   ├── 4.1 결제 수단 선택
│   └── 4.2 결제 완료
└── 5. 주문 확인
    ├── 5.1 주문 확인 화면
    └── 5.2 주문 추적
```

### GOMS 분석

Goals, Operators, Methods, Selection rules.

* 각 태스크의 인지/물리적 단계를 분석
* 클릭 수, 시선 이동, 인지 부하 측정
* 효율성 최적화에 사용

## 4. Happy Path vs Edge Cases

### Happy Path (주요 경로)

모든 것이 순조롭게 진행되는 이상적인 시나리오.

* 사용자가 올바른 정보를 입력
* 시스템이 정상 응답
* 결제가 성공

**반드시 먼저 설계하고 최적화**

### Edge Cases (엣지 케이스)

예상 밖 또는 극단적 상황. 무시하면 실제 사용자 경험이 망가짐.

#### 데이터 엣지 케이스

* **빈 데이터**: 검색 결과 0건, 알림 없음, 활동 내역 없음
* **과도한 데이터**: 1000개 항목 리스트, 매우 긴 텍스트
* **특수 문자**: 이름에 특수문자, 이모지, RTL 언어
* **경계값**: 재고 0개 직전, 잔액 정확히 상품 가격

#### 사용자 엣지 케이스

* **중복 제출**: 결제 버튼 연타
* **뒤로 가기**: 결제 완료 후 뒤로 가기
* **탭 전환**: 폼 작성 중 다른 탭 갔다 오기
* **오프라인**: 네트워크 끊김 상황
* **권한 없음**: 로그아웃 상태로 보호된 페이지 접근
* **동시 편집**: 두 사용자가 같은 데이터 수정

#### 시스템 엣지 케이스

* **서버 에러**: 500, timeout
* **느린 응답**: 3G 네트워크, 해외 사용자
* **결제 실패**: 카드 거절, 잔액 부족
* **외부 서비스 장애**: OAuth, 결제 게이트웨이

## 5. 에러 플로우 (Error Flows)

### 에러 플로우 설계 원칙

1. **예방 > 치료**: 에러가 발생하기 전에 방지 (유효성 검증, 확인 다이얼로그)
2. **명확한 설명**: 무엇이 잘못되었고, 어떻게 해결하는지
3. **복구 경로**: 에러에서 빠져나올 수 있는 명확한 경로
4. **데이터 보존**: 에러 발생 시 사용자가 입력한 데이터 유지

### 에러 플로우 예시: 결제 실패

```
[□ 결제 정보 입력]
    │
[▭ 결제 처리 중] (로딩)
    │
◇ 결제 성공?
├─ Yes → [□ 결제 완료 / 주문 확인]
│
└─ No → ◇ 에러 유형?
         ├─ 카드 거절 →   [□ 인라인 에러 "카드가 거절되었습니다.
         │                    다른 결제 수단을 사용하거나
         │                    카드사에 문의해 주세요"]
         │                    [다른 카드 사용] [카드사 문의]
         │
         ├─ 네트워크 →    [□ 재시도 안내
         │                    "일시적인 오류입니다.
         │                     잠시 후 다시 시도해 주세요"]
         │                    [재시도] [나중에]
         │
         └─ 시스템 에러 → [□ 일반 에러 페이지
                              "문제가 발생했습니다.
                               고객센터: xxx-xxxx"]
                              [홈으로] [고객센터]
```

## 6. 상태별 화면 (Screen States)

모든 화면은 최소 5가지 상태를 고려해야 한다:

### 5 States of UI

| 상태 | 설명 | 디자인 필요 |
|------|------|-----------|
| **Empty** | 데이터 없음 (첫 사용) | 일러스트 + 안내 + CTA |
| **Loading** | 데이터 로딩 중 | Skeleton, Spinner |
| **Partial** | 일부 데이터만 있음 | 빈 섹션 처리 |
| **Ideal** | 정상적인 양의 데이터 | 기본 디자인 |
| **Error** | 오류 발생 | 에러 메시지 + 복구 경로 |
| **Overflow** | 데이터 과다 | Pagination, 가상 스크롤 |

### Empty State 디자인

```
┌─────────────────────────────┐
│                             │
│      [일러스트/아이콘]       │
│                             │
│    아직 프로젝트가 없습니다    │  ← 상태 설명
│  첫 번째 프로젝트를 만들어     │  ← 행동 유도
│      보세요!                  │
│                             │
│    [+ 새 프로젝트 만들기]     │  ← CTA
│                             │
└─────────────────────────────┘
```

## 8. 유저 플로우 리뷰 체크리스트

* [ ] Happy path가 최적화되어 있는가? (최소 단계)
* [ ] 모든 결정 포인트에서 양쪽 경로가 설계되었는가?
* [ ] Empty, Loading, Error 상태가 포함되었는가?
* [ ] 뒤로 가기, 취소, 실행 취소 경로가 있는가?
* [ ] 권한/인증 관련 분기가 고려되었는가?
* [ ] 엣지 케이스가 식별되고 처리되었는가?
* [ ] 사용자가 어떤 시점에서든 "탈출구"가 있는가?
