# Design Critique

## 1. 디자인 크리틱이란?

디자인의 품질을 높이기 위한 구조화된 피드백 세션. "좋다/나쁘다"의 주관적 감상이 아닌, **목표와 원칙에 근거한 분석적 평가**.

### 크리틱 vs 피드백 vs 리뷰

| | 크리틱 | 피드백 | 리뷰 |
|---|---|---|---|
| 목적 | 디자인 개선 | 의견 공유 | 승인/반려 |
| 구조 | 높음 (규칙 있음) | 낮음 (자유) | 중간 |
| 참여자 | 디자인 팀 | 누구나 | 이해관계자 |
| 결과 | 개선 방향 | 다양한 의견 | Go/No-go |

## 2. 효과적인 크리틱 구조

### 세션 구조 (30-45분)

**1. Context Setting (5분)**

* 프레젠터가 배경 설명
* "이 디자인이 해결하려는 문제는..."
* "타겟 사용자는..."
* "이번 크리틱에서 집중적으로 봐주셨으면 하는 부분은..."

**2. Presentation (5-10분)**

* 디자인 워크스루 (유저 플로우 순서)
* 주요 디자인 결정과 근거 설명
* 질문은 프레젠테이션 후

**3. Clarifying Questions (5분)**

* 이해를 위한 질문만 (평가 아직 아님)
* "이 버튼은 어떤 상태에서 보이나요?"
* "모바일에서는 어떻게 되나요?"

**4. Structured Feedback (15-20분)**

* 참여자별 돌아가며 피드백
* 각 피드백에 근거 제시
* 프레젠터는 기록만 (방어 금지)

**5. Summary & Next Steps (5분)**

* 핵심 피드백 요약
* 액션 아이템 정리
* 다음 크리틱 일정

### 크리틱 규칙

1. **디자인을 비평하지, 디자이너를 비평하지 않는다**
2. **근거를 제시한다**: "사용자가 혼란스러울 것 같다" (X) → "Hick's Law에 따르면 7개 옵션은 결정 시간을 증가시킨다" (O)
3. **질문으로 시작한다**: "이 레이아웃을 선택한 이유는?" → 프레젠터가 스스로 문제 발견
4. **"나는" 대신 "사용자는"**: 개인 취향 배제
5. **솔루션 강요하지 않기**: 문제 식별까지만. 해결은 디자이너의 몫

## 3. 피드백 주는 법

### 구조화된 피드백 프레임워크

**I Like, I Wish, What If**

* **I Like**: 잘 작동하는 부분과 그 이유
* **I Wish**: 개선이 필요한 부분과 그 이유
* **What If**: 대안 아이디어 제안

**Situation-Behavior-Impact (SBI)**

* **Situation**: "로그인 화면에서..."
* **Behavior**: "에러 메시지가 폼 상단에만 표시되는데..."
* **Impact**: "사용자가 어떤 필드에 문제가 있는지 스크롤해서 찾아야 합니다"

### 좋은 피드백 vs 나쁜 피드백

| 나쁜 피드백 | 좋은 피드백 |
|-----------|-----------|
| "이거 좀 별로" | "CTA 버튼의 시각적 무게가 주변 요소와 비슷해서 눈에 띄지 않습니다" |
| "색상이 안 예뻐" | "이 색상 조합은 WCAG AA 대비 비율 3.2:1로 기준 미달입니다" |
| "나라면 이렇게 했을 텐데" | "사용자가 이 단계에서 뒤로 갈 방법이 없는데, 이탈 가능성이 높아 보입니다" |
| "전체적으로 좋아요" | "카드 간 간격이 일관적이고, 시각적 위계가 명확해서 스캔하기 좋습니다" |

### 피드백 분류

| 카테고리 | 설명 | 우선순위 |
|---------|------|---------|
| 사용성 문제 | 사용자가 태스크를 완료하지 못하는 문제 | 반드시 수정 |
| 개선 사항 | 사용 가능하지만 더 좋아질 수 있는 부분 | 권장 |
| 탐색적 제안 | 다른 접근법, "What if" 아이디어 | 검토 |
| 잘 된 부분 | 유지/강화해야 할 강점 | 유지 |

## 4. 피드백 받는 법

### 프레젠터의 마인드셋

1. **디자인 =/= 자아**: 내 작품이 아닌 팀의 산출물
2. **방어하지 않기**: 피드백 수집 모드. 반박은 나중에
3. **명확화 질문**: "그 부분 좀 더 구체적으로 말씀해 주시겠어요?"
4. **메모하기**: 모든 피드백 기록. 반영 여부는 크리틱 후 결정
5. **감사하기**: 시간 내서 봐준 것 자체가 가치

### 피드백 처리 프레임워크

모든 피드백에 대해:

1. **이해**: 피드백의 의도와 맥락 파악
2. **분류**: 사용성/접근성/비주얼/콘텐츠/기술적
3. **평가**: 사용자 데이터/원칙으로 타당성 판단
4. **결정**: 반영/보류/반려 + 이유 기록
5. **소통**: 결정과 근거를 피드백 제공자에게 공유

## 5. 디자인 의사결정 문서화

### Design Decision Record (DDR)

디자인 버전의 ADR(Architecture Decision Record).

```markdown
## DDR-001: 검색 결과 필터 위치

### 날짜
2024-01-15

### 상태
승인됨

### 맥락
검색 결과 페이지에서 필터를 어디에 배치할지 결정 필요.
사용성 테스트에서 5명 중 3명이 필터를 찾지 못함.

### 검토한 옵션
1. **좌측 사이드바** — 데스크톱 전통적 패턴. 모바일에서 공간 부족
2. **상단 수평** — 발견성 높음. 필터 수 제한
3. **상단 + 바텀시트** — 데스크톱은 상단, 모바일은 바텀시트

### 결정
옵션 3: 상단 필터바 + 모바일 바텀시트

### 근거
- 사용성 테스트에서 상단 배치 시 발견율 100%
- 모바일 바텀시트는 thumb zone 접근 최적
- NNGroup 연구: 수평 필터가 사이드바보다 모바일에서 효과적

### 결과
- 데스크톱: 상단에 주요 필터 4개 + "더 보기"
- 모바일: 상단에 필터 버튼 → 바텀시트
- 구현 복잡도 증가 (반응형 두 가지 패턴)
```

### 언제 DDR을 쓰는가?

* 두 가지 이상 유효한 대안이 있을 때
* 팀 내 의견이 갈릴 때
* 나중에 "왜 이렇게 했지?"라고 물을 것 같을 때
* 사용성 테스트 결과에 기반한 결정

## 6. 비동기 크리틱 (Async Critique)

리모트 팀을 위한 비동기 피드백 방법.

### Figma 코멘트

* 특정 프레임/요소에 직접 코멘트
* 카테고리 태그: [사용성], [접근성], [비주얼], [제안]
* @멘션으로 담당자 지정
* Resolve로 처리 완료 표시

### Loom + Figma

* 3-5분 비디오로 디자인 워크스루
* 비디오 보며 비동기 코멘트
* 시간대 다른 팀에 효과적

### 구조화된 양식

```markdown
## 피드백 양식

**리뷰어**: [이름]
**디자인**: [링크]

### 잘 작동하는 부분
- ...

### 우려되는 부분
- ...
- (이유/근거: ...)

### 질문
- ...

### 아이디어
- ...
```

## 7. 크리틱 빈도와 타이밍

| 프로젝트 단계 | 크리틱 유형 | 빈도 |
|-------------|-----------|------|
| 탐색/아이디에이션 | 방향성 크리틱 | 주 1-2회 |
| 와이어프레임 | 구조 크리틱 | 주 1회 |
| 비주얼 디자인 | 디테일 크리틱 | 주 1회 |
| 개발 중 | 디자인 QA | 필요 시 |
| 출시 후 | 회고 | 프로젝트 완료 후 |
