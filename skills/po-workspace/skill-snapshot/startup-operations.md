# Startup Operations

## 초기 스타트업 PO의 역할 (0→1)

### 스타트업 PO vs 대기업 PM

|           | 스타트업 PO (0→1)  | 대기업 PM (1→100) |
| --------- | -------------- | -------------- |
| **범위**    | 제품+사업+운영 전부    | 특정 제품/기능 영역    |
| **의사결정**  | 빠르게, 직감+데이터 혼합 | 데이터 중심, 합의 필요  |
| **리소스**   | 극도로 제한         | 상대적 풍부         |
| **실패 비용** | 회사 생존과 직결      | 기능 수준의 실패      |
| **고객 거리** | 매일 직접 대화       | 리서치팀 통해 간접     |
| **역할 정의** | 모호하고 유동적       | 명확한 R\&R       |

### 0→1 PO의 핵심 책임

```
1. Problem-Market Fit 확인
   → "이 문제가 진짜 존재하는가?"
   → "이 문제를 가진 사람이 충분한가?"

2. Solution-Market Fit 확인
   → "우리 솔루션이 이 문제를 해결하는가?"
   → "기존 대안보다 나은가?"

3. Product-Market Fit 달성
   → "사용자가 제품을 사랑하는가?"
   → "유기적 성장이 발생하는가?"
```

## MVP 정의와 Scope 관리

### MVP란?

> "최소한의 노력으로 최대한의 학습을 얻는 제품" — Eric Ries

MVP는 **"최소 기능 제품"이 아니라 "최소 학습 제품"**이다.

### MVP Scope 결정법

**1. 핵심 가설 식별**

```
우리의 가장 큰 가정은 무엇인가?
→ 이 가정을 검증하기 위해 최소한으로 필요한 것은?
```

**2. One Thing 원칙**

```
이 제품이 하나만 잘 해야 한다면 그것은 무엇인가?
→ 나머지는 V2, V3으로 미룬다
```

**3. Scope Hammer**

```
모든 기능에 대해 물어본다:
"이것 없이도 핵심 가설을 검증할 수 있는가?"
→ Yes면 빼라
```

### MVP 유형

| 유형                 | 설명           | 소요 시간 | 적합한 상황       |
| ------------------ | ------------ | ----- | ------------ |
| **Concierge**      | 수동으로 서비스 제공  | 1-2주  | 서비스형 제품      |
| **Wizard of Oz**   | 자동화인 척 수동 처리 | 2-4주  | 기술 검증 불필요 시  |
| **Landing Page**   | 가치 제안 + CTA만 | 1-3일  | 수요 검증        |
| **Video MVP**      | 제품 설명 영상     | 1주    | 복잡한 제품 개념 검증 |
| **Single Feature** | 핵심 기능 1개만    | 4-8주  | 기술 제품        |
| **Piecemeal**      | 기존 도구 조합     | 1-2주  | 워크플로우 검증     |

### Scope Creep 방지

```
방지 원칙:
1. MVP scope를 문서화하고 팀 전원이 합의
2. 새 요구사항은 별도 리스트에 기록 (V2 backlog)
3. Scope 변경 시 반드시 "무엇을 빼는가?" 함께 결정
4. "있으면 좋겠다"는 전부 V2
5. 주간으로 scope 점검: "우리 원래 뭘 검증하려 했지?"
```

## Customer Development (Steve Blank)

### 4단계 프로세스

```
Customer     Customer     Customer     Company
Discovery → Validation → Creation  → Building
(문제 발견)  (솔루션 검증) (수요 창출)  (조직 구축)

Search ←────────────────→ Execute
(탐색 단계)                (실행 단계)
```

### Customer Discovery

**목표**: 우리의 가설이 현실에 부합하는가?

```
1. Problem Interview (문제 인터뷰)
   → 이 문제가 실제로 존재하는가?
   → 얼마나 심각한가?
   → 현재 어떻게 해결하고 있는가?

2. Solution Interview (솔루션 인터뷰)
   → 우리 솔루션이 문제를 해결하는가?
   → 돈을 내겠는가? 얼마나?
   → 누가 의사결정자인가?
```

### Customer Validation

**목표**: 반복 가능한 세일즈 프로세스가 존재하는가?

```
- 실제로 구매하는 고객이 있는가? (LOI, pre-order)
- 세일즈 사이클은 얼마나 걸리는가?
- CAC와 LTV가 합리적인가?
- 같은 방법으로 다른 고객에게도 팔 수 있는가?
```

## Pivot 결정 기준

### Pivot이 필요한 신호

```
🚨 강한 신호:
- 3개월 이상 retention curve가 개선되지 않음
- PMF 설문에서 "매우 실망" 비율이 30% 미만 (정체)
- CAC가 LTV를 초과하며 개선 추세 없음
- 핵심 가설이 반복적으로 반증됨

⚠️ 약한 신호:
- 성장이 둔화 (다른 원인일 수 있음)
- 경쟁사가 빠르게 성장 (우리의 실행 문제일 수 있음)
- 팀 사기 저하 (리더십 문제일 수 있음)
```

### Pivot 유형 (Eric Ries)

| Pivot 유형             | 설명              | 예시                       |
| -------------------- | --------------- | ------------------------ |
| **Customer segment** | 다른 고객에게 같은 제품   | Slack (게임 → 기업)          |
| **Customer need**    | 같은 고객의 다른 문제    | PayPal (PDA → 이메일 결제)    |
| **Platform**         | 앱 → 플랫폼 (또는 반대) |                          |
| **Business model**   | 수익 모델 변경        |                          |
| **Channel**          | 유통 채널 변경        |                          |
| **Technology**       | 같은 문제, 다른 기술    |                          |
| **Zoom-in**          | 기능 → 전체 제품      | Instagram (Burbn의 사진 기능) |
| **Zoom-out**         | 제품 → 더 큰 제품의 기능 |                          |

### Pivot 결정 프로세스

```
1. 데이터 수집 (2주)
   → 현재 지표 정밀 분석
   → 고객 인터뷰 집중 실시
   → 팀 내부 솔직한 평가

2. 옵션 도출 (1주)
   → Persevere (현재 방향 유지)
   → Pivot (방향 전환) — 구체적 방향 3개
   → Stop (중단)

3. 검증 계획 (1주)
   → 각 옵션의 핵심 가설
   → 2-4주 내 검증 가능한 실험

4. 결정 (1일)
   → 경영진 + PO + 핵심 팀원
   → 명확한 결정 + 커뮤니케이션
```

## 리소스 제약 하의 우선순위

### "할 수 없는 것"을 인정하기

```
스타트업의 현실:
- 엔지니어 3명으로 모든 것을 할 수 없다
- 한 분기에 1-2개의 big bet만 가능
- 완벽한 제품은 나올 수 없다 — 완벽한 학습이 목표

원칙:
1. 하나에 집중 (Focus)
2. 완벽이 아닌 충분히 좋은 (Good enough)
3. 자동화보다 수동 (Do things that don't scale)
4. 기술보다 학습 (Learning > Building)
```

### 초기 스타트업 리소스 배분

```
PMF 전:
  80% Discovery + MVP 개발
  10% 기술 기반 (최소한)
  10% 운영/기타

PMF 후:
  50% 핵심 제품 강화
  30% Growth / Activation
  20% 기술 기반 + 운영
```

## Fundraising 관점에서의 제품 전략

### 투자자가 보는 것

| 단계           | 투자자 관심사        | 제품에서 증명해야 할 것             |
| ------------ | -------------- | ------------------------- |
| **Pre-Seed** | 팀, 시장 크기, 문제   | 문제가 실제로 존재한다              |
| **Seed**     | 초기 트랙션         | PMF 신호, 초기 사용자 반응         |
| **Series A** | PMF 증거, 성장 잠재력 | Retention, Unit economics |
| **Series B** | 스케일 가능성        | 성장률, 효율성, TAM 확장          |

### 제품 마일스톤과 펀딩 연결

```
Pre-Seed → Problem validation (고객 인터뷰 50+)
  ↓
Seed → MVP + Initial traction (100+ 사용자, 초기 retention)
  ↓
Series A → PMF evidence (NRR>100%, 유기적 성장)
  ↓
Series B → Scalable growth (3x YoY, healthy unit economics)
```

### 투자자 관점 제품 메트릭

```
"당신의 매출이 아니라 당신의 retention을 보겠습니다"
"Logo 수가 아니라 cohort별 behavior를 보겠습니다"
"기능 수가 아니라 사용자가 제품에서 얻는 가치를 보겠습니다"

핵심: 투자자는 과거 성과가 아닌 미래 성장 가능성을 봄
→ Leading indicator가 Lagging보다 중요
```
