# Stakeholder Management

## Stakeholder Mapping

### Influence × Interest Matrix

```
Influence (높음)
    │
    │  Manage Closely    Keep Satisfied
    │  (높은 영향력,       (높은 영향력,
    │   높은 관심)         낮은 관심)

    │   → 의사결정 참여     → 큰 변화 시 사전 공유
    │
    │  Keep Informed     Monitor
    │  (낮은 영향력,       (낮은 영향력,
    │   높은 관심)         낮은 관심)
    │   → 정보 공유        → 최소한의 소통
    │   → 피드백 수집      → 상황 변화 시 재평가
    │
    └──────────────────────────────
              Interest (높음)
```

## 기대치 관리

### 기대치 관리의 원칙

1. **일찍, 자주 소통한다** — 나쁜 소식은 빨리 전할수록 좋다
2. **약속은 보수적으로, 전달은 초과 달성** — Under-promise, over-deliver
3. **불확실성을 숨기지 않는다** — "모르겠습니다. 확인 후 공유하겠습니다"
4. **트레이드오프를 명시한다** — "A를 하면 B를 못 합니다"
5. **정기적으로 정렬한다** — 생각이 달라지기 전에 확인

### 흔한 기대치 불일치

| 상황       | PO 관점      | 이해관계자 관점 | 해결법                           |
| -------- | ---------- | -------- | ----------------------------- |
| 로드맵      | 방향과 의도     | 확정된 일정   | "이것은 계획이지 약속이 아닙니다"           |
| 기능 요청    | 우선순위 하위    | 당장 필요    | 데이터로 트레이드오프 설명                |
| 일정       | 예상이며 변동 가능 | 확정 날짜    | 범위를 유연하게, confidence level 공유 |
| "간단한 기능" | 숨은 복잡도     | 쉬워 보임    | Iceberg 비유 + 구체적 작업 분해        |

## Conflict Resolution

### 갈등 해결 5단계

```
1. 경청 (Listen First)
   → 상대방의 관점을 완전히 이해할 때까지
   → "제가 이해한 바로는 ~인데, 맞나요?"

2. 공통 목표 확인 (Common Ground)
   → "우리 모두 사용자에게 더 좋은 제품을 원하잖아요"
   → 대립이 아닌 같은 편

3. 데이터로 프레이밍 (Data-Based)
   → 의견 대 의견 → 데이터 대 데이터로 전환
   → "데이터를 보면..."

4. 옵션 제시 (Options)
   → "두 가지 방법이 있습니다: A는 ~, B는 ~"
   → 상대방에게 선택권 부여

5. 결정 & 기록 (Decide & Document)
   → 합의된 내용 문서화
   → 결정 근거와 다음 스텝 명확히
```

### PO가 자주 만나는 갈등 상황

**Engineering "기술 부채가 급하다" vs PO "기능이 급하다"** → 해결: Sprint 용량의 고정 비율(20%)을 tech debt에 할당하는 원칙 합의

**Sales "고객이 이 기능 없으면 안 산다" vs PO "로드맵에 없다"** → 해결: 해당 기능의 수요 규모(1 고객 vs N 고객), 전략 정합성 평가

**경영진 "왜 이걸 안 하나?" vs PO "데이터상 다른 게 더 중요"** → 해결: RICE/ICE 스코어링 결과 공유, 트레이드오프 명시

## 커뮤니케이션 주기 설계

### 정기 커뮤니케이션 스케줄

| 대상               | 형식           | 주기    | 내용                |
| ---------------- | ------------ | ----- | ----------------- |
| 경영진              | 1:1 또는 보고서   | 격주-월간 | 핵심 지표, 전략 정렬, 리스크 |
| Engineering Lead | 1:1          | 주간    | 기술적 결정, 의존성, 블로커  |
| Design Lead      | 1:1 또는 Sync  | 주간    | UX 방향, 디자인 리뷰     |
| Sales/CS         | 이메일/슬랙       | 월간    | 로드맵 업데이트, 고객 피드백  |
| 전체 팀             | All-hands/데모 | 격주    | 진행 상황, 성과, 방향     |
| 외부 이해관계자         | 보고서/프레젠테이션   | 분기    | 성과, 계획            |

## Executive Presentation 기법

### 경영진 보고의 원칙

1. **Bottom Line Up Front (BLUF)**: 결론부터, 배경은 나중에
2. **숫자로 말하기**: "좋아졌다"가 아닌 "+15%"
3. **So What?**: 모든 데이터에 "그래서 뭘 해야 하는가?" 연결
4. **시간 존중**: 짧고 명확하게, 디테일은 Q&A나 부록

### 경영진 보고 구조 (5분)

```
1. 핵심 메시지 (30초)
   → "이번 분기 activation rate를 15% 개선했고,
      다음 분기는 retention에 집중합니다"

2. 성과 (2분)
   → 핵심 지표 변화 (차트 1-2개)
   → 주요 출시 항목과 결과

3. 학습 (1분)
   → 실험에서 배운 것
   → 고객 인사이트

4. 계획 (1분)
   → 다음 분기 방향
   → 필요한 의사결정/지원

5. 질문 (30초)
   → "결정이 필요한 사항은 X입니다"
```

## "No" 전달 기술 (대안 제시)

### "No"의 4단계 프레임워크

```
1. Acknowledge (인정)
   → "좋은 포인트입니다. 이 기능이 필요한 맥락을 이해합니다."

2. Explain (설명)
   → "현재 데이터를 보면, [다른 영역]이 더 큰 임팩트를 만들 수 있습니다."

3. Alternative (대안)
   → "대신 [대안 A]를 먼저 시도해 보면 어떨까요?
      이것으로 요청하신 문제의 70%를 해결할 수 있습니다."

4. Timeline (시점)
   → "요청하신 기능은 Q3 로드맵 리뷰 시 다시 평가하겠습니다."
```

### 상황별 "No" 스크립트

**고객 요청:** "이 피드백 감사합니다. 현재 이 영역을 개선하기 위한 더 근본적인 솔루션을 탐색 중입니다. 진행 상황이 있으면 공유드리겠습니다."

**영업팀 요청:** "이 고객의 니즈를 이해합니다. 이 기능의 수요가 얼마나 넓은지 파악하기 위해 다른 고객들에게도 확인해 보겠습니다. 5건 이상 같은 요청이 있으면 우선순위를 재검토하겠습니다."

**경영진 요청:** "데이터를 보면 현재 [A]에 집중하는 것이 NSM에 3배 큰 영향을 줍니다. [요청 사항]은 [A] 완료 후 바로 다음 우선순위로 올리겠습니다. 순서를 바꾸시려면 [A]의 지연 영향을 함께 검토해야 합니다."
