# A/B Testing 가이드 전문

## A/B 테스트란?

"두 가지 이상의 변형(Variant)을 실제 사용자에게 동시에 노출하여, 어떤 것이 더 나은 결과를 만드는지 **통계적으로 검증**하는 방법."

## A/B 테스트 설계

### 1. 가설 수립

실험을 시작하기 전에 다음 요소들을 명확히 정의해야 합니다:

- **현재 상태**: 기준 데이터
- **변경 사항**: 수정 내용
- **기대 효과**: 지표 변화 예측
- **측정 지표**: 주요 + 보조 + 보호 지표

**구체적 사례:**
- 현재: 가입 후 첫 프로젝트 생성 전환율 30%
- 변경: 가입 직후 템플릿 선택 화면 추가
- 기대: 전환율 33% 상대적 개선
- Primary: 전환율 / Secondary: 7일 유지율 / Guardrail: 가입 완료율

### 2. 변수 설정

| 요소 | 의미 |
|------|------|
| 독립변수 | 우리가 수정하는 요소 |
| 종속변수 | 측정 결과 |
| 대조군 | 현재 버전 |
| 처리군 | 변경 버전 |
| 외부변수 | 결과에 영향 주는 요인들 |

### 3. 지표 분류

**Primary**: "이 지표가 개선되면 성공"

**Secondary**: "함께 개선되면 더 좋다"

**Guardrail**: "이 지표가 떨어지면 실험 중단"

## 통계적 유의성

### 핵심 개념

| 개념 | 설명 | 기준값 |
|------|------|--------|
| p-value | "차이가 우연일 확률" | <0.05 |
| 통계적 유의성 | 결과가 우연이 아닌 정도 | 95% 신뢰도 |
| 통계적 검정력 | 실제 차이 감지 확률 | 80% 이상 |
| MDE | 감지 대상의 최소 차이 | 비즈니스 의미 수준 |
| 효과 크기 | 실제 차이의 규모 | 클수록 샘플 절감 |

### 샘플 크기 계산

필수 정보:
- 기존 전환율
- 최소 감지 가능 효과(MDE)
- 신뢰도(일반적으로 95%)
- 통계적 검정력(일반적으로 80%)

**예시 계산:**
기존 10% 전환율에서 10% 상대 개선 목표(10→11%)를 감지하려면 신뢰도 95%, 검정력 80% 기준 약 14,700명 per 그룹이 필요합니다. 일일 트래픽 1,000명 기준 약 30일 소요.

### 실험 기간 결정

필수 샘플 크기를 일일 트래픽으로 나누되, 추가로 고려할 사항:

- "2주 미만 실험은 요일 편차로 왜곡될 수 있다"
- 최소 1-2주 추가 실행
- 사용자의 행동 완료까지 소요 시간 포함
- "결과가 보기 좋아서" 조기 종료 금지
- p-value가 0.05 근처에서 변동하면 더 오래 실행

## Multi-Armed Bandit

### A/B 테스트 vs Bandit

| 항목 | A/B 테스트 | Multi-Armed Bandit |
|------|-----------|-------------------|
| 트래픽 분배 | 고정 분배 | 성과에 따라 동적 조정 |
| 목적 | 학습 | 최적화 |
| 기간 | 미리 결정된 고정 기간 | 유동적 |
| 적합성 | 명확한 학습 필요 시 | 기회비용 최소화 필요 시 |

### Bandit 적용 상황

낮은 트래픽으로 인한 장시간 실험이 필요한 경우, 시간 제약이 있는 프로모션/이벤트 상황, 성능 차이가 명확한 상황에서 우수한 변형에 더 많은 트래픽 할당이 필요할 때 적합합니다.

## Feature Flag 기반 실험

### 활용 패턴

1. **Kill Switch**: 문제 발생 시 즉시 비활성화
2. **Gradual Rollout**: 단계적 확대(1% → 5% → 25% → 50% → 100%)
3. **A/B Test**: 실험 그룹 분리
4. **Beta Access**: 특정 사용자/조직 대상 공개
5. **Ops Toggle**: 운영 환경 제어

### 지원 도구

전문 솔루션(LaunchDarkly, Unleash, Flagsmith), 분석 플랫폼 통합(PostHog, Amplitude), 초기 스타트업용 자체 구현.

## 실험 결과 해석

### 의사결정 매트릭스

| Primary 개선 | Guardrail 유지 | 결정 |
|-------------|--------------|------|
| 유의적 ✓ | ✓ | 출시 |
| 유의적 ✓ | 악화 ✗ | 재검토 |
| 유의하지 않음 ✗ | ✓ | 실패 |
| 유의하지 않음 ✗ | 악화 ✗ | 즉시 중단 |

### 주의사항

1. **심슨의 역설**: 전체 데이터와 세분화된 데이터에서 상반된 결과 → 반드시 세그먼트별 분석
2. **참신성 효과**: 새로운 것에 대한 일시적 관심 → 2주 이상 실행
3. **초두 효과**: 기존 사용자의 변화 저항 → 신규 사용자 코호트 분석
4. **다중 검증 문제**: "여러 지표 동시 관찰 시 하나쯤은 우연히 유의할 수 있다" → Bonferroni 보정
5. **검정력 부족**: 샘플 부족 ≠ 차이 없음

```

원래 가설

```

## 실험 문화 구축

### 기본 원칙

1. **실패도 성공**: "가설이 틀린 것도 학습"
2. **데이터 중심**: 직관보다 데이터 우선
3. **소규모 고속**: 대규모 장기 프로젝트보다 단기 실험
4. **모든 기능은 검증 대상**: 확신 있는 기능도 측정
5. **결과 공유**: 성공과 실패 모두 팀 전체 공유

### 실험 주기

- **주간**: 1-2개 진행 중
- **월간**: 결과 리뷰 회의
- **분기**: 학습 내용 정리 및 다음 방향 수립
