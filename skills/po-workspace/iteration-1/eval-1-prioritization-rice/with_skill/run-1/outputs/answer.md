# Q1 기능 우선순위 분석 — RICE 스코어링

## 컨텍스트 및 가정

- **제품**: B2B 프로젝트 관리 SaaS
- **현재 MAU**: 15,000
- **리소스**: 2명 엔지니어 × 3개월 = **6 person-months**
- **분석 기준**: Q1 (3개월) 기간
- **가정 A**: MAU 15,000 중 실제 주간 활성 사용자(WAU) ≈ 60% → 9,000명 (B2B SaaS 특성상 업무일 기준)
- **가정 B**: B2B 특성상 의사결정 단위는 팀/조직이므로, Reach는 기능별로 실제 영향받는 사용자 수를 개별 추정
- **가정 C**: 분기당 Reach = 해당 기능을 실제로 사용/영향받을 MAU 비율 적용

---

## RICE 공식

```
RICE Score = (Reach × Impact × Confidence) / Effort
```

| 요소 | 단위 | 척도 |
|---|---|---|
| **Reach** | 분기당 영향받는 사용자 수 | 실제 사용자 수 |
| **Impact** | 개인에게 미치는 영향 | 3=massive, 2=high, 1=medium, 0.5=low, 0.25=minimal |
| **Confidence** | 추정의 확신도 | 100%=높음, 80%=보통, 50%=낮음 |
| **Effort** | 필요한 작업량 | person-months |

## 기능별 RICE 분석

### 1. 다크 모드

**Reach 추정: 15,000명**
- 다크 모드는 모든 현재 MAU가 UI 레벨에서 영향을 받음
- B2B 사용자도 화면을 직접 보는 사용자 전원이 대상
- **Reach = 15,000**

**Impact 추정: 0.5 (low)**
- 다크 모드는 개인 선호도 기능으로 핵심 업무 흐름을 바꾸지 않음
- B2B SaaS에서 다크 모드가 이탈률, 전환율, 생산성에 직접적으로 미치는 영향은 낮음
- 만족도 소폭 향상에 그치며, 유료 전환이나 리텐션에 대한 기여 미미
- 참고: `prioritization.md`의 다크 모드 예시에서도 Impact = 0.5 (low)로 명시
- **Impact = 0.5**

**Confidence 추정: 90%**
- 사용자 요청이 일반적으로 많고, 구현 결과가 예측 가능함
- 기술적 복잡도도 낮아 예상 노력이 빗나갈 가능성 낮음
- **Confidence = 90% = 0.9**

**Effort 추정: 1 person-month**
- 프론트엔드 CSS 변수 작업, 시스템 설정 연동 수준
- 엔지니어 1명이 3~4주면 완성 가능 (B2B 앱 기준, 테스트 포함)
- **Effort = 1 person-month**

**RICE 계산:**
```
RICE = (15,000 × 0.5 × 0.9) / 1
     = 6,750 / 1
     = 6,750
```

### 2. Slack/Discord 통합

**Reach 추정: 10,500명**
- 현재 B2B SaaS 사용자 중 Slack/Discord 같은 메신저를 업무에 사용하는 비율 추정
- B2B SaaS 업계 Slack 채택률은 기업 규모에 따라 다르나 SMB~Mid-market 기준 약 60~80%
- **가정**: MAU 15,000 중 70%가 Slack 또는 Discord 사용 → 10,500명
- **Reach = 10,500**

**Impact 추정: 2 (high)**
- 알림/업데이트를 이미 쓰는 채널(Slack/Discord)로 받으면 앱 접속 빈도↑, 작업 완료율↑
- B2B SaaS에서 통합 기능은 팀 협업 핵심 워크플로우에 직접 연결되어 리텐션에 큰 영향
- Slack 통합은 B2B SaaS에서 세일즈 단계에서도 자주 요구되는 기능으로 전환율 영향 있음
- **Impact = 2**

**Confidence 추정: 80%**
- Slack API는 잘 문서화되어 있어 기술적 확신은 높음
- 그러나 어떤 알림/이벤트를 통합할지, 사용자가 실제로 설정해서 쓸지에 대한 채택률은 확실하지 않음
- **Confidence = 80% = 0.8**

**Effort 추정: 2 person-months**
- OAuth 통합, 웹훅 구현, 알림 로직, 설정 UI, 테스트 필요
- 엔지니어 1명 기준 약 2개월 또는 2명이 1개월 집중 작업
- **Effort = 2 person-months**

**RICE 계산:**
```
RICE = (10,500 × 2 × 0.8) / 2
     = 16,800 / 2
     = 8,400
```

### 3. 칸반 보드 커스텀 필드

**Reach 추정: 9,000명**
- 칸반 보드를 사용하는 사용자가 대상 (전체 MAU 중 WAU 기준)
- 프로젝트 관리 SaaS에서 칸반 보드는 핵심 기능이므로 활성 사용자 대부분이 사용 → 60%
- **가정**: 15,000 × 60% = 9,000명
- **Reach = 9,000**

**Impact 추정: 2 (high)**
- 커스텀 필드는 B2B 고객의 워크플로우 맞춤화를 가능하게 하여 tool stickiness 대폭 향상
- 현재 자신의 프로세스에 맞지 않아 이탈하는 사용자들의 핵심 이유 중 하나
- 고객 세일즈 cycle에서 "커스텀 필드 없나요?"는 단골 질문 → 직접 전환율 영향
- **Impact = 2**

**Confidence 추정: 80%**
- 기술적으로 유연한 스키마 구현이 필요하며 복잡도가 있음
- B2B에서 커스텀 필드가 리텐션을 높인다는 것은 업계 사례로 검증됨
- 그러나 어떤 필드 타입을 우선할지, UX 설계의 복잡성은 변수
- **Confidence = 80% = 0.8**

**Effort 추정: 3 person-months**
- 데이터 모델 변경, 필드 타입별 UI (텍스트/숫자/날짜/드롭다운 등), 필터/정렬 연동, 마이그레이션
- 엔지니어 2명이 약 1.5개월 → 3 person-months
- **Effort = 3 person-months**

**RICE 계산:**
```
RICE = (9,000 × 2 × 0.8) / 3
     = 14,400 / 3
     = 4,800
```

### 4. 모바일 앱 (iOS)

**Reach 추정: 6,000명**
- B2B SaaS에서 모바일 앱은 "이동 중 확인" 용도로 사용하는 비율은 상대적으로 낮음
- **가정**: MAU의 40%가 모바일로도 접근하고 싶어하는 사용자 → 6,000명
- 단, 실제로 모바일 앱을 설치하고 활성 사용하게 될 비율은 더 낮을 수 있음 (보수적 추정)
- **Reach = 6,000**

**Impact 추정: 1 (medium)**
- 이동 중 빠른 확인과 알림은 편리함을 주지만, 핵심 작업(프로젝트 계획, 복잡한 편집)은 여전히 데스크탑에서 수행
- 리텐션에 기여하지만 모바일 앱 존재만으로 새로운 사용자를 획득하거나 이탈을 막기는 제한적
- 모바일 최적화된 웹앱으로 일부 커버 가능한 영역
- **Impact = 1**

**Confidence 추정: 50%**
- iOS 앱 개발은 기술적 확실성은 있으나 사용자 채택률, 사용 패턴에 대한 불확실성 높음
- B2B SaaS에서 모바일 앱이 핵심 지표(리텐션, 전환)에 얼마나 기여하는지 검증 데이터 부족
- 앱 스토어 심사, 인증 플로우, 플랫폼별 UX 등 예상치 못한 effort 증가 가능성
- **Confidence = 50% = 0.5**

**Effort 추정: 5 person-months**
- iOS 앱 신규 개발: 설계, 인증, 핵심 기능 구현, API 연동, 테스트, 앱 스토어 등록
- 엔지니어 2명이 약 2.5개월 → 5 person-months
- 리소스의 83%를 소비하는 프로젝트
- **Effort = 5 person-months**

**RICE 계산:**
```
RICE = (6,000 × 1 × 0.5) / 5
     = 3,000 / 5
     = 600
```

### 5. API v2 공개 (개발자 대상)

**Reach 추정: 3,000명**
- B2B SaaS에서 API를 직접 사용하는 개발자 또는 기술 담당자 비율은 상대적으로 낮음
- **가정**: MAU의 20%가 개발자/기술 담당자로 API를 활용할 수 있는 사용자 → 3,000명
- 단, API를 통해 integration 생태계가 만들어지면 간접적으로 더 많은 사용자에게 영향 가능 (중기 효과)
- **Reach = 3,000** (직접 영향 기준, Q1 범위)

**Impact 추정: 3 (massive)**
- B2B SaaS에서 공개 API는 파트너/통합 생태계를 만드는 전략적 자산
- Enterprise 영업 사이클에서 "API 있나요?"는 deal breaker 수준의 질문
- 타사 도구와 통합이 가능해지면 lock-in 효과가 강해져 장기 리텐션 및 확장 매출에 크게 기여
- 개발자 커뮤니티의 입소문(word of mouth) 효과로 신규 유입에도 기여
- **Impact = 3**

**Confidence 추정: 70%**
- API 설계 및 구현의 기술적 복잡도는 예측 가능
- 그러나 v2라는 점에서 기존 v1과의 하위 호환성, 문서화, 개발자 onboarding이 추가 effort를 요구
- 실제 채택률과 생태계 형성 타임라인은 불확실 (Q1 내 임팩트는 제한적, 중장기 효과)
- **Confidence = 70% = 0.7**

**Effort 추정: 4 person-months**
- API 설계, 인증(API key/OAuth), rate limiting, 문서(OpenAPI spec), SDK 기초, 보안 리뷰, 버저닝
- 엔지니어 2명이 약 2개월 → 4 person-months
- **Effort = 4 person-months**

**RICE 계산:**
```
RICE = (3,000 × 3 × 0.7) / 4
     = 6,300 / 4
     = 1,575
```

그러나 단순 RICE 순위 상위 2개 선택은 다크 모드의 낮은 전략적 가치 문제를 포함한다. 아래 정성적 분석 참조.

## 최종 선택: 상위 2개 기능

### 선택 1: Slack/Discord 통합 (RICE: 8,400)

**선택 이유:**
- RICE 스코어 1위 (8,400)로 정량적으로 명확히 1순위
- **Reach × Impact가 높음**: 10,500명 × Impact 2 = 업무 핵심 워크플로우에 직접 연결
- B2B SaaS에서 Slack 통합은 **리텐션과 액티베이션 동시 개선** — 알림을 통해 앱으로 돌아오는 빈도 증가
- **세일즈 enablement 효과**: "Slack 통합 있나요?"는 Enterprise 영업에서 자주 등장하는 질문. Q1에 출시하면 바로 세일즈에 활용 가능
- **Effort 2 PM**: 리소스 대비 ROI가 가장 높음
- 2명이 각 1개월 집중하면 완성 가능하여 팀 운영상 현실적

**근거 지표 연결**: AARRR 기준 Retention 단계에 직접 기여 (앱 → Slack 알림 → 재방문 루프)

---

### 선택 2: 칸반 보드 커스텀 필드 (RICE: 4,800)

**왜 RICE 2위(다크 모드, 6,750) 대신 RICE 3위(커스텀 필드, 4,800)를 선택하는가?**

RICE 스코어만 보면 다크 모드(6,750)가 2위다. 그러나 **`prioritization.md`가 명시하는 대로 "전략적 중요도는 RICE에 반영되지 않으므로 별도 고려"**가 필요하다.

정성적 조정 이유:

| 기준 | 다크 모드 | 커스텀 필드 |
|---|---|---|
| North Star와의 연결 | 간접적 (만족도) | 직접적 (핵심 워크플로우 stickiness) |
| B2B 핵심 가치 | 낮음 (편의 기능) | 높음 (맞춤화 = 이탈 방지) |
| 전환율 영향 | 미미 | 세일즈 사이클에서 요구 사항으로 등장 |
| 경쟁 차별화 | 없음 (모두 있음) | 중간 (차별화 요소) |
| 리소스 활용 | 1 PM (과소 활용) | 3 PM (적절) |

**커스텀 필드 선택 이유:**
- B2B SaaS에서 **tool stickiness의 핵심**: 자신의 워크플로우에 맞는 도구를 떠나기 어려움
- **이탈 방지 직결**: "우리 프로세스에 안 맞아서 다른 툴 써야겠다"는 이탈 이유 1위
- Effort 3 PM으로 Slack 통합(2 PM)과 합산하면 총 5 PM → 6 PM 내에 완료 가능
- 남은 1 PM으로 QA, 버그 수정, 문서화 여유 확보
- 다크 모드(1 PM) 대신 선택 시 **비즈니스 임팩트 차이가 압도적**

**리소스 배분:**
```
Slack/Discord 통합:  2 PM (엔지니어 2명 × 1개월)
칸반 커스텀 필드:    3 PM (엔지니어 2명 × 1.5개월)
QA/버퍼:            1 PM
────────────────────
합계:                6 PM (한도 내)
```

## 제외 기능 및 이유

### 제외 1: 다크 모드 (RICE: 6,750 → Q1 제외)

RICE 스코어는 2위이지만 Q1 우선순위에서 제외한다.

**제외 이유:**
- **Impact = 0.5 (low)**: Reach는 높지만 개인에게 미치는 영향이 최소 수준. 다크 모드가 없다고 이탈하거나, 있다고 전환하는 B2B 사용자는 극히 드뭄
- **전략적 비기여**: B2B SaaS의 핵심 지표인 리텐션, 전환율, 팀 활성화와 연결이 약함
- **편의 기능**: "있으면 좋은 것(nice-to-have)"으로 핵심 업무 문제를 해결하지 않음
- **Opportunity Score 관점**: 다크 모드에 대한 사용자 Importance는 중간이나, Satisfaction은 브라우저 OS 설정으로 어느 정도 해결되는 상황. 실제 기회 점수 낮음
- **향후 계획**: Q2~Q3 여유 스프린트에서 처리하거나, 엔지니어 1명이 단독 1주 내 처리할 수 있는 분기 이후 배치

---

### 제외 2: API v2 공개 (RICE: 1,575)

**제외 이유:**
- **Reach = 3,000 (낮음)**: Q1 기간 내 직접 혜택을 받는 사용자가 전체의 20%에 불과
- **Confidence = 70%**: 생태계 형성, 개발자 채택, 실제 integration 완성까지 Q1 내 임팩트가 불확실
- **중기 투자**: API v2의 전략적 가치는 높지만 효과가 나타나는 시점이 Q2~Q3 이후
- **Effort 4 PM**: 리소스의 67%를 소비하면서 Q1 내 눈에 보이는 사용자 지표 개선이 제한적
- **선결 조건**: 공개 API는 내부 API가 안정적이어야 하며, 문서화·보안 리뷰·개발자 지원 체계가 선행되어야 함. 6 PM 제약 하에서 이 모두를 Q1에 처리하면 품질 리스크

**향후 계획**: Q2에 커스텀 필드 데이터 모델 안정화 후 API v2로 연결하면 시너지 (커스텀 필드 API도 v2에 포함 가능)

### 제외 3: 모바일 앱 iOS (RICE: 600)

**제외 이유:**
- **RICE 최하위 (600)**: 모든 요소가 불리하게 작용
- **Effort = 5 PM**: 6 PM 제약에서 혼자 83% 소비. 선택 시 다른 모든 기능 불가
- **Confidence = 50%**: B2B SaaS에서 모바일 앱 채택률과 리텐션 기여도가 낮을 가능성. 리서치 먼저 필요
- **우선순위 원칙 위반**: `prioritization.md`의 "Confidence 50% 미만이면 먼저 리서치/실험"에 해당. 모바일 PWA(Progressive Web App) 실험으로 수요를 먼저 검증하는 것이 올바른 순서
- **B2B 사용 패턴**: 프로젝트 관리 SaaS의 핵심 작업(간트 차트, 복잡한 편집, 팀 배치)은 대화면에서 이루어짐. 모바일은 알림 확인/빠른 업데이트 수준

**향후 계획**: Q1에 모바일 웹 UX 개선(반응형 최적화)을 병행하고, Q3~Q4에 모바일 앱 수요 데이터 기반으로 재평가
