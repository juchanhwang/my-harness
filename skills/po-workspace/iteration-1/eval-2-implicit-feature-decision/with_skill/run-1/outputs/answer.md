# 의사결정 분석: 엔터프라이즈 고객의 커스텀 필드 기능 요청

**날짜**: 2026-04-09  
**결정자**: PO  
**상태**: 분석 완료 — 조건부 권고안 도출  
**사용 프레임워크**: One-Way/Two-Way Door, RICE, Value vs Effort Matrix, "No" 프레임워크, RAPID, Stakeholder Conflict Resolution

---

## 1. 상황 정의 (Context)

### 요청 개요
- **요청자**: 연 $50K 계약의 엔터프라이즈 고객 1개사
- **요청 내용**: 프로젝트에 커스텀 필드를 추가할 수 있는 기능
- **내부 압력**: 영업팀이 "이탈 가능성"을 근거로 즉각 구현 촉구
- **비즈니스 맥락**: 계약 갱신 시즌, ARR 대비 2% 비중

### 가정 (Assumptions, 데이터 없어 명시)

분석에 필요한 데이터가 없으므로 아래를 가정으로 설정한다:

| 가정 항목 | 가정값 | 근거 |
|---|---|---|
| 전체 ARR | $2.5M | $50K = 2% 역산 |
| 고객 수 | 약 100-200개사 | ARR 규모 추정 |
| 동일 요청을 한 다른 고객 수 | 알 수 없음 (핵심 미지수) | **추가 조사 필요** |
| 커스텀 필드 구현 Effort | 2-4 person-months | 데이터 모델 변경, UI, API 포함 추정 |
| 기회비용 (대체 가능 작업의 ARR 임팩트) | 알 수 없음 | **추가 조사 필요** |
| 실제 이탈 확률 | 알 수 없음 (영업팀 주관) | **추가 조사 필요** |

## 2. 편향 점검: 영업 압력에 속지 않기

이 요청에는 두 가지 인지 편향 위험이 존재한다.

**Recency Bias**: 갱신 시즌이라는 시점 압박이 판단을 흐린다. "지금 당장"의 위기감이 장기 전략을 덮을 수 있다.

**Survivorship Bias (역방향)**: "이 고객이 이탈하면 큰일 난다"는 프레이밍이 이탈하지 않을 가능성, 혹은 이 기능을 만들어도 이탈하는 시나리오를 보지 못하게 만든다.

**HiPPO 위험**: 영업팀의 목소리가 크다고 해서 그것이 데이터를 대체하지는 않는다. "이탈할 수 있다"는 말은 관찰이 아니라 추측이다.

---

## 3. One-Way Door vs Two-Way Door 분류

**커스텀 필드 기능 자체**는 Two-Way Door다. 출시 후 제거하거나 범위를 줄이는 것이 불편하지만 불가능하지 않다. 빠른 실험이 가능하다.

단, **이 결정에 딸려오는 부수 결정들**은 One-Way에 가깝다:

- 데이터 모델 변경 → 마이그레이션 비용 발생
- 엔터프라이즈 커스터마이제이션 방향 신호 → 향후 로드맵 방향에 영향
- "요청하면 해준다"는 선례 → 이후 고객사들의 기대치 상승

따라서 기능 자체는 빠르게 결정할 수 있지만, **어떤 형태로, 어느 범위로 만들지**는 신중하게 결정해야 한다.

## 4. 트레이드오프 분석

### 4-1. 만든다면 얻는 것

| 항목 | 내용 |
|---|---|
| 직접적 ARR 방어 | $50K 갱신 가능성 유지 |
| 엔터프라이즈 세그먼트 확장 신호 | 커스터마이제이션 = 엔터프라이즈 확장의 공통 요구사항 |
| 고객 충성도 | 이 고객이 헤비 유저가 될 가능성 |

### 4-2. 만든다면 잃는 것

| 항목 | 내용 |
|---|---|
| Opportunity Cost | 2-4 person-months를 다른 기능에 쓰지 못함 |
| 복잡도 증가 | 커스텀 필드는 데이터 모델, 검색, API, 내보내기 등 전방위에 영향 |
| 유지보수 부채 | 엔터프라이즈 커스터마이제이션 기능은 장기적으로 관리 비용이 높다 |
| 제품 방향 혼란 | 1개사의 니즈를 위해 핵심 제품 경험을 희생할 위험 |
| 선례 효과 | 이 요청을 수용하면 유사 요청이 줄을 선다 |

### 4-3. 안 만든다면

| 항목 | 내용 |
|---|---|
| $50K ARR 이탈 위험 | 최대 손실 (실제 이탈 확률은 미지수) |
| 엔터프라이즈 포지셔닝 약화 | 커스터마이제이션 경쟁에서 뒤처질 수 있음 |
| 영업 신뢰도 문제 | 영업팀이 고객에게 약속을 못 지키는 상황 |

## 5. 정량적 추론

### ARR 노출 분석

```
이탈 시 손실       = $50,000 / year
전체 ARR          = ~$2,500,000
손실 비율          = 2%
```

$50K는 절대금액으로 작지 않지만, 전체 ARR의 2%다. **이 하나의 고객을 위해 2-4 person-months를 쓰는 결정은 순수 방어적 관점에서만 보면 비효율적일 수 있다.**

### 기회비용 추산 (상대 비교)

RICE 스코어로 두 시나리오를 비교한다.

**시나리오 A: 커스텀 필드 (이 요청)**
```
Reach      = 1개사 (현재 알려진 수요)
             → 복수 고객 수요 확인 시 N배 상승 가능
Impact     = 2 (high — ARR 방어 + 엔터프라이즈 기능)
Confidence = 40% (실제 이탈 확률 불명, 다른 고객 수요 불명)
Effort     = 3 person-months (중간값)

RICE = (1 × 2 × 0.4) / 3 = 0.27 (매우 낮음)
```

**시나리오 B: 복수 고객이 원한다면 (다른 요청 확인 후)**
```
Reach      = 10개사 이상
Impact     = 2 (high)
Confidence = 70%
Effort     = 3 person-months

RICE = (10 × 2 × 0.7) / 3 = 4.67 (양호)
```

**결론**: 현재 데이터로는 RICE 스코어가 극도로 낮다. **다른 고객들의 동일 수요 확인이 선행되어야 한다.**

## 6. 추가로 필요한 데이터 (Data Gaps)

다음 데이터 없이는 완전한 결정이 불가능하다:

| 필요 데이터 | 확인 방법 | 소요 시간 |
|---|---|---|
| 다른 고객들도 커스텀 필드를 원하는가? | CS/영업팀 조회, 고객 인터뷰 | 1-2주 |
| 이 고객이 실제로 이탈을 결정했는가, 아니면 협상 전략인가? | 고객 담당 AE와 직접 대화 | 즉시 |
| 고객이 지금 이 기능 없이 어떻게 처리하는가? (Workaround) | 고객 인터뷰 | 1주 |
| 경쟁사는 이 기능을 제공하는가? | 경쟁 분석 | 1주 |
| 현재 로드맵의 기회비용 (밀려날 기능의 예상 ARR 임팩트) | 내부 RICE 재계산 | 즉시 |

## 7. 전략적 정렬 체크

아래 3가지를 확인해야 한다:

**Q1. 엔터프라이즈 세그먼트가 우리의 전략적 방향인가?**
- YES라면: 커스텀 필드는 엔터프라이즈 기능의 정석이며, 조기 투자가 합리적
- NO (SMB/PLG 집중)라면: 단일 고객 요청으로 방향이 흔들려서는 안 된다

**Q2. 현재 분기 OKR과 연관되는가?**
- ARR 방어 또는 엔터프라이즈 확장이 OKR에 있다면: 정렬됨
- 아니라면: Parking lot에 넣고 다음 분기 로드맵 리뷰 시 재평가

**Q3. 기술적으로 지금 이 작업을 시작하기에 좋은 시점인가?**
- 현재 데이터 모델이 커스텀 필드를 지원하기에 적절한가? (엔지니어링 의견 필요)

## 8. 대안 옵션 분석

단순한 Yes/No 외에 4가지 대안이 존재한다.

### 옵션 1: 스코프 축소 버전 (Scoped-Down MVP)
- "커스텀 필드"의 전체 범위(타입, 검색, API) 대신 **텍스트 타입 1가지**만 먼저 구현
- Effort: 0.5-1 person-month
- 고객에게 "부분 충족" 제공하며 갱신 유도
- 리스크: 고객 기대를 충족하지 못할 수 있음

### 옵션 2: Workaround 제공
- 고객이 현재 커스텀 필드 없이 어떻게 작업하는지 파악
- 기존 기능의 조합으로 80% 문제를 해결하는 가이드 제공
- Effort: 0 (기능 개발 불필요)
- 리스크: 고객이 이를 거절하고 이탈할 수 있음

### 옵션 3: 리서치 퍼스트 (Research First)
- **2주 안에** 다른 고객 5-10개사의 동일 수요를 조사
- 수요가 넓다면 Q 다음 분기 로드맵에 공식 편입
- 이 고객에게는 "로드맵에 들어갈 예정이며 갱신 이후 우선 접근권 보장" 약속
- Effort: 0 (즉시 구현 없음)
- 리스크: 갱신 시즌 내 결정을 못 내릴 수 있음

### 옵션 4: 계약 연동 조건부 약속 (Commitment-Based Roadmap)
- 갱신 계약에 "커스텀 필드 베타 참여자" 조항 삽입
- 고객이 갱신하면 베타 개발 파트너로 참여, 요구사항을 함께 정의
- Effort: 즉시 0, 이후 2-4 person-months
- 장점: 갱신을 확보하면서 단독 고객 요청이 아닌 "파트너십" 프레임으로 전환

## 9. RAPID로 역할 명확화

이 결정은 단일 PO가 혼자 내리기 어렵다.

| 역할 | 담당 | 내용 |
|---|---|---|
| Recommend | PO | 본 분석 + 권고안 제시 |
| Agree | CTO/Engineering Lead | 기술 부채 영향, 구현 가능성 |
| Agree | CFO/Head of Revenue | ARR 이탈 위험 허용 여부 |
| Input | Sales/AE | 고객의 실제 이탈 의사 온도 확인 |
| Input | CS | 다른 고객들의 동일 수요 집계 |
| Decide | VP Product 또는 PO | 최종 결정 |

**이 결정에서 Sales는 Input 역할이다. Decide 역할이 아니다.**

## 10. 영업팀 갈등 관리 (Stakeholder Conflict)

stakeholder-management.md의 갈등 해결 프레임워크를 적용한다.

```
Sales가 말하는 것: "이거 없으면 이탈한다"
PO가 확인해야 할 것: "이 기능의 수요가 얼마나 넓은가?"
```

영업팀에게 전달할 메시지:
> "이 고객의 니즈를 이해합니다. 이 기능의 수요가 얼마나 광범위한지 파악하기 위해  
> CS/영업 채널을 통해 다른 고객들에게도 확인해 보겠습니다.  
> **2주 안에 5개사 이상 동일 요청이 확인되면 이번 분기 로드맵에 즉시 편입하겠습니다.**  
> 그 전까지는 갱신을 확보하기 위한 대안(베타 파트너십 제안)을 먼저 시도해 주십시오."

이는 우선순위 결정 파일의 "Not Now 프레이밍"과 "대안 제시" 기법을 결합한 응답이다.

## 11. 최종 조건부 권고안

### 전제: 지금 당장 "만든다/안 만든다"를 결정하지 말 것

영업 압박의 시간 프레임(갱신 시즌)에 이끌려 불완전한 정보로 One-Way Door 결정을 내리는 것이 가장 큰 위험이다.

### 권고안 (조건 분기)

**조건 A — 다른 고객 5개사 이상 동일 수요 확인된 경우:**
> **빌드한다.** 단, 전체 기능이 아닌 MVP 범위(텍스트 타입 커스텀 필드, 검색 미지원)로 시작한다.  
> RICE 스코어가 급등하며 전략 정합성이 확보된다.  
> 이 고객을 베타 파트너로 참여시켜 갱신을 유도한다.

**조건 B — 동일 수요가 이 고객뿐인 경우:**
> **지금 빌드하지 않는다.**  
> 대신 계약 연동 조건부 약속(옵션 4)을 영업팀과 협의한다.  
> "갱신 완료 시 다음 분기 베타 파트너 참여 보장"을 계약에 반영하는 방식으로 이탈 위험을 낮춘다.  
> 이 고객을 잃더라도 $50K는 전체 ARR의 2%이며, 이를 막기 위해 3개월의 엔지니어링 리소스를 쓰는 것은 비례적이지 않다.

**조건 C — 고객의 이탈 의사가 협상용 레버리지라고 판단되는 경우 (AE 확인 후):**
> **영업 전략 문제로 전환한다.** PO가 결정할 사안이 아니다.  
> 고객 성공팀이 관계 강화 전략을 주도하도록 한다.

### 즉시 실행해야 할 것 (Action Items)

1. **이번 주 안에**: AE를 통해 고객이 실제로 계약 해지를 결정했는지, 아니면 협상 레버리지인지 확인
2. **2주 안에**: CS/영업 채널에서 커스텀 필드 수요를 가진 다른 고객 수 집계
3. **2주 안에**: 엔지니어링 리드와 스코프 축소 버전의 Effort 추정 진행
4. **결과 취합 후**: RAPID 프레임워크에 따라 VP Product/PO가 최종 결정 내림
