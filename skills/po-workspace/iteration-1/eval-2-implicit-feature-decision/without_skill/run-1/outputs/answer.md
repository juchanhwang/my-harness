# 커스텀 필드 기능 요청 — 제품 의사결정 분석

## 상황 요약

| 항목 | 내용 |
|------|------|
| 요청 고객 | 엔터프라이즈 고객 1사 |
| 계약 규모 | 연 $50,000 |
| ARR 기여도 | 전체의 2% |
| 요청 기능 | 프로젝트에 커스텀 필드 추가 |
| 비즈니스 압력 | 계약 갱신 시즌, 영업팀 이탈 경고 |

---

## 1. 전제 가정 (Assumptions)

이 분석은 아래 가정을 기반으로 한다. 실제 상황과 다르면 판단이 달라질 수 있다.

- **[가정 A]** 현재 제품에 커스텀 필드 기능이 없으며, 이는 신규 기능 개발에 해당한다.
- **[가정 B]** 개발 팀의 스프린트 여유 용량이 제한적이며, 이 기능을 구현하려면 기존 로드맵 아이템이 밀려야 한다.
- **[가정 C]** 전체 ARR은 $2.5M 수준이다 ($50K = 2%).
- **[가정 D]** 고객이 언급한 이탈 위협은 과장될 수 있으며, 영업팀의 긴급도 신호는 거버넌스 없이 제품 방향을 틀 수 있다.
- **[가정 E]** 커스텀 필드는 기술적으로 중간 복잡도 작업이다 (스키마 변경, UI, API 확장 포함, 예상 2~4주).

---

## 2. 핵심 질문 분해

의사결정을 내리기 전에 아래 질문들을 먼저 답해야 한다.

### Q1. 이 기능이 1개 고객만의 요청인가, 아니면 잠재 수요가 있는가?

**중요도: 매우 높음**

단일 고객 요청이라면 이는 "제품 기능"이 아니라 "고객 맞춤 개발"에 가깝다. 커스텀 필드는 일반적으로 다수 고객이 원하는 유틸리티 기능이지만, 반드시 확인해야 한다.

- 다른 고객 지원 티켓, 피드백 채널에서 유사 요청이 있었는가?
- 세일즈 사이클에서 커스텀 필드 부재가 딜 블로커로 등장한 빈도가 있는가?

**만약 이미 3개 이상 고객이 요청했다면** → 우선순위 상향 근거가 강화된다.  
**만약 이 고객만 요청했다면** → 제품 기능보다는 계약 협상 이슈로 봐야 한다.

### Q2. 이탈 위험이 실제인가, 레버리지인가?

**중요도: 높음**

영업팀의 "이탈 가능성" 경고는 정보이지 근거가 아니다.

- 고객이 실제로 경쟁사 POC를 진행 중인가?
- 고객이 공식적으로 이탈 의사를 표명했는가, 아니면 영업 대화에서 나온 맥락인가?
- 계약 갱신까지 남은 시간은 얼마인가?

**레버리지일 가능성이 높다** — 고객이 현재 플랫폼에 종속도(switching cost)가 높다면, 단기 이탈 위협은 실제보다 과장되는 경향이 있다.

### Q3. 이 기능이 현재 로드맵 전략과 정렬되는가?

**중요도: 높음**

제품 전략과 무관한 단기 고객 대응은 기술 부채와 방향성 희석을 초래한다.

- 커스텀 필드가 제품의 "유연성/확장성" 방향과 맞는가?
- 구현하면 장기적으로 유지보수 부담이 어느 정도인가?
- 현재 로드맵에서 밀려나는 아이템의 비즈니스 임팩트는 얼마인가?

## 3. 비용-편익 분석

### 편익 (Benefits)

| 항목 | 추정 가치 |
|------|-----------|
| 이탈 방어 (갱신 성공 시) | $50K/년 유지 |
| 신호 가치 (다른 고객 수요 확인 시) | 잠재 ARR 확대 |
| 영업 지원 (비슷한 요구 가진 프로스펙트 대응) | 딜 클로징 가속 |

### 비용 (Costs)

| 항목 | 추정 규모 |
|------|-----------|
| 개발 비용 | 2~4주 × 팀 평균 임금 (예: 엔지니어 2명 × 3주 ≈ $15K~$25K) |
| 기회 비용 | 밀려나는 로드맵 아이템의 가치 |
| 유지보수 부채 | 커스텀 필드 스키마 변경 시 지속적 영향 |
| 선례 리스크 | "한 고객 요청 → 빌드" 문화 고착 가능성 |

**단순 ROI 계산:**
- $50K 유지 vs 개발 비용 $20K → 표면적으로는 긍정적
- 그러나 기회 비용과 선례 효과를 포함하면 단순 계산으로는 부족

## 4. 대안 검토

기능을 즉시 빌드하는 것만이 선택지가 아니다.

### 대안 A: 단기 대안 제공 + 로드맵 약속
- 메타데이터/태그 기능 등 현재 있는 기능으로 고객 요구를 부분 충족
- 커스텀 필드를 Q2/Q3 로드맵에 공식 포함하고 문서화하여 고객에게 공유
- **장점:** 즉각 개발 없이 갱신 설득 가능  
- **단점:** 고객이 납득 안 하면 효과 없음

### 대안 B: 프로페셔널 서비스 / 맞춤 개발 계약
- 커스텀 필드를 추가 유료 구현으로 제공 (예: $10K~$20K 추가 계약)
- **장점:** 비용 회수, 선례 방지  
- **단점:** 계약 협상 복잡도 증가, 고객이 거부할 수 있음

### 대안 C: 우선순위 트레이드오프 공식화
- 내부적으로 이 기능 vs 로드맵 아이템의 비즈니스 가치를 RICE/ICE 스코어링으로 비교
- 데이터로 결정하고 영업팀에 근거 공유
- **장점:** 감정적 판단 방지, 조직 합의  
- **단점:** 시간 소요

### 대안 D: 즉시 빌드 (조건부)
- 아래 조건이 모두 충족될 때만:
  1. 다른 2개 이상 고객도 동일 요청
  2. 로드맵 전략 방향과 정렬됨
  3. 현재 스프린트에 여유 용량 존재

## 5. 의사결정 프레임워크 적용

### RICE 스코어 예시 (가정 기반)

| 항목 | 값 | 근거 |
|------|-----|------|
| Reach (영향 고객 수) | 낮음 (1~3개 고객) | 현재 확인된 요청 1건 |
| Impact (고객당 임팩트) | 중간 | 유연성 향상, 이탈 방어 |
| Confidence (확신도) | 낮음~중간 | 수요 미확인, 이탈 위협 진위 미확인 |
| Effort (개발 비용) | 중간 | 2~4주 추정 |

**결론: RICE 스코어 단독으로는 우선순위 상위권 진입 어려움**

### 전략 정렬 체크

- 커스텀 필드가 제품의 핵심 가치(예: 단순성, 표준화)를 해치는가? → 정렬 미흡
- 커스텀 필드가 제품의 확장성/엔터프라이즈 포지셔닝을 강화하는가? → 정렬 긍정

## 6. 최종 판단 및 권고

### 판단: **조건부 우선순위 상향 — 즉각 빌드보다 검증 먼저**

**단계별 액션:**

1. **즉시 (이번 주):** 영업팀과 고객 CS를 통해 다음을 파악
   - 다른 고객의 동일 요청 존재 여부
   - 이탈 위협의 구체적 근거 (경쟁사 POC, 정식 이탈 통보 등)
   - 계약 갱신 기한

2. **다른 고객 수요 없음 확인 시:**
   - 커스텀 필드를 공식 로드맵 아이템으로 등록 (중간 우선순위)
   - 고객에게 로드맵 타임라인 공유 및 갱신 협상
   - 대안 A (부분 기능 + 로드맵 약속) 시도

3. **이탈 위협이 실제임 확인 시 (경쟁사 POC 등):**
   - 긴급 스프린트 삽입 vs 기회 비용을 팀 리드와 공식 트레이드오프 검토
   - 단, 이를 예외 케이스로 문서화하여 선례 남기지 않음

4. **다른 고객 수요 확인 시 (2개 이상):**
   - 우선순위 상향 결정
   - 다음 스프린트 계획에 반영

### 하지 말아야 할 것

- 영업팀 압박만으로 즉각 빌드 결정
- $50K 규모를 "ARR 2%"로 과소평가 (단, 동시에 과대평가도 금물)
- 고객 1개의 요청을 검증 없이 전략적 방향으로 취급
