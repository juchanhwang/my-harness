**문서 버전**: v1.0  
**작성일**: 2026-04-09  
**작성자**: Product Team  
**상태**: Draft

## 1. 배경 및 문제 정의

### 1.1 현황

현재 신규 사용자가 가입 후 첫 프로젝트를 생성하기까지의 전환율이 **30%**로, 업계 벤치마크인 **50%**에 크게 미치지 못하고 있다. 이는 가입한 사용자 10명 중 7명이 실제 제품 가치를 경험하기 전에 이탈한다는 의미다.

### 1.2 문제의 중요성

B2B SaaS에서 "첫 프로젝트 생성"은 **Activation 지표**의 핵심이다. 사용자가 첫 프로젝트를 생성해야 협업 기능, 일정 관리, 리포팅 등 제품의 핵심 가치를 경험할 수 있다. Activation 실패는 다음과 같은 직접적인 비즈니스 손실로 이어진다:

- **Trial → Paid 전환율 저하**: Activation 없이 전환하는 사용자는 거의 없음
- **CAC 낭비**: 획득한 리드가 제품을 경험하지 못하고 이탈
- **Revenue 손실 기회**: 업계 벤치마크 수준 달성 시 전환 파이프라인 약 67% 증가 예상

### 1.3 가설

> [가정] 전환율 저하의 주요 원인은 신규 사용자가 첫 프로젝트 생성 과정에서 명확한 가이드 없이 UI의 복잡성에 압도되거나, 어디서 시작해야 할지 모르는 "빈 화면 문제(blank slate problem)"를 경험하기 때문이다.

## 2. 목표 (Goals & Success Metrics)

### 2.1 Primary Goal

| 지표 | 현재 | 목표 | 기간 |
|------|------|------|------|
| 가입 → 첫 프로젝트 생성 전환율 | 30% | 50% | 6주 |

### 2.2 Secondary Goals

| 지표 | 설명 | 목표 |
|------|------|------|
| Time-to-First-Project | 가입 후 첫 프로젝트 생성까지 소요 시간 | 현재 대비 40% 감소 |
| Onboarding Completion Rate | 온보딩 플로우 완료율 | 70% 이상 |
| 7일 리텐션 | 가입 후 7일 내 재방문율 | 현재 대비 15% 이상 증가 |
| 고객 지원 문의 | 온보딩 관련 지원 문의 건수 | 현재 대비 30% 감소 |

### 2.3 Non-Goals (이번 프로젝트 범위 외)

- 기존 사용자(가입 30일 이후) 경험 개선
- 결제/구독 플로우 변경
- 모바일 앱 온보딩 (웹 앱만 대상)
- 제품 핵심 기능 자체의 변경

## 3. Primary Persona

### 3.1 페르소나: "엔지니어링 팀 매니저 민준"

**인구통계**
- 직책: Engineering Manager / Tech Lead
- 팀 규모: 10~50명 엔지니어링 팀
- 회사 규모: 시리즈 A~C 스타트업, 또는 중견기업 내 개발 부서
- 의사결정 권한: 도구 선택 및 도입 결정권 보유 (예산 승인은 상위 직급)

**목표와 동기**
- 팀의 프로젝트 진행 상황을 투명하게 파악하고 싶다
- 개발 일정 관리와 리소스 배분을 효율화하고 싶다
- 스프린트 계획 및 회고를 체계적으로 운영하고 싶다
- 경영진에게 팀 성과를 명확하게 보고하고 싶다

**고충 (Pain Points)**
- 여러 도구(Slack, Jira, Notion 등)가 분산되어 맥락 전환 비용이 크다
- 새 도구를 도입할 때 팀원들의 저항이나 학습 곡선이 부담스럽다
- 도구를 직접 세팅하는 데 시간을 쓰기 싫다 — 빨리 팀에 도입하고 싶다
- 복잡한 설정 없이 바로 가치를 느끼고 싶다

**행동 패턴**
- 업무 시간 중 SaaS를 직접 탐색, 보통 데스크탑 사용
- 경쟁 제품을 이미 사용해봤거나 비교 중인 경우가 많음
- 트라이얼 기간 내 팀원 1~2명을 초대해 같이 써보는 경향
- 설정 가이드를 읽기보단 직접 클릭하며 탐색

> [가정] 위 페르소나 특성은 산업 내 일반적인 EM 프로필 기준으로 작성되었다. 실제 사용자 인터뷰 및 세그먼트 분석을 통해 검증이 필요하다.

## 4. 사용자 여정 분석 (Current vs. Target)

### 4.1 현재 여정 (As-Is)의 문제점

```
[가입 완료] → [대시보드 진입] → ❌ 빈 화면, 안내 없음
                                → ❌ "프로젝트 생성" CTA 눈에 띄지 않음
                                → ❌ 어떤 정보를 입력해야 할지 불명확
                                → ❌ 이탈 (70%)
```

> [가정] 위 흐름은 일반적인 Blank-Slate 문제 패턴 기반 가설이다. 실제 퍼널 데이터 및 세션 녹화(예: FullStory, Hotjar) 분석으로 병목 지점을 확인해야 한다.

### 4.2 목표 여정 (To-Be)

```
[가입 완료]
    ↓
[환영 모달 / 역할 선택] — 30초 내
    ↓
[인터랙티브 온보딩 체크리스트 진입]
    ↓
[첫 프로젝트 생성 가이드 (템플릿 선택 포함)] — 2분 내
    ↓
[팀원 초대 (선택)]
    ↓
[첫 태스크/이슈 생성]
    ↓
["Aha Moment" 달성] ← 이 지점에서 제품 핵심 가치 경험
```

## 5. 솔루션 개요 (Solution Overview)

### 5.1 핵심 기능 목록

#### Feature 1: 역할 기반 환영 플로우 (Welcome Flow)

가입 직후 간단한 인터랙티브 모달로 사용자의 역할과 사용 목적을 파악하여 맞춤형 온보딩 경로를 제공한다.

- 질문 1: "주로 어떤 역할로 사용하실 건가요?" (EM, 개발자, PO 등)
- 질문 2: "팀 규모가 어떻게 되나요?" (5명 이하 / 6~20명 / 21~50명 / 50명 이상)
- 질문 3: "주된 목적은?" (스프린트 관리, 로드맵 계획, 버그 추적 등)
- 최대 3단계, 각 단계 스킵 가능

**수용 기준**:
- 3단계 이내 완료
- 역할에 따라 다른 프로젝트 템플릿 추천
- 스킵 시에도 기본 온보딩으로 진입 가능

#### Feature 2: 인터랙티브 온보딩 체크리스트

대시보드 우측 또는 하단에 항시 표시되는 진행 상황 체크리스트. 완료 항목은 시각적으로 표시되며 달성률(%)을 보여준다.

체크리스트 항목 (우선순위 순):
1. ✅ 첫 프로젝트 생성하기 (필수)
2. ☐ 팀원 1명 초대하기
3. ☐ 첫 이슈/태스크 만들기
4. ☐ 마감일 설정하기
5. ☐ 대시보드 확인하기

**수용 기준**:
- 사용자가 항목 완료 시 실시간 업데이트
- "완료" 달성 시 축하 애니메이션 + 다음 단계 안내
- 온보딩 완료 후 체크리스트는 숨김 처리 (설정에서 재활성화 가능)

#### Feature 3: 프로젝트 템플릿 라이브러리

첫 프로젝트 생성 시 빈 화면 대신 업계별/역할별 템플릿을 선택할 수 있게 한다.

엔지니어링 팀 매니저 대상 추천 템플릿:
- **스프린트 보드** (Scrum/Kanban): 기본 열(Backlog, In Progress, Review, Done) 포함
- **버그 추적 프로젝트**: 우선순위, 심각도 필드 포함
- **분기 로드맵**: 마일스톤 및 에픽 구조 포함
- **팀 온보딩 체크리스트**: 신규 팀원 온보딩용

**수용 기준**:
- EM 역할 선택 시 엔지니어링 템플릿 우선 표시
- 빈 프로젝트 생성 옵션도 항상 제공
- 템플릿 미리보기 제공 (썸네일 또는 간단한 설명)

#### Feature 4: 컨텍스트 툴팁 & 코치마크 (Contextual Guidance)

첫 방문 시, 핵심 UI 요소에 대한 설명 툴팁을 순차적으로 표시한다.

- 최대 5개 단계의 코치마크 (과도한 안내 방지)
- 각 툴팁에 "다음" / "스킵" 버튼 제공
- 이미 숙련된 사용자는 "안내 건너뛰기" 로 전체 스킵 가능

**수용 기준**:
- 첫 로그인 시에만 자동 표시
- 설정 > 온보딩에서 재시작 가능
- 툴팁이 중요 UI 요소를 가리지 않아야 함

#### Feature 5: 이메일 Drip 시퀀스 (Activation Nudge)

가입 후 프로젝트를 생성하지 않은 사용자에게 시간 기반 이메일 시퀀스를 발송한다.

| 발송 시점 | 이메일 내용 |
|-----------|------------|
| 가입 후 1일 | "첫 프로젝트 시작하기" — 템플릿 소개 |
| 가입 후 3일 (미완료 시) | "팀 프로젝트를 3분 만에 설정하는 법" — 동영상 가이드 링크 |
| 가입 후 7일 (미완료 시) | "무엇이 막히시나요?" — 고객 성공팀 미팅 신청 CTA |

**수용 기준**:
- 프로젝트 생성 완료 시 이메일 시퀀스 자동 중단
- 이메일 수신 거부(unsubscribe) 옵션 포함
- A/B 테스트 가능한 구조로 구현

## 6. 기술 요구사항 및 제약사항

### 6.1 기술 요구사항

> [가정] 팀의 기술 스택 및 기존 인프라에 따라 아래 사항은 조정이 필요하다.

- **프론트엔드**: 기존 컴포넌트 라이브러리 내에서 구현. 온보딩 UI는 별도 라이브러리 없이 자체 구현 권장 (Shepherd.js 또는 Intro.js 등 경량 라이브러리 검토 가능)
- **백엔드**: 온보딩 상태(어떤 단계를 완료했는지)를 사용자 프로필에 저장하는 API 엔드포인트 필요
- **이메일 자동화**: 기존 이메일 서비스(예: SendGrid, Mailchimp) 연동 또는 신규 설정
- **Analytics**: 각 온보딩 단계별 이벤트 트래킹 (예: `onboarding_step_completed`, `first_project_created`)

### 6.2 비기능 요구사항

- 온보딩 플로우 추가로 인한 초기 로딩 성능 저하 없어야 함 (LCP 기준 영향 없음)
- 접근성: WCAG 2.1 AA 준수 (툴팁, 모달 등 키보드 접근 가능)
- 온보딩 콘텐츠 다국어 지원 구조로 설계 (현재 한국어/영어 지원 기준)

## 7. UX/디자인 요구사항

### 7.1 디자인 원칙

- **최소 마찰(Minimum Friction)**: 가입부터 첫 프로젝트 생성까지 3분 이내 완료 가능해야 함
- **점진적 노출(Progressive Disclosure)**: 모든 기능을 한 번에 보여주지 말고, 사용 흐름에 따라 순차 노출
- **즉각적 가치 확인(Immediate Value)**: 첫 프로젝트 생성 직후 바로 "사용 가능한 상태"처럼 느껴져야 함

### 7.2 핵심 UX 흐름

1. 가입 완료 → 환영 모달 (역할 선택) → 대시보드
2. 대시보드 진입 시 온보딩 체크리스트 자동 표시
3. "첫 프로젝트 생성" 클릭 → 템플릿 선택 모달 → 프로젝트 생성 완료
4. 생성 완료 후 프로젝트 내부로 자동 이동 + 코치마크 시작

### 7.3 목업/프로토타입 요구사항

- Figma 와이어프레임: 환영 모달, 온보딩 체크리스트, 템플릿 선택 화면
- 인터랙티브 프로토타입: 핵심 플로우 (가입 → 첫 프로젝트 생성) 클릭 데모

## 8. 출시 전략 (Go-to-Market & Rollout)

### 8.1 출시 단계

| 단계 | 기간 | 내용 |
|------|------|------|
| Alpha | Week 1~2 | 내부 테스트 + 주요 기능 빌드 |
| Closed Beta | Week 3~4 | 신규 가입자 20%에게 적용, 데이터 수집 |
| Full Rollout | Week 5~6 | 전체 신규 가입자 적용, 모니터링 |

### 8.2 A/B 테스트 계획

- **테스트 A**: 역할 선택 질문 포함 환영 플로우 vs. 미포함
- **테스트 B**: 온보딩 체크리스트 위치 (우측 사이드바 vs. 상단 배너)
- **테스트 C**: 이메일 발송 타이밍 (1/3/7일 vs. 즉시/2일/5일)

## 9. 타임라인 & 마일스톤

### 9.1 6주 스프린트 계획

```
Week 1: 발견 & 설계
├── 기존 온보딩 퍼널 데이터 분석 (세션 녹화, 드롭오프 포인트)
├── 신규 사용자 인터뷰 3~5건 실시
├── 디자인 와이어프레임 초안 작성
└── 기술 스펙 확정

Week 2: 디자인 & 설계 확정
├── Figma 목업 완성 (환영 모달, 체크리스트, 템플릿 갤러리)
├── 디자인 리뷰 및 피드백 반영
├── 백엔드 API 설계 (온보딩 상태 저장)
└── 이메일 템플릿 카피라이팅

Week 3~4: 개발
├── [FE] 환영 플로우 모달 구현
├── [FE] 온보딩 체크리스트 컴포넌트
├── [FE] 프로젝트 템플릿 선택 UI
├── [BE] 온보딩 상태 저장 API
├── [BE] 이메일 드립 시퀀스 연동
└── Analytics 이벤트 트래킹 구현

Week 5: QA & Closed Beta
├── 내부 QA 테스트 (기능, 엣지 케이스, 접근성)
├── Closed Beta 론칭 (신규 가입자 20%)
├── 지표 모니터링 및 버그 수정
└── 사용자 피드백 수집

Week 6: Full Rollout & 결과 분석
├── 전체 신규 가입자 대상 롤아웃
├── 지표 분석 및 결과 리포트
├── 추가 최적화 사항 백로그 등록
└── 팀 회고
```

## 10. 성공 측정 방법

### 10.1 측정 프레임워크

**Primary Metric**
- 가입 후 48시간 내 첫 프로젝트 생성 전환율 (목표: 30% → 50%)

**Leading Indicators (Weekly 측정)**
- 환영 플로우 완료율 (목표: 80% 이상)
- 온보딩 체크리스트 진입율 (목표: 신규 가입자 70% 이상)
- 템플릿 선택율 vs. 빈 프로젝트 생성 비율

**Guardrail Metrics (악화되면 안 되는 지표)**
- 기존 사용자 프로젝트 생성 전환율 (변화 없어야 함)
- 페이지 로딩 속도 (LCP 증가 없어야 함)
- 온보딩 관련 고객 지원 티켓 (감소해야 함)

### 10.2 리포팅 주기

- 매주 화요일: Closed Beta 지표 리뷰 (PM + 개발팀)
- Week 6 종료 후: 전체 결과 분석 및 이해관계자 공유

## 부록

### A. 용어 정의

- **Activation**: 신규 사용자가 제품의 핵심 가치를 처음 경험하는 순간. 이 PRD에서는 "첫 프로젝트 생성"으로 정의
- **Blank Slate Problem**: 새로운 도구에 처음 접속했을 때 빈 화면에서 무엇을 해야 할지 모르는 UX 문제
- **Drip 시퀀스**: 시간 간격을 두고 순차적으로 발송되는 자동화 이메일 시리즈
- **Aha Moment**: 사용자가 "이 제품이 나에게 필요하다"고 깨닫는 결정적 순간
