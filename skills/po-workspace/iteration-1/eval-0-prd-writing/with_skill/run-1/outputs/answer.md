## 1. 배경 & 문제

### 1.1 현재 상황 (AS-IS)

우리 제품은 B2B 프로젝트 관리 SaaS로, 주요 타겟은 10~50명 규모의 엔지니어링 팀이다.
최근 3개월(2026년 1월~3월) 데이터를 기반으로 신규 가입 퍼널을 분석한 결과 다음과 같은 구조가 확인됐다.

```
신규 가입자: 100% (베이스라인)
  ↓
첫 프로젝트 생성: 30%       ← Activation ★ 핵심 drop-off 지점
  ↓
W1 재방문 (추정): ~18%
  ↓
유료 전환 (추정): 추적 예정
```

**Activation rate 30%** — 업계 벤치마크 **50%** 대비 **-20%p 하회**. 이는 AARRR 퍼널에서 가장 큰 drop-off 단계이며, 하류 지표(Retention, Revenue) 모두를 왜곡하고 있는 근본 병목이다.

### 1.2 문제 정의

> "우리는 사용자가 가입하고 나서 무엇을 해야 할지 몰라서 이탈하고 있다."

신규 가입자의 70%가 첫 프로젝트를 생성하지 않고 이탈하고 있다. 이 시점은 사용자가 우리 제품에 처음으로 가치를 경험할 수 있는 유일한 기회다. 이 기회를 놓치면:

- 사용자는 경쟁사 또는 기존 솔루션(Jira, Linear, Asana)으로 회귀한다
- 팀 전체 도입으로 이어지지 않아 B2B 특성상 잠재 계약 규모가 통째로 사라진다
- CAC 대비 LTV가 악화되어 단위 경제학이 무너진다

**[가정 — 사용자 인터뷰 부재 시 데이터 기반 추론]**

정성적 인터뷰 데이터가 충분하지 않은 상태이므로, 아래는 B2B SaaS 온보딩 패턴 및 유사 제품 벤치마크를 기반으로 도출한 가설이다. 실제 인터뷰를 통해 검증 후 업데이트 필요.

| # | 가설 (이탈 원인) | 근거 | 검증 방법 |
|---|---|---|---|
| H1 | 빈 캔버스 문제 — 첫 화면이 비어있어 어디서 시작할지 모른다 | B2B SaaS 온보딩 패턴; 템플릿 없이 시작하는 제품의 공통 문제 | 세션 레코딩, 5초 테스트 |
| H2 | JTBD 불일치 — "프로젝트 생성"이 사용자의 멘탈 모델과 다르다 | 가입 후 30초 내 이탈 비율 높음 (가정) | 사용자 인터뷰, 퍼널 분석 |
| H3 | 설정 부담 — 팀원 초대, 워크스페이스 설정 등 작업이 너무 많다 | Enterprise SaaS의 공통 마찰 요인 | Heatmap, 이탈 지점 분석 |
| H4 | AHA moment 지연 — 가치를 느끼기까지 시간이 너무 걸린다 | TTV(Time-to-Value) 벤치마크 초과 (가정) | 행동 데이터 분석 |

**고객 목소리 (지원 티켓 및 이탈 설문 — 가상 예시, 실 데이터로 교체 필요)**

> "가입하고 나서 뭘 해야 할지 막막했어요. 그냥 Jira로 돌아갔습니다." — 이탈 사용자 A
>
> "팀원 초대부터 시작해야 하는데, 혼자 써보고 싶었는데 그게 안 됐어요." — 이탈 사용자 B
>
> "기능이 많은데 어떤 게 우리 팀에 맞는지 알 수가 없었어요." — 이탈 사용자 C

### 1.3 기회

온보딩 전환율을 50%로 개선하면:

- 동일 CAC 기준, Activation 단계 사용자 수 **+67%** 증가
- 이후 Retention, Revenue 퍼널 전체가 연쇄적으로 개선
- B2B 특성상 한 명의 엔지니어링 매니저 활성화 = 팀 전체 시트 계약으로 이어질 가능성
- **비용 없이 growth**: 추가 마케팅 비용 없이 기존 유입 사용자에서 더 많은 가치 추출

### 1.4 비즈니스 맥락

이 프로젝트는 2026년 H1 핵심 목표인 "Product-led Growth 기반 확장" 전략의 선행 조건이다. Activation rate를 먼저 수리하지 않으면 Acquisition 투자 증가가 효과를 발휘할 수 없다 (새는 양동이 문제).

## 2. 목표 & 비목표

### 2.1 Goals

| # | 목표 | 기준선 | 타겟 | 기간 |
|---|---|---|---|---|
| G1 | **Activation rate 개선** — 신규 가입 후 첫 프로젝트 생성 전환율 | 30% | **50%** | 6주 후 측정 (4주 롤아웃 + 2주 관찰) |
| G2 | **TTV(Time-to-Value) 단축** — 가입 후 첫 프로젝트 생성까지 평균 소요 시간 | 측정 예정 (현재 미추적) | **50% 단축** | 동일 |
| G3 | **온보딩 완료율 추적 인프라 구축** — 각 온보딩 스텝별 완료율 대시보드 | 미존재 | **W2 내 구축** | 구현 2주차 |

### 2.2 Non-Goals (이번에 하지 않는 것)

- **팀원 초대 온보딩 개선**: 개인 사용자의 첫 활성화에 집중. 팀 초대 플로우는 별도 프로젝트
- **유료 전환 최적화**: Activation 이후 단계는 다음 분기 과제
- **모바일 온보딩**: 이번 범위는 웹 앱에 한정
- **기존 사용자 리온보딩**: 신규 가입자 플로우에만 집중
- **Onboarding 콘텐츠 현지화(다국어)**: 한국어/영어 외 언어 지원 제외
- **소셜 로그인 추가**: 가입 자체의 마찰 감소는 별도 트랙
- **엔터프라이즈 커스터마이징**: 10~50명 규모 타겟에 집중, 대기업 커스텀 온보딩 제외

## 3. 대상 사용자

### 3.1 Primary Persona: 엔지니어링 매니저 "민준"

```
이름 (대표명): 민준
역할: Engineering Manager / Tech Lead
팀 규모: 10~50명 엔지니어링 팀
```

**행동 패턴**

- 팀의 스프린트 진행상황을 주 2회 이상 확인한다
- Slack, Notion, GitHub을 병행 사용하며 컨텍스트 스위칭이 잦다
- 새 도구 도입 결정 권한이 있지만, 팀원 설득이 필요하다
- 주로 업무 시간 중 빠르게 새 도구를 탐색하며, 집중 시간은 부족하다
- 무료 체험 → 팀 전체 구독 결정까지 본인이 먼저 검증하는 패턴

**핵심 Job (JTBD)**

```
When 새 프로젝트 관리 도구를 탐색할 때,
I want to 우리 팀 워크플로우에 맞는지 10분 안에 파악하고 싶다,
So I can 팀원에게 자신 있게 추천하고, 도입을 결정할 수 있다.
```

**페인포인트 (우선순위순)**

1. 새 도구 탐색에 할애할 시간이 없다 — "지금 당장 돌아가는 프로젝트가 있어서 도구 설정에 1시간을 쓸 수 없다"
2. 빈 화면에서 뭘 먼저 해야 할지 모른다 — 첫 화면의 인지 부하가 크다
3. 팀 워크플로우와 맞는지 확신이 없다 — 실제 업무에 써보기 전까지는 판단 불가
4. 팀원 설득 부담 — 혼자 결정할 수 없고 팀의 동의가 필요하다

**현재 대안**

- Jira: 기능 풍부하지만 무겁고, 설정 복잡
- Linear: 개발자 친화적이지만 매니저 뷰 부족
- Notion: 유연하지만 프로젝트 관리 특화 기능 없음
- GitHub Projects: 코드와 연동되지만 비개발 직군과 협업 어려움

**성공 기준**

> "10분 안에 우리 팀의 스프린트처럼 보이는 것을 만들어보고, 팀원에게 '이거 써봐'라고 바로 보낼 수 있으면 된다."

**대표 인용구**

> "새 도구는 일단 내가 써봐야 팀에 권할 수 있는데, 첫날 세팅하다가 막히면 그냥 포기하게 됩니다."

### 3.2 Secondary Persona: 시니어 개발자 "지현"

- 역할: Senior Software Engineer (팀 리드 후보)
- 특성: 매니저의 추천을 받고 도구를 탐색하거나, 본인이 팀 내 도구를 제안하는 역할
- JTBD: "팀 컨텍스트를 잃지 않고 내 작업만 빠르게 관리하고 싶다"
- 이번 PRD에서의 비중: Secondary — 온보딩 플로우는 매니저 경험 중심으로 설계

### 3.3 Non-Target

- **대기업(50명+) IT 관리자**: 복잡한 권한 관리, SSO, 컴플라이언스 요구 → 별도 Enterprise 트랙
- **1인 프리랜서**: 팀 협업 기능 가치 낮음, 타겟 세그먼트 외
- **비기술직 프로젝트 매니저**: 현재 제품의 핵심 가치(개발 워크플로우 통합)와 맞지 않음

## 4. 제안 솔루션

### 4.1 솔루션 개요

세 가지 개입 레이어로 온보딩 경험을 재설계한다.

1. **인터랙티브 온보딩 체크리스트** — 가입 직후 "다음에 할 일"을 명확히 안내
2. **프로젝트 템플릿 라이브러리** — 빈 캔버스 대신 실제 팀 워크플로우에 맞는 출발점 제공
3. **온보딩 이메일 시퀀스** — 이탈 사용자 재유입 및 단계별 nudge

이 세 가지를 함께 실행해야 시너지가 발휘된다. 어느 하나만 개선하면 효과가 제한적이다.

### 4.2 User Flow

#### 가입 후 온보딩 플로우 (전체)

```
[가입 완료]
    ↓
[Welcome 화면] ← 新 (지금은 대시보드로 바로 이동)
    - "민준님, 안녕하세요! 10분 안에 첫 프로젝트를 만들어보세요."
    - 진행률 바 (0/4 완료)
    ↓
[스텝 1: 팀 유형 선택] ← 新
    - "어떤 팀인가요?" (Engineering / Design / Product / Mixed)
    - 선택에 따라 추천 템플릿 개인화
    ↓
[스텝 2: 템플릿 선택] ← 新 (지금은 빈 프로젝트만)
    - "스프린트 보드", "버그 트래커", "신규 기능 런칭", "빈 보드"
    - 각 템플릿 미리보기 제공
    ↓
[스텝 3: 프로젝트 생성] ← 기존 (간소화)
    - 프로젝트 이름 입력 (pre-fill: "민준의 첫 프로젝트")
    - 생성 → ★ ACTIVATION MOMENT ★
    ↓
[스텝 4: 샘플 태스크 확인] ← 新
    - 템플릿 기반 샘플 태스크 3개 미리 생성
    - "이 태스크를 수정하거나 새 태스크를 추가해보세요"
    ↓
[온보딩 완료] ← 新
    - "첫 프로젝트 완료! 팀원을 초대해보세요 (선택)"
    - 초대 건너뜀 옵션 명확히 제공
    ↓
[대시보드] — 온보딩 체크리스트 배너 유지 (이후 7일간)
```

#### 이탈 재유입 플로우 (이메일 시퀀스)

```
[가입 후 24시간] — 미활성화 감지
    → 이메일 #1: "시작하는 방법" (템플릿 링크 포함)

[가입 후 72시간] — 여전히 미활성화
    → 이메일 #2: "10분 체험" (짧은 데모 영상 링크)

[가입 후 7일] — 여전히 미활성화
    → 이메일 #3: "팀에서 이런 방식으로 씁니다" (케이스 스터디)
```

### 4.3 User Stories & Acceptance Criteria

---

#### Epic: 신규 사용자 온보딩 개선

---

**Story 1: Welcome & 팀 유형 설정**

```
As an engineering manager,
I want to tell the product what kind of team I have when I first sign up,
So that I get a more relevant starting experience without reading documentation.
```

**Acceptance Criteria**

```
Given 사용자가 가입을 완료하고
When 처음 대시보드에 접근하면
Then Welcome 화면이 표시되고
  And 팀 유형 선택지(Engineering / Design / Product / Mixed)가 보이며
  And 선택 없이 건너뛸 수 있는 옵션("나중에 설정")이 있다

Given 팀 유형을 선택하면
When 다음 단계로 이동할 때
Then 선택한 유형에 맞는 템플릿이 우선 노출된다

Given 사용자가 이미 Welcome 화면을 완료한 경우
When 다시 대시보드에 접근하면
Then Welcome 화면은 표시되지 않는다 (중복 표시 방지)
```

---

**Story 2: 프로젝트 템플릿 라이브러리**

```
As an engineering manager,
I want to choose from pre-built project templates that match my team's workflow,
So that I can start with a meaningful project structure instead of a blank canvas.
```

**Acceptance Criteria**

```
Given 사용자가 템플릿 선택 단계에 있을 때
When 화면을 보면
Then 최소 4개의 템플릿이 표시된다: "스프린트 보드", "버그 트래커", "신규 기능 런칭", "빈 보드"
  And 각 템플릿에 미리보기 썸네일과 1줄 설명이 있다
  And 팀 유형 선택 기반으로 추천 템플릿이 상단에 정렬된다

Given 사용자가 템플릿을 선택하고 프로젝트를 생성하면
When 프로젝트가 생성되면
Then 템플릿에 정의된 기본 컬럼(예: To Do / In Progress / Done)과
  And 샘플 태스크 3개가 미리 생성되어 있다
  And 샘플 태스크에는 현실적인 예시 내용이 들어있다 (예: "스프린트 목표 설정", "일일 스탠드업 아젠다 준비")

Given 사용자가 "빈 보드"를 선택하면
When 프로젝트가 생성되면
Then 기본 컬럼만 있고 샘플 태스크는 없다
```

---

**Story 3: 인터랙티브 온보딩 체크리스트**

```
As an engineering manager,
I want a persistent checklist that guides me through the key setup steps,
So that I always know what to do next without feeling lost.
```

**Acceptance Criteria**

```
Given 사용자가 첫 프로젝트를 생성한 이후
When 대시보드에 접근하면
Then 온보딩 체크리스트 배너가 우측 패널 또는 상단 배너에 표시된다
  And 현재까지 완료한 스텝과 남은 스텝이 시각적으로 구분된다
  And 각 스텝은 클릭 가능하며 해당 기능으로 바로 이동한다

Given 체크리스트의 모든 항목을 완료하면
When 마지막 스텝이 체크되면
Then "온보딩 완료" 축하 메시지와 함께 체크리스트가 자동으로 닫힌다
  And 완료 이벤트가 Analytics로 전송된다

Given 사용자가 체크리스트를 명시적으로 닫으면
When 닫기(X) 버튼을 클릭하면
Then 체크리스트가 숨겨지고
  And "온보딩 가이드 다시 보기" 링크가 헬프 메뉴에 표시된다

성능 요구사항:
- 체크리스트 렌더링 1초 이내
- 각 스텝 전환 애니메이션 200ms 이내
```

**체크리스트 항목 (4단계)**

| 순서 | 항목 | 완료 기준 | 예상 소요 시간 |
|---|---|---|---|
| 1 | 첫 프로젝트 만들기 | 프로젝트 1개 생성 | 2분 |
| 2 | 첫 태스크 추가하기 | 태스크 1개 이상 생성 또는 수정 | 1분 |
| 3 | 태스크 상태 변경하기 | 태스크를 다른 컬럼으로 드래그 | 30초 |
| 4 | 팀원 초대하기 (선택) | 초대 발송 또는 "나중에 하기" 선택 | 1분 |

**Story 4: 온보딩 이메일 시퀀스**

```
As a new user who signed up but didn't complete the first project,
I want to receive helpful guidance in my inbox,
So that I can return and successfully set up my workspace when I have time.
```

**Acceptance Criteria**

```
Given 사용자가 가입 후 24시간 내에 첫 프로젝트를 생성하지 않은 경우
When 24시간이 경과하면
Then 이메일 #1이 발송된다 ("시작하는 방법")
  And 이메일에는 "지금 시작하기" CTA 버튼과 직접 링크가 포함된다

Given 이메일 #1 수신 후 48시간이 더 지나도 미활성화인 경우
When 총 72시간이 경과하면
Then 이메일 #2가 발송된다 ("10분 데모")

Given 이메일 #2 수신 후에도 7일 내 미활성화인 경우
When 7일이 경과하면
Then 이메일 #3이 발송된다 ("고객 사례")

Given 사용자가 어느 시점에든 첫 프로젝트를 생성하면
When Activation event가 발생하면
Then 이후 모든 온보딩 이메일 시퀀스가 중단된다 (불필요한 이메일 방지)

Given 사용자가 이메일 수신을 거부한 경우
When 언구독(unsubscribe)하면
Then 시퀀스 전체가 즉시 중단된다
```

Then 스텝별 funnel 차트가 표시된다
  And 각 스텝의 이탈률이 실시간으로 확인 가능하다
  And 팀 유형별, 템플릿별 세그먼트 분리가 가능하다

### 4.4 대안 검토

| 옵션 | 설명 | 장점 | 단점 | 선택 여부 |
|---|---|---|---|---|
| **A: 인터랙티브 체크리스트 + 템플릿 + 이메일** (선택) | 3개 레이어 동시 구현 | 최대 임팩트, 이탈 사용자 재유입까지 커버 | 구현 복잡도 높음 | ✅ 선택 |
| **B: 제품 투어(Tooltip)** | 첫 로그인 시 tooltip으로 UI 안내 | 빠른 구현 | 사용자가 건너뜀, 빈 캔버스 문제 미해결 | ❌ — 충분하지 않음 |
| **C: 영상 튜토리얼** | 온보딩 영상 링크 제공 | 제작 난도 낮음 | 시청률 낮음, 비선형적 학습 지원 안 됨 | ❌ — 보조 수단으로만 활용 |
| **D: Sales-led 온보딩** | CS가 1:1로 신규 사용자 온보딩 | 높은 완료율 | 비용 높음, 확장성 없음, PLG 전략과 불일치 | ❌ — 엔터프라이즈 세그먼트에만 적용 |
| **E: 체크리스트만 (이메일 제외)** | 이메일 시퀀스 없이 인앱만 | 빠른 구현 | 이미 이탈한 사용자 커버 안 됨 | ❌ — 범위 축소 시 검토 |

**선택 이유 (A)**

Activation rate 30%→50% 목표를 달성하려면 두 가지 사용자를 모두 잡아야 한다:
1. 가입 후 제품을 탐색하는 사용자 → 체크리스트 + 템플릿으로 즉시 경험
2. 가입 후 이탈했다가 돌아오는 사용자 → 이메일 시퀀스로 재유입

어느 한 레이어만으로는 목표 달성이 어렵다.

## 5. 성공 지표

### 5.1 Primary Metric (최우선 지표)

| 지표 | 정의 | 현재 | 목표 | 측정 방법 | 타이밍 |
|---|---|---|---|---|---|
| **Activation Rate** | 가입 후 N일 이내 첫 프로젝트 생성 비율 | 30% | **50%** | Analytics funnel | 롤아웃 2주 후 |

> **측정 기준**: "가입 후 7일 이내 첫 프로젝트 생성"을 Activation 이벤트로 정의. 현재 정의가 다르다면 먼저 통일 필요.

### 5.2 Secondary Metrics (보조 지표)

| 지표 | 정의 | 현재 | 목표 | 측정 방법 |
|---|---|---|---|---|
| **TTV (Time-to-Value)** | 가입 → 첫 프로젝트 생성까지 중앙값 소요 시간 | 미추적 | **현재 대비 50% 단축** | Analytics |
| **온보딩 완료율** | 체크리스트 4개 항목 모두 완료한 사용자 비율 | 해당없음 (신규) | **30% 이상** | Analytics |
| **템플릿 사용률** | 프로젝트 생성 시 템플릿 선택 비율 | 0% (기능 없음) | **70% 이상** | Analytics |
| **이메일 재유입 비율** | 이메일 클릭 후 Activation 완료 비율 | 해당없음 (신규) | **15% 이상** | 이메일 도구 + Analytics |

### 5.3 Guardrail Metrics (방어 지표 — 악화되어선 안 되는 것)

| 지표 | 정의 | 현재 | 수용 한계 | 모니터링 이유 |
|---|---|---|---|---|
| **온보딩 이탈률** | Welcome 화면에서 즉시 이탈하는 비율 | 측정 예정 | 현재 대비 악화 없음 | 온보딩 강제화로 인한 거부감 모니터링 |
| **W1 Retention** | 첫 프로젝트 생성 후 7일 내 재방문 비율 | 측정 예정 | 변화 없음 이상 | 품질 낮은 Activation 방지 |
| **에러율** | 온보딩 플로우 내 오류 발생률 | 해당없음 | **1% 미만** | 새 플로우 안정성 |
| **페이지 로드 시간** | 온보딩 화면 LCP | 측정 예정 | **2.5초 이하 유지** | UX 저하 방지 |

### 5.4 측정 일정

```
Week 1-2: 개발 시작 + Analytics 이벤트 설계
Week 3:   Analytics 구현 완료 및 검증
Week 4:   내부 테스트 (QA)
Week 5:   10% 사용자 대상 A/B 테스트 시작
Week 6:   데이터 수집, 결과 분석
Week 7:   전체 롤아웃 또는 수정 후 재테스트
```

### 5.5 AARRR 맥락에서의 위치

```
Acquisition (방문자) → 유지
  ↓
Activation (첫 프로젝트 생성) 30% → 50% ← ★ 이번 프로젝트 집중 구간
  ↓
Retention (W1 재방문) → 개선 예상 (연쇄 효과)
  ↓
Revenue (유료 전환) → 미래 프로젝트
  ↓
Referral (추천) → 미래 프로젝트
```

## 6. 기술 고려사항

### 6.1 신규 컴포넌트 & API

| 항목 | 설명 | 복잡도 | 담당 |
|---|---|---|---|
| **OnboardingState 데이터 모델** | 사용자별 온보딩 진행 상태 저장 (완료 스텝, 팀 유형 선택, 온보딩 완료 여부) | 중 | Backend |
| **Template 데이터 모델** | 템플릿 정의 (컬럼 구조, 샘플 태스크, 메타데이터) | 중 | Backend |
| **POST /api/projects (템플릿 지원)** | 기존 프로젝트 생성 API에 `template_id` 파라미터 추가 | 낮 | Backend |
| **GET /api/onboarding/state** | 현재 사용자의 온보딩 상태 조회 | 낮 | Backend |
| **PATCH /api/onboarding/state** | 온보딩 스텝 완료 업데이트 | 낮 | Backend |
| **온보딩 체크리스트 UI 컴포넌트** | 대시보드 내 배너/패널 형태 | 중 | Frontend |
| **Welcome 화면** | 가입 직후 표시되는 신규 화면 | 낮 | Frontend |
| **템플릿 선택 모달** | 템플릿 갤러리 + 미리보기 | 중 | Frontend |
| **이메일 시퀀스 자동화** | 이탈 감지 → 조건부 이메일 발송 (예: SendGrid + 내부 스케줄러) | 높 | Backend |
| **Analytics 이벤트 레이어** | 온보딩 스텝별 이벤트 추적 | 중 | Frontend + Backend |

### 6.2 의존성 & 통합

- **이메일 발송**: 현재 사용 중인 이메일 서비스 확인 필요 (SendGrid 가정). 온보딩 전용 시퀀스 설정 필요
- **Analytics**: 현재 사용 중인 Analytics 도구에 온보딩 이벤트 스키마 추가 (PostHog / Amplitude / Mixpanel — 도구 확인 필요)
- **기존 프로젝트 생성 플로우**: 기존 API에 breaking change 없이 `template_id` 옵션 파라미터로 추가
- **Feature Flag**: 점진적 롤아웃을 위한 Feature Flag 활용 권장 (예: LaunchDarkly 또는 자체 구현)

### 6.3 성능 요구사항

- 온보딩 Welcome 화면 LCP: **2.5초 이하**
- 템플릿 목록 로딩: **1초 이하** (서버 사이드 캐싱 권장)
- 프로젝트 생성(템플릿 포함): **3초 이하**
- 온보딩 체크리스트 상태 업데이트: **낙관적 UI(optimistic update) 적용** (UX 지연 방지)

### 6.4 데이터 마이그레이션 & 기존 사용자 처리

- 신규 OnboardingState는 **신규 가입자에게만 적용** — 기존 사용자 영향 없음
- 기존 사용자가 실수로 온보딩 화면에 진입하는 경우 방지: 계정 생성일 기준 분기

### 6.5 보안 & 개인정보

- 이메일 시퀀스: GDPR / 개인정보보호법 기반 opt-out 링크 필수
- 온보딩 상태 데이터: 기존 사용자 데이터 접근 정책과 동일하게 적용
- 팀 유형 선택 정보: PII 아님, 분석 목적으로 저장 가능

### 6.6 기술 부채 & 리스크

| 리스크 | 가능성 | 영향도 | 완화 방안 |
|---|---|---|---|
| 이메일 발송 인프라 없거나 미구축 | 중 | 높 | W1에서 확인 후 외부 도구(Customer.io 등) 도입 검토 |
| Analytics 이벤트 누락 | 중 | 높 | QA 단계에서 이벤트 별도 검증 체크리스트 운용 |
| 기존 프로젝트 생성 API 사이드이펙트 | 낮 | 높 | 회귀 테스트 필수, 기존 플로우 별도 테스트 스위트 |
| 온보딩 강제로 인한 이탈 역효과 | 중 | 중 | 건너뜀(skip) 옵션 항상 제공, Guardrail metric으로 모니터링 |

## 7. 디자인 방향

### 7.1 핵심 UX 원칙

이번 온보딩 디자인에서 지켜야 할 원칙:

1. **점진적 공개 (Progressive Disclosure)**: 한 번에 모든 정보를 보여주지 않는다. 스텝별로 필요한 것만 노출
2. **빠른 성공 경험 (Quick Win)**: 사용자가 2분 이내에 "내가 뭔가 만들었다"는 느낌을 받아야 한다
3. **탈출구 보장**: 모든 스텝에서 "건너뜀" 또는 "나중에" 옵션을 제공. 온보딩은 강요가 아닌 안내
4. **컨텍스트 유지**: 온보딩 완료 후에도 체크리스트가 남아 계속 가이드 제공

### 7.2 디자인 레퍼런스

유사 제품 온보딩 벤치마크:
- **Linear**: 템플릿 선택 + 빠른 첫 이슈 생성 플로우
- **Notion**: 개인화 질문으로 시작하는 온보딩
- **Slack**: 진행률 기반 체크리스트

- [ ] **Figma 링크**: [W1 완료 후 추가 예정]

## 9. 미결 사항

- [ ] **Q1: Activation 이벤트 정의 통일** — "첫 프로젝트 생성 후 N일 이내"에서 N의 기준이 현재 팀 내에 통일되어 있는가? (7일 기준 제안, 확인 필요)
  - 담당: PO + Data Analyst | 기한: W1 내

- [ ] **Q2: 이메일 발송 인프라** — 현재 사용 중인 이메일 도구가 조건부 시퀀스를 지원하는가? 아니면 Customer.io 또는 Brevo 등 별도 도구 도입 필요한가?
  - 담당: Engineering Lead | 기한: W1 내

- [ ] **Q3: Feature Flag 도구** — 점진적 롤아웃을 위한 Feature Flag 인프라가 있는가? 없으면 간단한 내부 구현으로 시작 가능한가?
  - 담당: Engineering Lead | 기한: W1 내

- [ ] **Q4: 기존 사용자 온보딩** — "이미 가입한 사용자 중 첫 프로젝트를 생성하지 않은 사람"에 대해 온보딩 체크리스트를 소급 적용할 것인가? (범위 외로 분류했으나 CS 팀에서 요청 가능성 있음)
  - 담당: PO | 기한: W2 내

- [ ] **Q5: 템플릿 관리 권한** — 초기 템플릿 4종 외에 관리자가 커스텀 템플릿을 추가할 수 있어야 하는가? (Non-goal로 분류했으나 Sales에서 요청 가능성 있음)
  - 담당: PO | 기한: W2 내

- [ ] **Q6: 사용자 인터뷰 일정** — 이탈 원인 가설을 검증하기 위한 인터뷰를 W1에 최소 2~3건 확보 가능한가? Customer Success 팀에 이탈 사용자 연락처 확인 필요
  - 담당: PO + Customer Success | 기한: W1 내

- [ ] **Q7: A/B 테스트 샘플 크기** — 현재 주간 신규 가입자 수가 얼마인가? 통계적 유의미한 결과를 얻기 위해 6주로 충분한지 사전 계산 필요
  - 담당: PO + Data Analyst | 기한: W1 내

### 10.4 용어 정의

| 용어 | 정의 |
|---|---|
| Activation | 신규 사용자가 "핵심 가치"를 처음 경험하는 시점. 이 PRD에서는 "첫 프로젝트 생성" |
| TTV (Time-to-Value) | 가입부터 Activation 시점까지의 소요 시간 |
| AHA Moment | 사용자가 제품의 핵심 가치를 체감하는 순간 |
| Feature Flag | 특정 기능을 일부 사용자에게만 노출하는 제어 장치. 점진적 롤아웃에 사용 |
| Guardrail Metric | 개선 과정에서 악화되어서는 안 되는 방어 지표 |
